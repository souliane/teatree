# BLUEPRINT Appendix — Loop Topology (deep)

Detail behind [BLUEPRINT.md](https://github.com/souliane/teatree/blob/main/BLUEPRINT.md) §5.6. The umbrella retired/subsumed paragraphs stay top-level (they satisfy doc-alignment tests). The deep mechanics — scanners, dispatch, multi-overlay/multi-host scanning, statusline rendering, mode + training-wheel, availability — live here. Cross-references `BLUEPRINT §5.6`, `§5.6.1`, `§5.6.2`, `§5.6.3` resolve here.

Each tick runs three stages:

1. **Scan** — pure-Python scanners under `teatree.loop.scanners` collect signals from external sources in parallel. Deterministic, mockable, integration-tested against stubbed backends; they never invoke Claude.
2. **Dispatch** — the tick decides what each signal means and either acts inline (mechanical fix-and-push, n8n webhook trigger, statusline note) or delegates to one of the seven phase agents shipped with the plugin (§ 11.2), invoked via the Task tool the same way `t3:orchestrator` invokes them; the loop is just one more caller.
3. **Render** — `teatree.loop.statusline.render` writes `${XDG_DATA_HOME:-$HOME/.local/share}/teatree/statusline.txt` with three zones (§ 5.6.1). The reader-side hook (`hooks/scripts/statusline.sh`) honours `$TEATREE_STATUSLINE_FILE` for tests/overrides; the renderer always writes the default path so hook and writer can never disagree on location.

**Mini-loops ([#1432](https://github.com/souliane/teatree/issues/1432)).** `teatree.loops.<domain>`; `Orchestrator` discovers via `pkgutil`, gates via `LoopsConfig` + `MiniLoopMarker`, isolates errors. `run_tick` stays wire-compatible. The live tick reaches the registry via the `loop_tick` `jobs_builder` seam → `teatree.loops.fanout.build_registry_jobs` (so `teatree.loop` never imports `teatree.loops` up-stack), sharing the `teatree.loops.gating` enable+cadence gate with the orchestrator and staying behaviour-equal to legacy `build_default_jobs` under a parity test ([#1481](https://github.com/souliane/teatree/issues/1481)).

**Dreaming cron — off the live tick (#1933).** The idle-time memory-consolidation pass is heavier than a scanner tick and must not run on or re-arm the live 12-minute loop. Unlike every scanner above it is **not** dispatched by the tick: the `dream` mini-loop is registered (so its cadence is configurable under `[loops.dream]` and the statusline shows its countdown) but carries the new `MiniLoop.off_live_tick = True` flag, so `teatree.loops.fanout.build_registry_jobs` (the live fan-out) and the `Orchestrator` skip it. It is driven only by its own low-frequency cron — `t3 dream tick` (cadence-gated, schedule ~04:00) — with `t3 dream run [--since <iso>] [--dry-run]` as the manual escape hatch (`run` ignores cadence; `--dry-run` does everything except writing rows). Both delegate to the `dream` management command, which takes the in-flight `LoopLease` (`dream-tick`, leased for `DREAM_LEASE_SECONDS` ≥ the wall-clock pass budget, not the 120s `acquire` default) so two passes never overlap even across a long pass, invokes the distillation-engine **seam** `teatree.loops.dream.engine.run_consolidation` (a typed stub in this scaffold; the real replay → dedup/merge → distill → re-index → archive engine is a follow-up PR), and on a clean pass stamps `DreamRunMarker` succeeded — a failed pass bumps only the attempt timestamp (a `--dry-run` stamps nothing, even on raise), so the **48h staleness alarm** (`t3 doctor`, `_check_dream_staleness`) keeps firing until a clean pass lands; the alarm's remedy points at scheduling `t3 dream tick` (which advances the cadence ledger), not a one-off `t3 dream run`. The output store is the DB-backed `ConsolidatedMemory` ledger (not in-place file rewrite).

**Agent dispatch via DB (not stdout, not files).**

Every ``DispatchAction(kind="agent", …)`` produced by the dispatcher is persisted as a ``Ticket`` + initial ``Task`` row by ``teatree.loop.persistence.persist_agent_actions``:

- ``t3:reviewer`` agent actions create a ``Ticket(role="reviewer", issue_url=<pr_url>)`` plus a ``Task(phase="reviewing")``.
- ``t3:orchestrator`` auto-start actions create a ``Ticket(role="author")`` plus a ``Task(phase="coding")``.

The ``/loop`` slot drains pending Tasks via ``t3 loop claim-next`` (Django queryset, no file IO): each call **atomically claims** the oldest pending dispatchable Task and only then emits its dispatch payload, so the claim boundary IS the spawn boundary — the slot spawns the sub-agent in-session via its ``Agent`` tool for the already-claimed entry.

Atomicity is a single conditional ``UPDATE ... WHERE status='pending'`` (compare-and-swap on the status column), **not** ``select_for_update(skip_locked=True)`` — production is SQLite, where ``has_select_for_update_skip_locked`` is ``False`` and that clause is silently dropped. The CAS is correct on SQLite and Postgres, so two concurrent ticks never double-dispatch one Task (#786 N4/B1). (The older ``pending-spawn``/``spawn-claim`` pair — list-all-then-claim-after — is retained for compatibility but is the spawn-then-claim race ``claim-next`` replaces.)

The claim carries two orthogonal attribution columns: ``Task.claimed_by`` (the worker role/label) and ``Task.claimed_by_session`` (the worker session id that took it, #1917). ``claimed_by_session`` rides the conditional-UPDATE **SET clause only** — never the ``WHERE status='pending'`` CAS predicate — so the claim semantics are unchanged whether or not a session is supplied. ``t3 loop claim-next`` / ``loop_dispatch claim-next`` default ``--claimed-by-session`` to ``current_session_id()`` (empty when no session resolves), and the reclaim/reap sweeps clear it alongside ``claimed_by``. It is inert for every existing caller: an unsupplied session is the empty string.

**Admit-budget gate before the CAS (#1796 WI-1).** This live claimer is the SINGLE claim point — ``orchestrate_phase`` no longer claims; it is a read-only PLANNER that persists a per-tick admit *budget* (the clamped ``speed=full``/``boost``/``slow`` fan-out cap) to the ``tick-meta.json`` sidecar (``teatree.loop.admit_budget``). Before the CAS, ``claim-next`` reads that budget and refuses (emitting the existing empty no-work payload) once the standing in-flight CLAIMED dispatchable WIP (``Task.objects.in_flight_claimed_count``) has reached the ceiling — a stateless standing-WIP cap, with the CAS still serializing the marginal claim. This reconciles the two formerly-unreconciled paths (the tick used to claim AND this claimer claimed): with the gate, claimed ≡ spawned and there is no orphan window. Absence of a budget — ``medium`` speed, the default-OFF ``orchestrate_claim_enabled`` toggle, or a stale budget past its ``~2× cadence`` TTL (a dead loop wrote it) — reads as UNCLAMPED, byte-identical to today's throughput; a dead loop never wrongly throttles live dispatch.

The statusline file is **display-only**, never an orchestration channel; tick stdout never ferries dispatches. When the reviewing task completes, ``Ticket.mark_reviewed_externally`` short-circuits the reviewer-role ticket to ``DELIVERED`` and writes the reviewed SHA back to ``ReviewerPrsScanner``'s cache, preventing re-spawn at the same SHA.

**Why pure-Python scanners (not subagents):** the scan stage is deterministic I/O — fetch PR statuses/mentions, query the DB. A Claude agent would burn tokens for work a typed Python function does cheaper, more reliably, with reproducible tests. Claude is invoked only for judgment (review the diff, decide the fix, draft a reply), via the existing phase agents.

**Why fat-loop, not many small loops:** `/loop` is session-scoped with a 50-task / 7-day expiry. One fat loop costs one slot; N small loops would saturate the slot budget.

**Per-loop owning-session layer (#1834, agent-teams Track-A).** The machine-wide single loop-owner (the `loop-owner` `LoopLease` row, claimed by the fat-loop tick gate; #1073) keeps its exact pid-anchored, hijack-guarded semantics and stays the default. **Alongside** it, ownership generalizes to a per-loop key: a dedicated loop claims `loop:<name>` (e.g. `loop:dispatch`, `loop:review`) via the SAME name-parameterized `LoopLease.objects.claim_ownership` CAS + pid-anchor + TTL machinery — there is no parallel weaker path, so the empty-owner guard, pid-liveness, and take-over apply per-loop too, and two dedicated loops can be owned by two different sessions concurrently with no false cross-loop hijack. `teatree.core.loop_lease_manager.per_loop_owner_slot(name)` is the single canonical-key function (the bare `dispatch` is qualified UP to `loop:dispatch` at every boundary); the `loop:` namespace is disjoint from the reserved `loop-owner` slot and from the infra-slot leases (`loop-tick` etc., which use `-` not `:`), so a per-loop claim can never collide with or evict the global owner. This layer is **opt-in and inert under the default fat loop**. The default `t3 loop list` and the statusline loop line **scope to the current session's owned loops** (WI-2): the per-loop block / `loop:<name>` chunks are filtered to the loops `current_session_id()` owns via the shared `teatree.loops.live.owned_per_loop_owners(report, session_id)` seam (and the statusline-side `teatree.loop.loop_scoping.owned_per_loop_slots`), so a foreign session's dedicated loops are subtracted. `t3 loop list --all` stays the read-only cross-session health view that lists every per-loop owning session (session id, pid-liveness, live/stale) beneath the global owner line. Scoping is **fail-open**: an anonymous / cron tick (empty `current_session_id()`) shows the FULL view, never an empty one. The **single-owner default** (no `loop:<name>` lease — `dedicated_loops` off) has an empty per-loop set, so the default output short-circuits to **byte-identical** to today.

**Dedicated-loop slot generator (#1838, agent-teams Track-A — the first claimer).** Behind the default-off `[loops] dedicated_loops` toggle (`UserSettings.dedicated_loops`, per-overlay overridable, `T3_DEDICATED_LOOPS` env; fail OFF), the single fat `/loop` slot is replaced by N **dedicated** `/loop <cadence>` slots, each driving ONE dedicated loop — a named GROUP of registered mini-loops — via a SCOPED `t3 loop tick --slot <name>`. The grouping is one declarative source, `teatree.loops.dedicated.DEDICATED_LOOPS` (dispatch/review/ship/inbox/followup/housekeeping/resource), with a coverage **fitness test** asserting every `iter_loops()` name maps to exactly one group (no orphan, no double-assignment). A scoped tick claims `loop:<name>` (the per-loop owner above, with all its #1073 invariants) instead of `loop-owner`, and runs `Orchestrator.tick()` filtered to that group's members ONLY — disjoint slots, so the global-owner gate and a scoped tick never block each other and two dedicated loops run under two sessions concurrently. The slot generator (`t3 loop start`, inspectable via `t3 loop start --print-slots`) emits the N dedicated slots when the toggle is ON, the single fat slot when OFF. **Default OFF ⇒ byte-identical to today**: the no-`--slot` tick claims `loop-owner` and runs the full `run_tick`, which stays as the fat-mode fallback (not deleted). Dedicated-mode and fat-mode are mutually-exclusive deployment modes — when the toggle is ON the fat slot is not started, and vice-versa.

**Inert team-role registry (#1838, agent-teams Track-B — ships dark).** The optional WORK-team layer (pane-backed teammates that execute claimed work and split context; the loops themselves never leave the lead session) declares its three roles as PURE DATA in `teatree.teams.roles`: `TeamRole.{CORE_MAKER, OVERLAY_MAKER, REVIEWER}`. Each role's canonical `claimed_by` key is `team:<role>`, produced only via the single normalization seam `team_claim_slot(role)` (a bare `core-maker` is qualified UP to `team:core-maker`; an already-qualified key is returned unchanged — mirroring `per_loop_owner_slot`). The `team:` namespace is provably disjoint from `loop-owner`, the `loop:` per-loop prefix, and the `loop-*` infra leases, so a team claim can never collide with or evict a loop owner. Each maker role carries a declarative overlay-seam claim filter (a pure `Q`): CORE-MAKER → `overlay == ""` (core units), OVERLAY-MAKER → `overlay != ""` (overlay-specific units) — the disjoint `overlay` seam IS the context split; REVIEWER is read-only and carries none. A teammate is just another `claimed_by` value on the existing free-form `Task.claimed_by` CharField, so there is **no schema change and no migration**. The whole registry ships behind the default-off `[teams] enabled` toggle (`UserSettings.teams_enabled`, per-overlay overridable, `T3_TEAMS_ENABLED` env; fail OFF) and is **inert**: nothing in the loop / dispatch / claim execution path references it yet (an AST import-scan fitness test pins that), so it is dead code until a later PR wires the pane-spawn helper and the live teammates in. **Default OFF ⇒ zero behaviour change**, and even ON nothing runs until a consumer lands.

**Pane display layer (#1838, agent-teams Track-B — WI-5, presentation-only).** `teatree.teams.pane_display` is the optional half that renders a maker pane's in-process SDK session in a VISIBLE terminal pane (the pane's SDK session always runs in-process — `build_pane_options`'s resolved session id is the single source of truth; the display never replaces it). `spawn_pane(pane, options)` issues `tmux split-window -c <cwd> -P -F '#{pane_id}' -- claude --resume <session_id> --model <model> --permission-mode bypassPermissions` then `select-pane -T 'team:<role>'`; the `--resume` id is the SAME SDK session, and the visible child is plain interactive `claude` (NEVER `claude -p`) so it routes through the interactive-billing lane, not a metered-headless backdoor. The mechanism is tmux control mode: under `tmux -CC` iTerm2 renders `split-window` panes as NATIVE splits in one tab, and elsewhere the identical plain commands degrade to ordinary tmux panes — the native rendering is a property of how the USER attached, so teatree gets it for free (mirroring Claude Code's own `teammateMode: tmux | in-process | auto`). `detect_multiplexer()` probes `$TMUX` / a `tmux` binary / a TTY and returns `"tmux"` or `"none"`; on `"none"`, no tmux, no TTY, or ANY `spawn_pane` failure the in-process path stands unchanged. Teardown is DB-DRIVEN: the pane-reaper scanner's demote-to-STOPPED triggers `reconcile_orphan_panes(live_claim_slots=…)`, which `kill-pane`s every `team:*`-titled pane whose slot is no longer a live claim (the DB lease is authoritative, tmux is the downstream effect reconciled to it), and the same reconcile reaps orphaned panes at startup; `teardown_pane` is idempotent (a missing pane is not an error). The mode is `[teams] display = "auto" | "tmux" | "none"` (`UserSettings.teams_display`, default **`none`** — read from the `[teams]` table, per-overlay overridable via `[overlays.<name>].teams_display`, `T3_TEAMS_DISPLAY` env wins; a bad value fails safe to `none` so the presentation layer can never escalate itself on by a typo). **Default OFF (`none`) + no tmux + headless/CI ⇒ byte-identical to today** (the load-bearing anti-regression). `pane_display` is a pure leaf reached on the live path only via the same sanctioned pane-reaper scanner, so the #2320 inertness scan stays green.

**Multi-overlay scanning.** A user with more than one overlay (e.g. one per code-host identity) sees PRs, issues, and Slack mentions from all in one tick. `teatree.core.backend_factory.iter_overlay_backends()` returns one `OverlayBackends` bundle (`hosts`, messaging, `ready_labels`, auto-start config, `identities`) per overlay; `run_tick(TickRequest(backends=...))` fans out one set of scanners per overlay through a single thread pool, tags each signal with its overlay name, and prefixes rendered statusline lines with `[overlay]`. `t3 loop tick` defaults to every overlay; `--overlay <name>` restricts to one. Inputs flow through a typed `TickRequest` dataclass (`scanners`, `host`, `messaging`, `backends`, `notion_client`, `ready_labels`) keeping the call surface narrow.

**Multi-host scanning per overlay (#976).** An overlay with both a GitHub PAT and a GitLab PAT scans both forges in one tick. `OverlayBackends.hosts` is a tuple of every `CodeHostBackend` whose token resolved; `tick.build_default_jobs` emits one `MyPrsScanner` / `ReviewerPrsScanner` / `AssignedIssuesScanner` / `TicketDispositionScanner` per host. `TicketCompletionScanner` is overlay-scoped (reads the local `Ticket` table), emitting at most once per overlay regardless of host count. `OverlayBackends.host` aliases `hosts[0]` for legacy single-platform callers — GitHub wins when both tokens are set, mirroring `get_code_host`'s precedence.

**Multi-identity scanning per host (#976).** A user owning more than one identity on a forge (personal + org-account login under one PAT, or different handles on GitHub vs GitLab) configures `[teatree] user_identity_aliases = ["handle-a", "handle-b"]`. `UserSettings.user_identity_aliases` parses it and `iter_overlay_backends` threads it onto `OverlayBackends.identities`; `MyPrsScanner` / `ReviewerPrsScanner` / `AssignedIssuesScanner` union-query each alias and dedup by URL so a PR returned for two aliases renders once. Empty (default) keeps legacy single-identity behaviour — scanners scan only `host.current_user()`.

**Auto-start on assigned issues.** Each overlay opts in via `auto_start_assigned_issues = True` and optionally `max_concurrent_auto_starts` on its `OverlayConfig` (base default `1`; the in-repo dogfooding overlay overrides to `3`). `AssignedIssuesScanner` dedupes ready issues against active tickets (any state except `delivered`/`ignored`), counts in-flight auto-started tickets (states `not_started..shipped` with `extra.auto_started == True`), and emits at most `max - in_flight` `assigned_issue.ready` signals per tick with `payload.auto_start = True`. The dispatcher routes auto-start signals to `t3:orchestrator` (chains `coder → tester → reviewer → shipper`, halts at PR creation per `require_human_approval_to_merge`); `auto_start = False` signals go to the `action_needed` zone for manual triage. Phase failures halt the chain; the failing ticket stays visible via `MyPrsScanner` and the statusline.

**Ticket dispositions (phase 7).** `TicketDispositionScanner` walks active pre-PR `Ticket` rows once per tick — states `not_started..reviewed`, active overlay — calling `CodeHostBackend.get_issue(ticket.issue_url)` to detect drift between local intent and remote reality. Three reasons trigger a `ticket.disposition_candidate` signal: `issue_closed` (remote state ∈ `{closed, completed, cancelled}`, often a different PR's `Fixes #N` keyword auto-closed it), `unassigned` (the active user dropped from assignees), `label_removed` (no configured `ready_labels` remain). Multiple reasons emit multiple signals so the statusline shows each independently. **Detection-only**: it never transitions the ticket or tears down worktrees — disposition is a judgment call (was the work superseded? does the colleague's reassignment override mine?) and is uncommon enough that asking the operator is cheap. Tickets past `REVIEWED` are skipped: once a PR exists, `TicketCompletionScanner` covers post-PR state.

**Reassign noise filter (#975).** One operator may carry several handles across platforms (a GitHub login, a GitLab username, an internal id); a reassignment between two is plumbing, not an actionable handoff. `UserSettings.user_identity_aliases` (default `[]`, in `[teatree]`, per-overlay overridable via `OVERLAY_OVERRIDABLE_SETTINGS`) lists them; the scanner drops the `unassigned` signal when `old_owner ∈ aliases` AND every `new_owners` entry ∈ aliases. Reassigns crossing the alias boundary (alias↔colleague, alias→mixed) still render — the colleague handoff is the actionable case. Default empty list keeps legacy behaviour. **Implicit self-group from `backend.identities` (#1113).** When neither `user_identity_aliases` nor an explicit `identity_alias_groups` resolves, `scanner_factories._jobs_for_backend_hosts` unions `backend.identities` (the #976 list) into the scanner's `identity_alias_groups` as an implicit one-human self-group, closing the gap where aliases are empty but identities known. Explicit group config takes precedence.

**Ticket completion (phase 8).** `TicketCompletionScanner` covers the gap dispositions leave: post-ship tickets (states `shipped`, `in_review`, `merged`) whose upstream issue indicates all work is done. Unlike dispositions (detection-only because pre-PR closure is ambiguous), post-ship completion is mechanical — the user shipped, so a closed issue means the work landed. The scanner calls `OverlayBase.is_issue_done(issue_data)` per overlay: default (GitHub) checks `state ∈ {closed, completed}`; GitLab overlays override to check for a process label (e.g. `Process:DEV Review`), since multi-repo tickets have multiple MRs and the issue stays open until the last merges. The dispatcher routes `ticket.completion_detected` as a `mechanical` action — the tick transitions the ticket `request_review → mark_merged → retrospect` without agent involvement. `t3 <overlay> ticket sync-completions` runs the same logic as a one-off bulk cleanup.

## 5.6.1 Statusline rendering

The statusline is the **only persistent UI surface** — written to a file by the loop, read by the statusline hook (`hooks/scripts/statusline.sh`). The hook composes a header line (model, context-window %, 5h/7d rate-limit usage with color-coded thresholds, RAM, disk, repo freshness, loaded skills) from Claude's stdin JSON and the `tick-meta.json` sidecar, `cat`s the loop's pre-rendered zones file, then chains any scripts in `[teatree] statusline_chain`. The header carries **no loop or tick information** (#130): loop state has one home — the dedicated loop line atop the zones file (`live_loops_anchor`, below) — so the next-tick countdown lives there, not the header.

**Auto-load is opt-in (#256).** Both the statusline hook and the `SessionStart`/first-prompt loop machinery (the tick-owner claim, the `CronCreate` registration nudge) self-silence unless the session opted into teatree (the `<session>.teatree-active` marker) AND the operator enabled auto-load. The Python side is the single gate `hook_router._loop_auto_load_active`; the bash statusline mirrors its `_loops_auto_load_enabled` resolution (env `T3_LOOPS_AUTO_LOAD` first, then `[loops] auto_load` in `~/.teatree.toml`). It defaults **OFF** and fails closed, so a fresh clone — even one loading a teatree skill for dogfooding — never sees the loop statusline or a register-the-cron nag. The loop owner opts in once (`[loops] auto_load = true`) and keeps the existing behaviour; the `[loops] enabled` kill-switch (default ON, fail-open) is the orthogonal "owner can pause their own loop" knob.

**Header line segments:** `model=Opus 4.8 · ctx=42% │ 5h=18% (until 14:30) · 7d=63% │ t3=0(30m) acme=3(2h) │ ram=71% 51/64G · disk=22% 410G free │ skills: code review`. Segments are grouped (context · usage · freshness · resource · skills) and color-coded (green < 80%, yellow 80–94%, red ≥ 95% for percentages; light-grey for labels). Repo freshness is derived from `tick-meta.json` (`freshness`): commits behind `origin/main` and fetch age for `$T3_REPO` and each overlay with a `path` in `~/.teatree.toml` (green = 0, yellow = 1–5, red = 6+).

**Zones (three, fixed order):**

1. **Anchors** — always shown: active overlay, current ticket (if any), branch, context-window usage.
2. **Action needed** — items requiring my attention this tick: failing pipelines on my PRs, mentions and DMs the loop couldn't auto-handle, PRs with new pushes since my last review, new assigned issues awaiting kickoff.
3. **In flight** — what the loop is doing: PR sweeps in progress, headless tasks claimed, current `/loop` cadence and tick count.

The hook reads the file in <10 ms. The render-to-file pattern means the loop can spend tens of seconds composing the statusline content without slowing the hook.

The render module lives at `src/teatree/loop/statusline.py` (`StatuslineZones` dataclass + `StatuslineEntry(text, url)` for URL-aware lines + `render(zones, target=..., colorize=...)`). The default file path is `${XDG_DATA_HOME:-$HOME/.local/share}/teatree/statusline.txt`; the hook honours `TEATREE_STATUSLINE_FILE` for tests and overrides.

**Color and hyperlinks.** Each zone carries a distinct ANSI color (dim grey for anchors, bright red for action_needed, bright cyan for in_flight). Set `NO_COLOR=1` (or pass `colorize=False` to `render()`) to disable. Signals that include a `url` payload render as OSC 8 terminal hyperlinks (clickable in iTerm2, Kitty, WezTerm, Ghostty); the renderer falls back to `text <url>` when colorize is off so non-OSC8 terminals still receive the URL.

**Per-overlay one-line collapse.** `rendering.py` folds each overlay's action-needed signals onto a single `[ov] …` row rather than one line per item, so the red zone stays one line per colour per overlay. Two readability refinements ride on that row: the `unassigned` disposition carries `old_owner`/`new_owners` (a typed `_DispositionReason` set by `TicketDispositionScanner`) and renders as `reassigned (from <old> → to <new>): #N` instead of a bare `reassigned` (signals lacking owner data still render an unambiguous `reassigned:`); and `StaleTicketsScanner` emits a concise `#N stale (3d)`-shaped summary plus `overlay`/`issue_url`/`stale` payload keys so stale tickets collapse to one `[ov] N stale: #a #b #c` row with every ref a clickable link.

**Dedicated loop line (#1163, #1400, #130).** `teatree.loop.statusline.live_loops_anchor()` folds every live `LoopLease` row into a **single** loop line at the top of the zones file: `<short-name> <Nm> · … · waiting: N questions`. It leads with one `<short-name> <next-tick>` chunk per live lease (the `loop-` prefix is stripped; the next-tick is a relative whole-minute countdown per the loop's own cadence), surfacing the parallel-loop topology the old single `loop-owner` row hid. There is no leading `loop running` state word — the `tick <next-tick>` chunk already carries the loop's liveness (both derive from the same live-lease set), so the word was redundant with the tick token. The per-session `loop-owner:` badge is prepended to the FRONT of this line by `hooks/scripts/statusline.sh` so the user reads ownership first. The trailing `waiting: N questions` clause appears **only** when the loop is blocked on the user (unresolved `DeferredQuestion` rows). When **no** loop is live the function returns `[]` and the line is silenced entirely (no `idle` line). The foreign-hijack case is NOT shown here — it is RED, routed to the action line by `loop_owner_anchor`. Fail-open: a read error degrades to no line.

**Open-PR anchor (#271).** `teatree.loop.open_prs.open_prs_anchor()` folds the user's open PRs (drafts included) for the configured overlays into an anchor block: a headline `open PRs: N (M draft)` followed by one clickable `#iid title` row per PR (capped at five, with a `(+K more)` overflow; draft rows carry a `[draft]` marker). It never queries the code host — the count is read from the `open-prs.json` sidecar that `run_tick` writes beside the statusline from the `my_pr.*` signals the `MyPrsScanner` already produced (`teatree.loop.open_prs`: `open_prs_from_signals` → `write_open_prs_cache`, read back by `read_open_prs_cache`). The whole open-PR concern (cache + anchor) lives in `teatree.loop.open_prs` beside the data, not in `statusline.py` (locality of behaviour; keeps the statusline module under its size budget). The tick is the single fetch point and the sidecar is the cache, so a statusline render costs zero API calls and stays under the rate limit (the same shape as the `tick-meta.json` freshness sidecar). The cache is rewritten every tick — even an empty one — so it never goes stale after the last PR merges. Returns `[]` (no line) when no PR is open; a missing or corrupt cache degrades to no line.

**Canonical item shape (#1015, #1156).**

Every state-line item — the anchor `ready:`/`started:`/`tested:` rows and the action-needed `ready:` row — renders in the same `#N (short desc) !M1 (MR1 title) !M2 (MR2 title)` shape so the operator never has to mentally translate between two formats. The shape lives in `teatree.loop.rendering_items` (`_render_canonical_item` + `_format_mr_ref` + `_short_desc`, which tail-elides the title to 40 chars with a single-codepoint Unicode ellipsis).

Rendering details:

- Every number is a hyperlink (OSC 8 / `text <url>` fallback); the description chunk is omitted when the title is empty.
- #1156: each MR carries its own `(title)` chunk (threaded from the `my_prs`/`reviewer_prs` payload via `_PRRef.title`) and MRs are joined by a single space (no outer parens — each MR already brings its own).
- The active-ticket signal's `title` prefers `Ticket.short_description` (the AI-generated terminal-friendly summary written by `manage.py ticket_short_describe`) over the cached `extra["issue_title"]`; a blank `short_description` falls back to the cached title — see `ActiveTicketsScanner`.
- A ticket whose source tracker reports 404 (`extra["tracker_404"]`) emits `issue_url=""` so the renderer prints a bare `#N` instead of an unreachable link.

Two scanner-side gates make the row honest:

- `MyPrsScanner` / `ReviewerPrsScanner` accept `allowed_url_prefixes` (built from the overlay's `workspace_repos` × the host's web origin in `teatree.loop.tick_resolvers._allowed_url_prefixes_for_host`) so two overlays sharing the same PAT no longer render each other's MRs.
- `OverlayConfig.identity_aliases: list[list[str]]` groups one human's handles across forges so the disposition scanner suppresses self-handoff churn without conflating distinct humans — the active overlay's `identity_aliases` is the source of truth, with `[overlays.<name>] identity_aliases` in `~/.teatree.toml` as the fallback for TOML-only overlays.

**Review-permalink staleness filter (#1156).** `build_review_post_permalinks` now joins each `ReviewRequestPost` row to its `PullRequest` and excludes MERGED PRs from the permalink map — a stale review-thread post link on a merged MR routinely 404s in Slack (the thread may have been archived or the post deleted), so the renderer never surfaces it.

**Ticket↔MR index — third resolution source (#1113).** `teatree.loop.pr_ticket_index.build_ticket_index` resolves each MR URL to its parent ticket via three sources, highest priority first: (1) `PullRequest.ticket` FK row, (2) `Closes/Fixes #N` keyword parsed from the MR description, and (3) **`Ticket.extra["prs"]["<url>"]`** — last-priority fallback for a bare manually-opened MR with no FK row and no close-keyword footer. The ship pipeline already records every MR under its ticket's `extra["prs"]` map (see `backends/gitlab/sync_prs`), so the third source closes the index gap and a manual `!145` buckets under `#142` instead of rendering detached as an orphan at the tail of the overlay's PR group.

**Slack-user-reply contract — no statusline action (#1113).** `SlackDmInboundScanner` emits a `slack.user_reply` signal per drained user message (`{ts, channel, user_id, text}` payload, summary `"Slack user reply {ts}: {text[:80]}"`). The signal is routed **mechanically** by `dispatch._MECHANICAL_BY_KIND["slack.user_reply"] = ("mechanical", "slack_user_reply")` to its real consumer — the reactive Slack-answer loop (`teatree.loop.slack_answer`) that drains the `PendingChatInjection` rows the scanner records. The dispatcher never produces a statusline action for this kind, and `rendering_classification._is_slack_user_reply` is a defence-in-depth guard that drops the action shape before the `c.other` catch-all so the raw `ts`/`text` cannot leak into the red zone even if dispatch regresses.

**Review-channel post permalink (#1113).** When a `ReviewRequestPost` row exists for an MR (the ship pipeline records it on review-channel post), the statusline surfaces a clickable Slack permalink alongside the MR ref so the operator can jump from the statusline straight to the review thread. `teatree.loop.rendering_permalinks.build_review_post_permalinks` reads the row and rebuilds the canonical archive URL (`https://slack.com/archives/{channel}/p{ts_no_dot}` — same shape `teatree.backends.slack` already produces); `enrich_pr_refs_with_permalinks` copies it onto each matching `_PRRef.review_permalink`; `_render_canonical_item` and `_render_pr_group._label` append a `(review)` chunk when present. Empty input or DB error fails open to no chunk (the MR ref renders normally), so a missing/partial row never breaks a render.

## 5.6.2 Mode + training-wheel

The loop respects the active overlay's `mode` (§ 10.1, canonical default `interactive`). When an overlay opts into `mode = "auto"`, three training wheels keep specific actions gated while the rest of the pipeline runs autonomously.

- **`[teatree] require_human_approval_to_merge = true` (default).** Push and PR creation run autonomously, but merge stays gated — it requires a user reaction (👍 or `/merge`) on the statusline entry or the PR thread.
- **`[teatree] require_human_approval_to_answer = true` (default).** Gates the `t3:answerer` capability the same way: the agent drafts a reply to an inbound `question` intent, DMs the user for approval, and posts only on confirmation. Set it `false` per-overlay to let answers post directly.
- **`[teatree] on_behalf_post_mode = "draft_or_ask"` (default).** The tri-state *pre*-gate over every post the agent makes under the user's identity to a colleague/customer surface — a PR/MR comment, an issue comment, a Slack channel/thread message, a Notion post, a PR/MR **approval**, or a reaction on someone else's message.

The three `on_behalf_post_mode` values form an autonomy ramp. The gate
covers colleague-**VISIBLE** posts only — a draft (`post-draft-note`) is
colleague-invisible, so it is exempt under every mode and never needs
approval:

- `immediate` — gate off; every action publishes directly.
- `ask` — every colleague-VISIBLE action requires an explicit recorded approval before publishing; a draft is exempt and auto-publishes (+ DMs the user) without approval.
- `draft_or_ask` (the new default) — identical to `ask`: `t3 review post-draft-note` publishes autonomously (drafts are colleague-invisible and revocable, and the agent DMs the user with the publish/delete commands); every colleague-visible action BLOCKs until the user records an approval.

The legacy boolean `ask_before_post_on_behalf = true/false` keeps working through one deprecation release (true → `ask`, false → `immediate`). The setting is resolved through the env (`T3_ON_BEHALF_POST_MODE`) → active-overlay → global → default chain (`teatree.on_behalf_gate.resolve_on_behalf_verdict`); it is the *pre*-gate companion to the notify-*after* path (#949). Both ship on; the user flips this one off per-overlay first.

The gate is **enforced uniformly** at the chokepoints that cover every on-behalf publish path:

- `teatree.core.reply_transport._BaseReplier` — Slack thread reply, Slack channel post, GitLab MR comment, GitHub PR comment.
- `teatree.cli.review.ReviewService` — post-comment, post-draft-note, publish-draft-notes, reply-to-discussion, resolve-discussion, update-note, delete-discussion.
- the `t3 pr post-evidence` command (`teatree.core.management.commands.pr.PrCommand.post_evidence`) — gated inline so PR *creation*, which is not an on-behalf colleague-facing post, stays ungated.
- the two `post_transition` signals that publish reactions on a colleague-facing message — `PullRequest.approve` ✅ on the requester's review-request message, and the ticket-transition emoji signal.

The gate is **satisfiable, not pure suppression**: it mirrors the #953 `DbApproval` / §17.4 `MergeClear` shape — a durable, single-use, scoped, maker≠checker `OnBehalfApproval` row that `teatree.core.on_behalf_gate_recorded.require_on_behalf_approval` consumes single-use (writing an `OnBehalfAudit` row). A chat-only operator satisfies the gate **without a TTY** by recording one (`t3 review approve-on-behalf <target> <action> --approver <id>`); the next on-behalf attempt then publishes and the row is consumed. Under `draft_or_ask` the `AUTO_DRAFT` verdict additionally records a `BotPing` row keyed by `on_behalf_autodraft:{target}:{action}` so each (target, action) pair generates exactly one user DM regardless of retries.

With no recorded approval (BLOCK verdict) the helper raises `OnBehalfPostBlockedError` and the caller surfaces the blocked post — never a silent drop, never an unattended post. The FSM transition itself is never blocked, only the on-behalf side effect.

DMs *to the user themselves*, the `Replier.post_dm` path, the DailyDigest user thread, the `AskUserQuestion` Slack mirror, and internal-only orchestration writes (our own backlog issues, durable memory, task bookkeeping, the sanctioned `t3 ticket clear/merge`) are out of scope and remain ungated. The user flips any training wheel to `false` / `immediate` only when comfortable. In `interactive` overlays, every publishing action still prompts; the loop surfaces work but never publishes silently.

**On-behalf gate scope (explicit list).**

The pre-gate covers every action that publishes *to a third party* under the user's identity: `post_comment`, `publish_draft_notes`, `reply_to_discussion`, `resolve_discussion`, `update_note`, `delete_discussion`, `post_evidence`, and the two FSM-driven reactions (`PullRequest.approve` ✅ on the requester's review-request message + the ticket-transition emoji signal — both `post_transition` signal handlers).

It deliberately does *not* cover:

- `Replier.post_dm` — a bot→user DM, not on-behalf.
- the smoke-test self-DM — the agent DMing its own user.
- the watching-eye 👀 acknowledgement reaction — TBD; reads as bot-identity bookkeeping, may be added later if it begins to read as user-on-behalf to the recipient.
- `t3 pr create` / overlay `pr create` — PR creation is authorship, not commentary; the recorded-approval channel does not gate authorship, only the colleague-facing comment paths the author then walks.

The scope rule is: "publishing commentary or a reaction on someone else's surface" ⇒ gated; "creating one's own artefact or messaging oneself" ⇒ ungated.

`UserSettings.require_human_approval_to_merge`, `UserSettings.require_human_approval_to_answer`, `UserSettings.loop_cadence_seconds` (default 720), `UserSettings.user_identity_aliases` (default empty; consumed by `TicketDispositionScanner` for reassign-noise filtering, see §5.6 phase 7), and `UserSettings.statusline_chain` (default empty) live in `src/teatree/config.py`; the first four are toml-overridable in `[teatree]` and per-overlay via `[overlays.<name>]` once registered in `OVERLAY_OVERRIDABLE_SETTINGS`. `statusline_chain` is global-only (not per-overlay).

## 5.6.3 Availability — 24/7 dual question-mode (#58, §17.3 C3)

Single-session 24/7 operation has two question modes (BLUEPRINT §17.1 invariant 9):

- `present` — the user is reachable; `AskUserQuestion` runs interactively the standard way and the §807 Stop gate enforces the structured tool.
- `away` — the user is unreachable; the `handle_route_away_mode_question` PreToolUse hook converts the `AskUserQuestion` tool call into a durable `DeferredQuestion` row and emits a `permissionDecision=deny` whose reason names the recorded row id. **Live-turn escape (#189):** when the turn is user-driven (`PRESENCE.is_live_user_turn` — a `UserPromptSubmit` for the same session within `LIVE_TURN_FRESHNESS` = 90 s, shorter than the 15-min `PRESENCE_FRESHNESS` used for the schedule-upgrade layer), the hook passes through instead: the question renders in-client without deferring. Each such in-client render slides the live window forward — `availability.PRESENCE.refresh_live_turn` re-stamps the same-session heartbeat to *now* — so a multi-question `/checking` walk-through keeps EVERY question live even across an intervening background task-notification turn, which never refreshes the heartbeat and would otherwise age the window out and defer the second question (#2058). The slide re-stamps only an *already-live* same-session turn, so it can never promote an autonomous turn to live. Autonomous/loop-driven turns are unaffected (no recent user prompt → still defers). This preserves invariant 9 — the user receives the question synchronously on their user-driven turn — and is what makes `/checking` work without an availability flip. For deferred calls: the tool_use block still appears in the transcript — the §807 gate sees it and lets the turn complete; the agent then proceeds with any work that does not depend on the answer. The user resolves the question on their own clock with `t3 questions answer <id> <text>` / `t3 questions dismiss <id>`; a `DeferredQuestionAudit` row records the resolution (mirroring the `MergeAudit` / `OnBehalfAudit` / `DbAudit` family).

**Mode resolution is a single deterministic precedence** (each layer is independently testable; the production resolver is `teatree.core.availability.resolve_mode()`):

1. **Manual override (unexpired)** — recorded on disk at `$T3_DATA_DIR/availability_override.json` by `t3 availability away|present|auto [--until ISO8601]`. The write is atomic (`tmp.replace`) so a torn write never leaves a half-encoded override. `auto` deletes the file so the schedule decides again. A deliberate `away` (holiday) is authoritative.
2. **Live presence (upgrade-only)** — a `UserPromptSubmit` heartbeat (`$T3_DATA_DIR/availability_presence`, stamped fail-open by `handle_record_presence`) within `PRESENCE_FRESHNESS` (15 min) is direct evidence the user is at the keyboard now. The schedule only guesses at reachability, so a fresh prompt **upgrades** a schedule `away` to `present` (source `live`) — fixing the #58 case where a user typing outside work hours had `AskUserQuestion` calls deferred. It only upgrades a schedule `away`; never downgrades a present schedule, never overrides a manual `away`.
3. **Cron-window schedule** — `[teatree.availability].windows` is a list of cron expressions (e.g. `["* 9-16 * * 1-5"]`) evaluated against `[teatree.availability].timezone`. Each is read as a *span*: the last fire keeps you `present` for its cadence, capped at 1 h (so a sparse `0 9 * * 1-5` presents only ~1 h per fire). Multiple windows OR together — any active = present.
4. **Default** — `present` (the conservative default: an agent with no configured schedule answers in-band, never silently muted).

The away-mode path **never bypasses the §807 structured-question gate** — the recorded `tool_use` block satisfies it (a sanctioned destination, not a prose fallback). The dedicated loop line carries an `availability: <present|away> (<source>)` segment read live, so the user sees the resolved value and which layer decided it. The agent also sees pending questions in `additionalContext` each turn (`handle_inject_pending_questions`), so it sequences independent work and never re-asks.
