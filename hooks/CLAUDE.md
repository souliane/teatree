# hooks — local conventions

See the root [`CLAUDE.md`](../CLAUDE.md) for the code-quality bar. This file adds only what is specific to `hooks/`.

- **One router, many events.** `scripts/hook_router.py` dispatches on `--event`. Registered events: `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `TaskCreated`, `Stop`, `SubagentStop`, `PreCompact`, `SessionEnd`, `InstructionsLoaded`. Wiring lives in `hooks.json`. `SubagentStop` carries `session_id` / `cwd` / `transcript_path` and fires once per sub-agent termination — `handle_subagent_stop_no_commit` (#1205) records a `terminated_without_commit` signal when the sub-agent's worktree (the harness `cwd`) is a work branch with 0 commits, so the orchestrator does not assume lost work landed.
- **Adding a gate** = a new handler branch in `hook_router.py` for an existing event (usually `PreToolUse` or `Stop`). To **deny** a `PreToolUse` tool call, call the shared helper and return its `True`:

  ```python
  return emit_pretooluse_deny(reason)
  ```

  The helper writes the modern nested `hookSpecificOutput` schema that Claude Code 2.1.146+ actually reads, alongside the legacy flat keys for back-compat. `main()` translates the helper's `True` return into `sys.exit(2)` — exit 0 deny payloads are silently ignored by the harness (changelog: "Fixed `PreToolUse` hooks that emit JSON to stdout and exit with code 2 not correctly blocking the tool call"). See #1447 for the diagnosis.

- **A gate only fires for tools wired into its event's matcher (#171).** `mcp__*` tools are ordinary `PreToolUse` calls merely omitted from the matcher — adding the matcher string makes them fire, e.g. `mcp__glab__glab_mr_.*` routes a glab-MR-MCP create/update through the live `handle_validate_mr_metadata` validator (same metadata check as the `glab` CLI; fail-closed with the `T3_MR_VALIDATE_ALLOW_BROKEN_ENV=1` escape). The `Agent` **tool** likewise reaches `PreToolUse` (a foreground Agent dispatch fires it with `run_in_background` in the tool_input — verified against the installed binary), but no `Agent` matcher is wired (the registered matchers are `Bash|Edit|Write`, `AskUserQuestion`, `mcp__.*[Ss]lack.*`; `Agent` has only ever been in the PostToolUse matcher). So the three `PreToolUse` `Agent` arms (`handle_enforce_agent_plan_gate`, `handle_dispatch_prompt_quote_scanner`, and the `Agent` branch of `handle_enforce_orchestrator_boundary` → `_deny_foreground_agent_dispatch`, #1442) are phantom — but adding an `Agent` matcher WOULD make them live. They are kept unwired **deliberately**: the orchestrator-boundary deny sits on the orchestrator's own foreground Agent-dispatch hot path, so enabling it is a lockout risk to be validated attended (#1646), and it additionally ships default-OFF behind `[teatree] orchestrator_boundary_agent_gate_enabled` (off-ramps: sub-agent context, `run_in_background: true`, per-call `[fg-ok: <reason>]` prompt token). Do NOT conflate this with the SEPARATE `Task`/`Workflow` fan-out vehicle (next bullet), which genuinely bypasses `PreToolUse` and fires `TaskCreated` — that is why the plan-gate and dispatch-quote concerns ALSO carry reachable `TaskCreated` counterparts (the `TaskCreated` schema has no `run_in_background`, so the orchestrator-boundary arm has no such counterpart and the Agent-matcher path is its only fix).

- **The MR-metadata gate watches every surface that sets a title/description, not just the inline create command ([#1703](https://github.com/souliane/teatree/issues/1703)).** `_extract_mr_fields` covers four: (1) inline `glab mr create/update --title/--description`; (2) the SAME command's file-based / heredoc / `$(…)` description (`-F`/`--description-file`), read via `_read_message_file` rather than passed through as a falsely-empty (trivially "valid") string; (3) out-of-band `glab api`/`gh api` PUT/POST to a `.../merge_requests` or `.../pulls` endpoint (`--field title=…`/`-f description=…`/`-f body=…`) — the web-UI-equivalent edit that bypasses the create/update CLI entirely; (4) the `mcp__glab__glab_mr_*` MCP tools. The out-of-band API path validates ONLY the fields the command actually sets: a partial edit touching neither title nor description (e.g. `--field state_event=close`) is skipped (never-lockout), and a single-field edit mirrors the set field onto the untouched one (a valid `type(scope): … (ticket_url)` line is, by the canonical grammar, simultaneously a valid title and a valid description first line) so it can never false-block on `Title is empty.`. This closed the gap where a non-compliant description set via a REST-API field edit reached the remote unvalidated and then failed the remote MR-title/description CI job. The `T3_MR_VALIDATE_ALLOW_BROKEN_ENV=1` escape and `T3_MR_VALIDATE_SCRIPT` override are unchanged.

- **`TaskCreated` is the only non-`PreToolUse` deny path, with a DIFFERENT schema (#1488).** The harness Workflow/Task fan-out vehicle (what `ultracode` and any teammate spawn use) **bypasses `PreToolUse`/`PostToolUse` hooks** — so a gate that must govern the fan-out rides the `TaskCreated` event instead (no matcher, fires on every task creation; stdin schema `session_id`, `task_id`, `task_subject`, `task_description`, optional `teammate_name`/`team_name`). It does NOT use `emit_pretooluse_deny`. To block task creation, emit the teammate-stop envelope and return `True`:

  ```python
  return emit_task_create_deny(reason)  # writes {"continue": false, "stopReason": reason}
  ```

  `continue: false` is what the harness honours to block (it sets `preventContinuation`); `main()` translates the `True` return into `sys.exit(2)` the same as a `PreToolUse` deny. Three `TaskCreated` gates ride this event, each the fan-out counterpart of a `PreToolUse`-only gate the fan-out bypasses: `handle_enforce_skill_loading_on_task_create` (#1488; forces the matching teatree skill onto a fanned-out task), `handle_enforce_plan_gate_on_task_create` (#1639; requires a fresh `/plan`), and `handle_dispatch_prompt_quote_scanner_on_task_create` (#171; scans the task subject/description for a HIGH verbatim user quote, the fan-out arm of `handle_dispatch_prompt_quote_scanner`). Each is fail-open / never-lockout: the skill-loading gate has kill-switch `[teatree] skill_loading_gate_enabled = false` (default on), per-call `[skip-skill-gate: <reason>]` token, and `t3 <overlay> gate skill-loading disable` self-rescue; the plan-gate and dispatch-quote gates ship default-OFF (opt-in via `[teatree] agent_plan_gate_on_task_create_enabled` / `dispatch_quote_gate_on_task_create_enabled = true`) and clear on a `[skip-plan-gate: <reason>]` / `[quote-ok: <reason>]` token respectively. All three also fail-open on a missing `session_id` and on `main()`'s per-handler exception swallow.

- **Skill-loading PreToolUse gate narrows to task-intent + has a per-call escape (#1567).** `handle_enforce_skill_loading` (matcher `Bash|Edit|Write`) hard-blocks until every *resolvable* unloaded skill in `<session>.pending` loads. The demand set must derive from genuine task-intent text only: `_build_skill_loader_input` runs the prompt through `_strip_ambient_context` first, removing the harness-injected `<system-reminder>` / `<command-*>` wrappers (the CLAUDE.md body, the MEMORY.md index, the skills listing). Without that, an ambient MEMORY.md index line containing a topic keyword (e.g. `blog`) keyword-matched the supplementary `~/.teatree-skills.yml` map and hard-blocked an unrelated autonomous loop. The gate also honours a per-call `[skill-load-ok: <reason>]` token in the current tool call's command/args (`command` for Bash, `new_string`/`content`/`file_path` for Edit/Write, first 512 chars; empty reason rejects) — modelled on `[skip-skill-gate:]` and `[fg-ok:]` — so a false trigger can never wedge the loop, while a genuine intent match still blocks every call lacking the token. Because `_strip_ambient_context` runs on every `UserPromptSubmit`, its input is length-capped to `_AMBIENT_STRIP_MAX_CHARS` (64 KiB) before the DOTALL regexes run, so a large pasted log quoting literal `<system-reminder>` open tags cannot blow the 5s timeout — preserving the fast-hook guarantee.
- **The banned-terms carve-out prefers the offline allowlist over the live probe.** `publish_surface.carve_out_applies` (the #1415 banned-terms / #126 private-repo carve-out) resolves a target repo's visibility from the `[teatree] private_repos` substring allowlist FIRST — that is the reliable, network-free, in-hook mechanism and the recommended way to declare private overlay repos. The live `gh`/`glab` visibility probe is a best-effort fallback only: the PreToolUse subprocess inherits a restricted PATH/env, so the probe resolves the tool against an augmented PATH (`_PROBE_PATH_EXTRA`) but still fails to "unknown" when the tool is absent or auth differs. Unknown ALWAYS fails closed (the target is treated as NOT private, the deny stands) — and the gate emits a one-line stderr `NOTE` (#1657) naming the unknown slug and pointing at `private_repos` so the over-block is diagnosable instead of silent. The posting-path carve-out is an ALLOWLIST (`command_is_pure_private_gh_glab_post`), not a denylist: it DOWNGRADES only when it can PROVE every top-level segment is a benign `cd`/env nav segment or a structurally-pure `gh`/`glab` post (`words[0]` EXACTLY `gh`/`glab` after a benign prefix; every token a flag / opaque flag-value / positional with no execution-transport or substitution construct) targeting a known-private repo, with at least one posting segment — and FAILS CLOSED on everything else. This INVERTS the prior detect-hidden-bad approach (six reworks of enumerating shell `-c` / `env -S` / here-string / `eval` / pipe-to-shell that leaked on every un-enumerated transport): a hidden public post now requires a second non-`gh` verb, a transport construct, or a public `--repo`, all of which fail the proof. The price is that an exotic-but-legit private post using `$()` / a pipe / a here-doc in its body or chain hard-blocks (safe and recoverable: split it into a plain post). A chained `git commit` downgrades only when its CWD is private and every chained segment is provably publish-inert or a pure private post. A `git commit -F <relpath>` body file that is unreadable from the cold hook's reset cwd is retried against the dir whose repo the commit LANDS in (`_body_file_resolution.commit_body_file_base` → the command's own `cd`/`-C`/`--git-dir`, else the harness `cwd`), so the gate scans the real body and applies the carve-out instead of fail-closing on an unread body — a body file that exists NOWHERE still fails closed, and a PUBLIC-repo body file still hard-blocks. The shared `_body_file_resolution` extractor backs the bare-reference gate (#1530) too, so both gates resolve the body file the same way.
- **Every `PreToolUse` deny routes through a repeated-denial circuit breaker.** `emit_pretooluse_deny` (the single deny chokepoint — `_fail_open_or_deny` and every gate funnel through it) calls `_apply_deny_circuit_breaker` before writing the deny. A `<session>.deny-streak` JSON file counts CONSECUTIVE identical denials (matched by a volatility-normalised `_deny_fingerprint`); a genuine ALLOW in `main()` resets it. At the K-th identical denial (`[teatree] deny_circuit_breaker_threshold`, default 3): a UX/non-safety gate on the allow-list (`_DENY_CIRCUIT_UX_GATE_PREFIXES`, the skill-loading gate at minimum) FAILS OPEN that one call to break a token-burning retry loop and resets the streak; a SAFETY gate (everything else) NEVER auto-relaxes — it keeps denying with an escalation appended to the reason. Either way a durable `loop_circuit_broken` signal is recorded to `<session>.circuit-broken` (the same per-session state-file seam the SubagentStop no-commit signal uses). Kill-switch `[teatree] deny_circuit_breaker_enabled = false` makes the breaker a pure pass-through. The breaker is crash-proof: any internal error falls back to the gate's original decision.
- **Hooks must be crash-proof and fast.** `hooks.json` sets tight timeouts (3–5s); a hook that raises or hangs blocks the user. Fail open and stay silent on the happy path.
- **Statusline state** is regenerated, not authored: `/tmp/claude-statusline/<session>.<suffix>` (override dir via `TEATREE_CLAUDE_STATUSLINE_STATE_DIR`). Never read it as a source of truth.
