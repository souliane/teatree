# hooks — local conventions

See the root [`CLAUDE.md`](../CLAUDE.md) for the code-quality bar. This file adds only what is specific to `hooks/`.

- **One router, many events.** `scripts/hook_router.py` dispatches on `--event`. Registered events: `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `TaskCreated`, `Stop`, `SubagentStop`, `PreCompact`, `SessionEnd`, `InstructionsLoaded`. Wiring lives in `hooks.json`. `SubagentStop` carries `session_id` / `cwd` / `transcript_path` and fires once per sub-agent termination — `handle_subagent_stop_no_commit` (#1205) records a `terminated_without_commit` signal when the sub-agent's worktree (the harness `cwd`) is a work branch with 0 commits, so the orchestrator does not assume lost work landed.
- **Adding a gate** = a new handler registered on an existing event (usually `PreToolUse` or `Stop`). `hook_router.py` is a module-health-capped, **shrink-only** god-module: adding the handler body to it fails `check_module_health.py` (LOC and public-function ratchets only allow the file to shrink). So put a NEW handler in a **bare sibling module** (`hooks/scripts/<concern>.py`, like `django_bootstrap` / `unknown_repo_push_gate`) and import it into the router's `_HANDLERS` chain — the router puts its own dir on `sys.path`, so a bare `from <sibling> import <handler>` resolves both as the live hook and when imported as `hooks.scripts.hook_router` in tests. If the import + one-line registration still nudges the router over its cap, move a cohesive helper OUT in the same change so the file nets smaller. To **deny** a `PreToolUse` tool call, call the shared helper and return its `True`:

  ```python
  return emit_pretooluse_deny(reason)
  ```

  The helper writes the modern nested `hookSpecificOutput` schema that Claude Code 2.1.146+ actually reads, alongside the legacy flat keys for back-compat. `main()` translates the helper's `True` return into `sys.exit(2)` — exit 0 deny payloads are silently ignored by the harness (changelog: "Fixed `PreToolUse` hooks that emit JSON to stdout and exit with code 2 not correctly blocking the tool call"). See #1447 for the diagnosis.

- **A gate only fires for tools wired into its event's matcher (#171).** `mcp__*` tools are ordinary `PreToolUse` calls merely omitted from the matcher — adding the matcher string makes them fire, e.g. `mcp__glab__glab_mr_.*` routes a glab-MR-MCP create/update through the live `handle_validate_mr_metadata` validator (same metadata check as the `glab` CLI; fail-closed with the `T3_MR_VALIDATE_ALLOW_BROKEN_ENV=1` escape). The `Agent` **tool** likewise reaches `PreToolUse` (a foreground Agent dispatch fires it with `run_in_background` in the tool_input — verified against the installed binary), and an **`Agent` PreToolUse matcher is now wired (#1646)** (the registered matchers are `Bash|Edit|Write`, `AskUserQuestion`, `mcp__.*[Ss]lack.*`, `mcp__glab__glab_mr_.*`, `Agent`). So the two `PreToolUse` `Agent` arms — `handle_dispatch_prompt_quote_scanner` and the `Agent` branch of `handle_enforce_orchestrator_boundary` → `_deny_foreground_agent_dispatch` (#1442) — are now **LIVE**. The orchestrator-boundary deny is **default-ON** (#1733: `[teatree] orchestrator_boundary_agent_gate_enabled` defaults True after its attended pre-install dry-run #1646). Every off-ramp keeps it never-lockout: sub-agent context, `run_in_background: true`, a per-call `[fg-ok: <reason>]` prompt token, the `= false` kill-switch, the deny-circuit-breaker, AND — via `_fail_open_or_deny` (#1692) — the self-rescue allowlist + the master `danger_gate_fail_open` switch. Do NOT conflate this with the SEPARATE `Task`/`Workflow` fan-out vehicle (next bullet), which genuinely bypasses `PreToolUse` and fires `TaskCreated` — that is why the dispatch-quote concern ALSO carries a reachable `TaskCreated` counterpart (the `TaskCreated` schema has no `run_in_background`, so the orchestrator-boundary arm has no such counterpart and the Agent-matcher path is its only fix).

- **The MR-metadata gate watches every surface that sets a title/description, not just the inline create command ([#1703](https://github.com/souliane/teatree/issues/1703)).** `_extract_mr_fields` covers four: (1) inline `glab mr create/update --title/--description`; (2) the SAME command's file-based / heredoc description (`-F`/`--description-file`), read via `_read_message_file` rather than passed through as a falsely-empty (trivially "valid") string — but a double-quoted `$(…)`/`${…}`/`$VAR`/backtick value the hook cannot resolve before shell expansion is SKIPPED (`_looks_dynamic_value`), never validated as the truncated literal fragment (e.g. `$(cat` from `--description "$(cat "$F")"`; the remote CI gate is the backstop); (3) out-of-band `glab api`/`gh api` PUT/POST to a `.../merge_requests` or `.../pulls` endpoint (`--field title=…`/`-f description=…`/`-f body=…`) — the web-UI-equivalent edit that bypasses the create/update CLI entirely, and the GitHub PR-create path (`gh api repos/<o>/<r>/pulls`); (4) the `mcp__glab__glab_mr_*` MCP tools. The `gh pr create/edit` CLI is deliberately NOT a separate metadata surface: it is already governed by the AI-signature gate (`handle_block_ai_signature`) in the same PreToolUse chain, and double-gating would let the metadata deny preempt that gate's body scan. Both the out-of-band API path AND the `glab mr update` CLI path (`extract_cli_mr_fields` in the sibling module `mr_cli_fields`) validate ONLY the fields the command actually sets: a partial edit touching neither title nor description (e.g. `--field state_event=close`, or `glab mr update --reviewer X`) is skipped (never-lockout), and a single-field edit back-fills the untouched one (a valid `type(scope): … (ticket_url)` line is, by the canonical grammar, simultaneously a valid title and a valid description first line; an unset description on a title-only update is back-filled with a known-good `## What` body) so it can never false-block on `Title is empty.` or a missing What/Why. `glab mr create` keeps the stricter both-fields contract. This closed the gap where a non-compliant description set via a REST-API field edit reached the remote unvalidated and then failed the remote MR-title/description CI job. **The CLI verb is detected against the command with quoted spans and heredoc bodies stripped (`strip_quoted_and_heredoc`)**, so the phrase merely embedded in a `git commit -m '… glab mr create …'` message, a doc string, or a `<<EOF`/`-` heredoc body no longer false-fires the gate (the residual: a real create/update fed *through* a stripped span — `bash -c "glab mr create …"` or a heredoc piped to a shell — is a backstopped false-negative). The `T3_MR_VALIDATE_ALLOW_BROKEN_ENV=1` escape and `T3_MR_VALIDATE_SCRIPT` override are unchanged.
- **NEVER directly assign a reviewer — `handle_block_self_reviewer_assign` blocks every reviewer-assignment surface.** Reviewers must NEVER be set directly on an MR/PR — least of all the user's OWN MR (this happened and is forbidden). Review is REQUESTED via the Slack/approval channel only (`review-request post`); there is no sanctioned direct-assignment path. The gate (a bare sibling module `hooks/scripts/no_self_reviewer_assign.py`, registered in the `PreToolUse` chain right after `handle_validate_mr_metadata`) blocks every reviewer-assignment surface: the CLI `glab mr create/update <iid> --reviewer/--reviewers <user>` (the update surface drove the bug; create assigns at creation), the GitHub CLI `gh pr create --reviewer/-r` and `gh pr edit --add-reviewer/--reviewer`, an out-of-band `glab api`/`gh api` WRITE setting `reviewer_ids`/`reviewers`/`requested_reviewers` on a `merge_requests`/`pulls` endpoint (a GET *read* of the same field — e.g. `gh api .../requested_reviewers` to LIST them — is allowed; the block is gated on the effective HTTP method), and the `mcp__glab__glab_mr_update`/`mcp__glab__glab_mr_create` MCP tools carrying a reviewer arg. It is SEPARATE from the MR-metadata gate, which deliberately SKIPS a `--reviewer` update (it validates only title/description — never-lockout) and so never saw the bug. The verb is detected on the quote/heredoc-stripped skeleton (reusing `mr_cli_fields.strip_quoted_and_heredoc`) so the phrase inside a commit message / doc string does not false-fire. Never-lockout: a per-call `[reviewer-ok: <reason>]` token (a vetted one-off on a colleague's MR) and the `[teatree] no_self_reviewer_assign_gate_enabled = false` kill-switch both ALLOW; the deny routes through `_fail_open_or_deny`.
- **The MR-metadata gate keys overlay resolution to the MR's TARGET repo, not the agent's cwd.** `mr_cli_fields.extract_mr_target_repo` parses the target slug from the same command — `-R`/`--repo <slug>` on `glab mr`, the URL-decoded namespace of a `glab api .../projects/<ns>/merge_requests…`, or the `owner/repo` after `repos/` in a `gh api repos/<o>/<r>/pulls…` — and `handle_validate_mr_metadata` forwards it as `t3 tool validate-mr --repo <slug>`. Without it, an MR targeting an overlay with stricter title rules, created with cwd in a repo owned by a more-lenient overlay (the teatree clone a dispatched agent inherits), was graded against the CWD overlay's weaker rules, and the any-overlay-pass fallback let it through because the lenient overlay accepted it. When the target maps to exactly one registered overlay, `validate-mr` validates ONLY against that overlay with NO any-overlay-pass fallback, and a crash in that overlay's validator FAILS CLOSED (deny) rather than skipping. A blank or unmatched `--repo` (no target flag / unknown slug) falls back to the established cwd-keyed resolution, so today's behaviour is unchanged when the MR targets the cwd repo.

- **`TaskCreated` is the only non-`PreToolUse` deny path, with a DIFFERENT schema (#1488).** The harness Workflow/Task fan-out vehicle (what `ultracode` and any teammate spawn use) **bypasses `PreToolUse`/`PostToolUse` hooks** — so a gate that must govern the fan-out rides the `TaskCreated` event instead (no matcher, fires on every task creation; stdin schema `session_id`, `task_id`, `task_subject`, `task_description`, optional `teammate_name`/`team_name`). It does NOT use `emit_pretooluse_deny`. To block task creation, emit the teammate-stop envelope and return `True`:

  ```python
  return emit_task_create_deny(reason)  # writes {"continue": false, "stopReason": reason}
  ```

  `continue: false` is what the harness honours to block (it sets `preventContinuation`); `main()` translates the `True` return into `sys.exit(2)` the same as a `PreToolUse` deny. Two `TaskCreated` gates ride this event, each the fan-out counterpart of a `PreToolUse`-only gate the fan-out bypasses: `handle_enforce_skill_loading_on_task_create` (#1488; forces the matching teatree skill onto a fanned-out task) and `handle_dispatch_prompt_quote_scanner_on_task_create` (#171; scans the task subject/description for a HIGH verbatim user quote, the fan-out arm of `handle_dispatch_prompt_quote_scanner`). Each is fail-open / never-lockout: the skill-loading gate has kill-switch `[teatree] skill_loading_gate_enabled = false` (default on), per-call `[skip-skill-gate: <reason>]` token, and `t3 <overlay> gate skill-loading disable` self-rescue; the dispatch-quote gate ships default-OFF (opt-in via `[teatree] dispatch_quote_gate_on_task_create_enabled = true`) and clears on a `[quote-ok: <reason>]` token. Both also fail-open on a missing `session_id`. NOTE for `TaskCreated`: `main()`'s per-handler exception swallow is NOT a safe backstop here — it writes a stderr traceback, and the harness aborts `TaskCreated` on ANY handler stderr (a lockout, not a fail-open). So each `TaskCreated` handler must catch its own exceptions and fail OPEN in-handler (the skill-loading gate does this via `unreferenced_demand_reason`'s guard + `is_file_safe`, so a 255+ byte pending name's `OSError` never reaches `main()`).

- **Skill-loading PreToolUse gate narrows to task-intent + has a per-call escape (#1567).** `handle_enforce_skill_loading` (matcher `Bash|Edit|Write`) hard-blocks until every *resolvable* unloaded skill in `<session>.pending` loads. The demand set must derive from genuine task-intent text only: `_build_skill_loader_input` runs the prompt through `_strip_ambient_context` first, removing the harness-injected `<system-reminder>` / `<command-*>` wrappers (the CLAUDE.md body, the MEMORY.md index, the skills listing). Without that, an ambient MEMORY.md index line containing a topic keyword (e.g. `blog`) keyword-matched the supplementary `~/.teatree-skills.yml` map and hard-blocked an unrelated autonomous loop. The gate also honours a per-call `[skill-load-ok: <reason>]` token in the current tool call's command/args (`command` for Bash, `new_string`/`content`/`file_path` for Edit/Write, first 512 chars; empty reason rejects) — modelled on `[skip-skill-gate:]` and `[fg-ok:]` — so a false trigger can never wedge the loop, while a genuine intent match still blocks every call lacking the token. Because `_strip_ambient_context` runs on every `UserPromptSubmit`, its input is length-capped to `_AMBIENT_STRIP_MAX_CHARS` (64 KiB) before the DOTALL regexes run, so a large pasted log quoting literal `<system-reminder>` open tags cannot blow the 5s timeout — preserving the fast-hook guarantee.
- **Plan-before-code edit-block gate has three never-lockout escapes.** `handle_block_edit_before_planned` (`PreToolUse`, matcher `Edit|Write`) denies file edits when the cwd's worktree ticket is still in `STARTED` FSM state — enforcing that a plan must be recorded before coding begins. Three escapes preserve the never-lockout contract: (1) **per-call token** — place `[skip-plan-gate: <reason>]` in the Edit `new_string`/Write `content`/either `file_path` (first 512 chars; empty reason rejects) to ALLOW that single call only; (2) **config kill-switch** — `[teatree] plan_edit_gate_enabled = false` in `~/.teatree.toml` disables the gate globally (default on, fails OPEN to enabled on missing/broken config); (3) **self-rescue CLI** — `t3 <overlay> gate plan disable` flips the kill-switch without touching the overlay machinery (same surface as `gate skill-loading`). The ticket state is resolved via `_ticket_state_for_cwd` → git toplevel → `Worktree.extra['worktree_path']` → `Ticket.state`; a non-STARTED state, a missing ticket, or any resolver error all fail OPEN. The deny itself routes through `_fail_open_or_deny` (the master fail-open switch and the usual circuit-breaker apply). The `[skip-plan-gate: <reason>]` token is the per-edit escape for *this hook*; the FSM-level plan gate (`check_plan_artifact`, the `plan()` transition) has its own lightweight escape for a trivial ticket — `t3 <overlay> ticket skip-planning <id> --reason "<why>"` records an audited `trivial_plan_skip` marker that advances STARTED → PLANNED with no `PlanArtifact` (the heavyweight sibling, `ticket plan-bypass --human-authorize`, stays for the no-real-plan-but-needs-sign-off case). Once PLANNED, edits are no longer blocked by this hook.
- **Supplementary keyword skills are advisory, never a hard demand (#1683).** The supplementary `~/.teatree-skills.yml` map is loose free-form regex (`\bruff\b` → `ac-adopting-ruff`) with no priority/exclude/word-boundary discipline, so a bare token in genuine prompt text ("run ruff check") over-fires even after the #1567 ambient strip. `suggest_skills` returns an `advisory` key — the suggestions sourced ONLY from the supplementary config (`SkillSelectionResult.advisory_skills`, computed as the suggestion set minus the intent/framework/overlay/companion `hard_resolved` set). `handle_user_prompt_submit` still PRINTS them in the "LOAD THESE SKILLS" message but writes only the non-advisory subset to `<session>.pending`, so the PreToolUse gate never hard-blocks a `Bash`/`Edit`/`Write` on an incidental supplementary keyword. A skill that is ALSO an intent/framework match stays in `hard_resolved` and keeps enforcing load-first; only supplementary-only skills are demoted.
- **The suggester + the engagement seam are default-OFF (#256).** `handle_user_prompt_submit` early-returns (after clearing `<session>.pending`) when `_teatree_engaged(session_id)` is false, so a fresh, not-engaged session gets NO skill suggestion, NO `.pending` write, and NO T3 CLI reminder — and the empty pending means the PreToolUse skill-loading gate has nothing to demand (never-lockout, pinned by `tests/test_lockout_regression_corpus.py::TestDefaultOffSessionNeverHardBlocks`). `_teatree_engaged` = `_autoload_enabled() OR _teatree_active(session) OR <session>.t3-engaged`: the owner's `[teatree] autoload` flip (cold-read by `teatree_settings.autoload_enabled`, env `T3_AUTOLOAD` first), a teatree-requiring skill loaded (`.teatree-active`), or **any** `t3:` skill loaded (`.t3-engaged`, set by `_maybe_engage_t3` in `handle_track_skill_usage` from the ORIGINAL `t3:`-namespaced token). The two markers are deliberately distinct: `.t3-engaged` engages only the suggester, while the loop machinery (`_loop_auto_load_active`) consults exclusively `.teatree-active`, so a plain lifecycle skill never arms loops. `handle_session_start_bootstrap` touches `.teatree-active` when `autoload` is ON (so the loop bootstrap fires unchanged) and otherwise emits the one-line how-to advisory (`_TEATREE_NOT_ACTIVE_ADVISORY`) on a FRESH start while still merging on a compact/resume so recovery context is preserved.
- **The banned-terms carve-out prefers the offline allowlist over the live probe.** `publish_surface.carve_out_applies` (the #1415 banned-terms / #126 private-repo carve-out) resolves a target repo's visibility from the `[teatree] private_repos` substring allowlist FIRST — that is the reliable, network-free, in-hook mechanism and the recommended way to declare private overlay repos. The allowlist match (`_repo_visibility.slug_is_allowlisted_private`) is HOST-QUALIFICATION-SYMMETRIC (#2067): an entry may be written bare (`owner/repo`) or host-qualified (`host/owner/repo`, the form a repo URL and `slug_for_cwd` carry), and either form matches both the commit surface (host-qualified cwd slug) and the pr-create surface (the BARE `owner/repo` slug `gh pr create --repo` supplies). Without the symmetry, a host-qualified entry downgraded a private repo's own commit but hard-blocked its `gh pr create` — the same repo classified two ways across surfaces. The live `gh`/`glab` visibility probe is a best-effort fallback only: the PreToolUse subprocess inherits a restricted PATH/env, so the probe resolves the tool against an augmented PATH (`_PROBE_PATH_EXTRA`) but still fails to "unknown" when the tool is absent or auth differs. Unknown ALWAYS fails closed (the target is treated as NOT private, the deny stands) — and the gate emits a one-line stderr `NOTE` (#1657) naming the unknown slug and pointing at `private_repos` so the over-block is diagnosable instead of silent. The posting-path carve-out is an ALLOWLIST (`command_is_pure_private_gh_glab_post`), not a denylist: it DOWNGRADES only when it can PROVE every top-level segment is a benign `cd`/env nav segment or a structurally-pure `gh`/`glab` post (`words[0]` EXACTLY `gh`/`glab` after a benign prefix; every token a flag / opaque flag-value / positional with no execution-transport or substitution construct) targeting a known-private repo, with at least one posting segment — and FAILS CLOSED on everything else. This INVERTS the prior detect-hidden-bad approach (six reworks of enumerating shell `-c` / `env -S` / here-string / `eval` / pipe-to-shell that leaked on every un-enumerated transport): a hidden public post now requires a second non-`gh` verb, a transport construct, or a public `--repo`, all of which fail the proof. The price is that an exotic-but-legit private post using `$()` / a pipe / a here-doc in its body or chain hard-blocks (safe and recoverable: split it into a plain post). A chained `git commit` downgrades only when its CWD is private and every chained segment is provably publish-inert or a pure private post. A `git commit -F <relpath>` body file that is unreadable from the cold hook's reset cwd is retried against the dir whose repo the commit LANDS in (`_body_file_resolution.commit_body_file_base` → the command's own `cd`/`-C`/`--git-dir`, else the harness `cwd`), so the gate scans the real body and applies the carve-out instead of fail-closing on an unread body. For a `git commit` the unreadable-body case is no longer a hard-block: a `git commit -F -` stdin / heredoc / `printf|git commit -F -` body is RESOLVED and scanned (so a real banned term in a readable commit body still blocks), and a genuinely-unreadable body (a missing `-F <file>`, an opaque `cat|git commit -F -`, a `-m "$VAR"` not in env) DOWNGRADES to a warn regardless of the landing repo's visibility — PUBLIC included — because a commit is LOCAL and the pre-push gate (`refuse-public-push-with-leak.sh`, #703) re-scans every commit message before a public push (`command_targets_local_commit`, #1415; the prior provably-public hard-block stuck multiple coders mid-commit). The chained-segment proof keeps a commit chained to a real PUBLIC `gh`/`glab` post hard-blocked, and a `gh`/`glab` POST (no push gate behind it) still hard-blocks an unreadable PUBLIC body file. The shared `_body_file_resolution` extractor backs the bare-reference gate (#1530) too, so both gates resolve the body file the same way. The own-slug downgrade (`term_is_own_repo_slug`, #1951) matches by token-CONTAINMENT, not just whole-slug equality: the org PREFIX token of a multi-token private slug (`acmecorp` of `acmecorp-engineering`) qualifies, since a work-item URL `host/acmecorp-engineering/.../-/issues/N` tokenizes the prefix out of the repo's OWN identity (#1958), while a SUPERSET term (`acmecorp-engineering-services`) is longer than the entry's token run and stays blocked. The own-slug downgrade still requires a PROVABLY-PRIVATE landing repo (it runs `commit_branch_downgrades`): a bare `git commit` lands wherever git runs (the cwd), not where an `-F` body file lives, so a private body file must never vouch for a divergent landing repo's visibility. A SEPARATE URL-derived dimension covers a post to a PUBLIC surface whose body cites the address of a configured private repo: `own_repo_url_carve_out.term_only_inside_own_repo_urls` blanks every forge URL whose host+namespace matches a `private_repos` entry (host-qualification-symmetric) and downgrades to a warning ONLY when the term then has NO remaining occurrence — so the structurally-required URL of an own work item is not a leak, while a bare term outside such a URL (or a term in a foreign URL) still hard-blocks. It fires only for the structured `gh`/`glab` posting verbs (`is_gh_glab_posting_command`), never for `git commit` (the commit path keeps its own landing-repo carve-out above), so a public-landing commit citing the same URL stays blocked.
- **Every `PreToolUse` deny routes through a repeated-denial circuit breaker.** `emit_pretooluse_deny` (the single deny chokepoint — `_fail_open_or_deny` and every gate funnel through it) calls `apply_deny_circuit_breaker` (the bare sibling module `hooks/scripts/deny_circuit_breaker.py`, re-exported into the router as `_apply_deny_circuit_breaker`; #2384 Wave-2 router split, PR3) before writing the deny. A `<session>.deny-streak` JSON file counts CONSECUTIVE identical denials (matched by a volatility-normalised `_deny_fingerprint`); a genuine ALLOW in `main()` resets it via the re-exported `reset_deny_streak`. At the K-th identical denial (`[teatree] deny_circuit_breaker_threshold`, default 3): a UX/non-safety gate on the allow-list (`_DENY_CIRCUIT_UX_GATE_PREFIXES`, the skill-loading gate at minimum) FAILS OPEN that one call to break a token-burning retry loop and resets the streak; a SAFETY gate (everything else) NEVER auto-relaxes — it keeps denying with an escalation appended to the reason. Either way a durable `loop_circuit_broken` signal is recorded to `<session>.circuit-broken` (the same per-session state-file seam the SubagentStop no-commit signal uses). Kill-switch `[teatree] deny_circuit_breaker_enabled = false` makes the breaker a pure pass-through. The breaker is crash-proof: any internal error falls back to the gate's original decision.
- **Hooks must be crash-proof and fast.** `hooks.json` sets tight timeouts (3–5s); a hook that raises or hangs blocks the user. Fail open and stay silent on the happy path.
- **Statusline state** is regenerated, not authored: `/tmp/claude-statusline/<session>.<suffix>` (override dir via `TEATREE_CLAUDE_STATUSLINE_STATE_DIR`). Never read it as a source of truth.
