# Prerequisites

> Load when setting up a new machine, checking tool availability, or debugging missing dependencies.

---

## Platform Support

macOS and Linux. Shell scripts use `#!/usr/bin/env bash` for portability — never hardcode a platform-specific path like `/opt/homebrew/bin/bash`.

Scripts that need Bash 4+ features (associative arrays) include a version guard that auto-detects and re-execs with a modern bash from well-known locations (`/opt/homebrew/bin/bash`, `/usr/local/bin/bash`, `/home/linuxbrew/.linuxbrew/bin/bash`). Use platform-agnostic alternatives for common utilities: `md5sum` on Linux vs `md5 -q` on macOS, `stat -c '%Y'` on Linux vs `stat -f '%m'` on macOS.

## Required Tools

- `bash` 5+ (macOS: `brew install bash` and ensure it's first on `$PATH`; system `/bin/bash` is 3.x)
- `zsh` or `bash` (scripts are shell functions sourced into the current shell)
- `git` (worktree lifecycle, branch detection, cleanup)
- `docker` with `docker compose` plugin
- PostgreSQL client CLIs: `psql`, `createdb`, `dropdb`, `pg_restore`
- `direnv` (required by `t3_setup` and environment loading helpers)
- `lsof` (port checks)

## Optional Tools

- `dslr` (fast DB snapshot/restore path)
- `uv` (used only to auto-install `dslr` when missing)
- `jq` (required only by agent platform integration scripts, e.g., statusline hooks)
- `md5sum` or `md5` (dirty-state hashing in statusline; auto-detected per platform)
- Agent platform hook support (e.g., Claude Code hooks in `integrations/claude-code-statusline/*`)

## Shell Script Portability Checklist

When writing or modifying shell scripts, verify:

1. **Shebang:** `#!/usr/bin/env bash` — never a hardcoded path.
2. **Bash version features:** If the script uses `declare -A` (associative arrays), add the Bash 4+ version guard with re-exec (see `statusline-command.sh` for the pattern).
3. **macOS-only commands:** Replace or wrap with platform detection:
    - `md5 -q` → check `command -v md5sum` first
    - `stat -f '%m'` → use `stat -c '%Y'` on Linux
    - `open` → fall back to `xdg-open`
    - `pbcopy`/`pbpaste` → fall back to `xclip` or `xsel`
4. **Smoke test:** After modifying a script, run it end-to-end — linting alone does not catch runtime failures.

## External Files Expected

- `$T3_WORKSPACE_DIR/<repo>/.git` (required; workspace repositories this skill manages)
- `$T3_WORKSPACE_DIR/<repo>/docker-compose.yml` (optional; used by `_wt_services`)
- `$T3_WORKSPACE_DIR/<repo>/manage.py` (optional; enables Django framework auto-detection)
- `$T3_WORKSPACE_DIR/<repo>/.venv`, `$T3_WORKSPACE_DIR/<repo>/.python-version`, `$T3_WORKSPACE_DIR/<repo>/.data`, `$T3_WORKSPACE_DIR/<repo>/node_modules` (optional; shared into worktrees when present)
- `$T3_WORKSPACE_DIR/<ticket-dir>/.env.worktree` (generated by `t3_setup`, then used by subsequent commands)

## User Configuration

Configuration is stored in `~/.teatree` (a sourceable shell file). See `t3-setup` for the interactive setup wizard.

Environment variables with auto-detected defaults:

| Variable | Default | Purpose |
|---|---|---|
| `T3_WORKSPACE_DIR` | `~/workspace` | Root directory containing all repos |
| `T3_BRANCH_PREFIX` | Lowercase initials from `git config user.name` (fallback: `wt`) | Directory + branch name prefix |
| `T3_ISSUE_TRACKER` | Auto-detected from `git remote -v` | Issue/MR platform (`gitlab`, `github`) |
| `T3_CHAT_PLATFORM` | `none` | Team chat for review requests (`slack`, `teams`, `none`) |
| `DJANGO_SUPERUSER_EMAIL` | `$(git config user.email)` | Django superuser email |
| `DJANGO_SUPERUSER_PASSWORD` | `test` | Django superuser password |
