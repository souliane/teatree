## 1. What TeaTree Is

A personal code factory for multi-repo projects. It turns a ticket URL into a merged pull request by coordinating the full lifecycle ‚Äî intake, coding, testing, review, shipping, delivery ‚Äî across multiple repositories, worktrees, and agent sessions.

**Target:** service-oriented projects with databases and CI pipelines (any language). Not for docs-only repos or CLI tools.

**Operating mode.** TeaTree runs as a long-lived interactive Claude Code session orchestrated by a fat `/loop` (~10‚Äì15 min cadence). The loop fans out to in-session subagents per tick to sweep PRs, auto-review PRs assigned to the user, intake assigned issues, watch messaging mentions/DMs, and render a multi-line statusline that is the **only persistent UI surface**. There is no HTML dashboard. The loop runs in the same session the user types into, so debugging stays direct.

**Code-host neutrality.** Pull requests are the canonical concept. Both **GitHub** and **GitLab** are first-class in core; GitLab MRs map onto the PR abstraction at the Protocol layer.

**Messaging-backend pluggability.** Mentions, DMs, and outgoing posts go through a `MessagingBackend` Protocol declared per overlay. Slack (Socket Mode bot) is the first implementation. A `Noop` default lets overlays opt out.

**Core principle.** Infrastructure is deterministic code; development work is skill-guided. State management, port allocation, provisioning, task routing, code-host sync, and messaging integration are Python code with >90% branch coverage. The actual development ‚Äî coding with TDD, debugging, reviewing, shipping ‚Äî is driven by skills that encode methodology, guardrails, and domain knowledge.

**Core stays generic.** No customer-, tenant-, or product-specific names appear in the scanned roots (`src/teatree`, `docs`, `skills`, `agents`, `tests`, `scripts` + top-level `README.md`/`BLUEPRINT.md`/`AGENTS.md`); per-overlay specifics live in the overlay package and `~/.teatree.toml`. A gate (`scripts/hooks/check_no_overlay_leak.py`) enforces this ‚Äî cited as `BLUEPRINT ¬ß 1` from `tests/test_no_overlay_leak_hook.py`. It runs two passes: the configured term list (whole-token via `teatree.hooks.term_match`) and an always-on opaque-ID pass (`teatree.hooks.opaque_id`) flagging a real-shaped Slack/forge id (`C0‚Ä¶`/`U0‚Ä¶`/`T0‚Ä¶`) ‚Äî a leak class with no dictionary word ‚Äî while an obviously-synthetic placeholder and any `leak-scan:allow` line are exempt. As a pre-commit hook it sees only staged files; the full-tree CI job `overlay-leak-tree` walks every root and passes `--require-terms`, making a missing `TEATREE_OVERLAY_LEAK_TERMS` secret a LOUD MISCONFIGURED failure (exit 2) rather than a silent no-op.

## 2. Architecture Principle: Code-First, Not Skills-First

Infrastructure and orchestration are code; development methodology is skill-guided prose. The split is load-bearing:

1. **Skills are prose, not code.** Prose produces different results depending on model, context, and what else is loaded. Anything that must be deterministic ‚Äî state transitions, port allocation, provisioning, sync, the `/loop` tick ‚Äî is code.
2. **Coordination needs transactional guarantees.** An FSM + ORM provide atomic transitions and row-locked workers. Coordination through JSON files cannot.
3. **Code is testable; prose is not.** Core logic must reach >90% branch coverage.
4. **One ABC with a handful of methods beats thirty thin extension points.** Overlay customization goes through `OverlayBase` ‚Äî typed methods with defaults, no priority system, no plugin registries.
5. **Terminal today, SDK/API-portable by design.** Teatree is driven through Claude Code's interactive terminal app today, but every architectural decision is made so the same flow can be driven through the Anthropic Agent SDK / API. Don't adopt a mechanism that only works in the terminal CLI when an SDK/API-portable equivalent exists. The terminal-driven flow keeps working; SDK/API-portability is the binding constraint when the two pull apart. This is why the headless executor (`agents/headless.py`) is kept deliberately slim ‚Äî it is the swap point for an SDK runtime ‚Äî and why dispatch, state, and skill-loading route through code rather than terminal-only affordances.

Five core lifecycle models in `teatree.core.models/`, all FSM-driven (`django-fsm`): **Ticket**, **Worktree**, **Session**, **Task**, **TaskAttempt**. Around them, supporting rows under the same package: `AttachmentManifest` (the intake attachment-fetch manifest, below), `BotPing`, `ConsolidatedMemory`/`DreamQaProbe`/`DreamRunMarker`/`InstructionComplianceSnapshot`/`InstructionComplianceRecord` (memory-consolidation, below), `DailyDigestThread`/`DailyDigestMessage`, `DbApproval`, `DeferredQuestion`, `EvalRunRecord`/`EvalScenarioResult` (eval-run ledger), `SessionAuditRecord` (eval audit layer), `HonestyEscalation`, `IncomingEvent`, `IntentClassification`, `LandscapeArtifact` (the intake landscape survey the planner consumes, below), `LivePostApproval`, `LocalStackQueueItem`, `LocalStackReaperMarker`, `Loop` (DB-configured autonomous loop ‚Äî name, exactly one of prompt-FK/script, run_in_sub_agent/description/overlay, nullable delay, enabled, ¬ß5.6), `LoopLease`, `LoopState` (durable per-loop control plane, ¬ß5.6), `MergeClear`, `OnBehalfApproval`, `OutboundClaim` (drift verifier), `PendingChatInjection`, `ProjectLearning` (durable per-repo knowledge store, below), `Prompt`/`PromptVersion` (first-class reusable prompt with templated `params` + edit-history snapshots, the `Loop.prompt` FK target and the `/prompts` trigger, ¬ß5.6), `PullRequest`, `RedCardSignal`, `ReplyDispatch`, `ReviewAssignment`, `ReviewEvidence` (per-ticket review-evidence artifact ‚Äî cold-review + integration-review kinds ‚Äî gating review-request and cross-repo close, below), `ReviewLoop`/`ReviewLoopRound` (the bounded author‚Üîreviewer ping-pong FSM + its round-leg slot, ¬ß5.6), `ReviewRequestPost`, `ReviewVerdict`, `ScannedBroadcast`, `SelfImproveFiring`, `SessionHandover`, `TicketTransition`, `WaitingItem` (the manual entries of the durable waiting-on-you lane ‚Äî questions/merge-authorizations/review-requests are computed live, ¬ß17.1), `WorktreeEnvOverride`. Every row name above is cited by name from ¬ß5.6, ¬ß17.1, ¬ß17.4, or ¬ß4 prose.

**Memory-consolidation data layer ([#1933](https://github.com/souliane/teatree/issues/1933)).** Three durable rows back the idle-time "dreaming" pass that clusters session feedback and distils each cluster into one imperative rule. `ConsolidatedMemory` is the consolidation ledger (the DB row is canonical state + audit trail; a rendered topic file is the output), idempotency-anchored on a sha256 `cluster_key` and owning its status ladder `CANDIDATE ‚Üí VERIFIED ‚Üí PROMOTED` (plus `SUPERSEDED`/`EXPIRED`); it refuses an uncited rule, refuses to silently expire a BINDING row, and gates index-line pruning on a terminal status with a recorded durable home. **Pass 2 ‚Äî memory‚Üífix-and-merge ([#2426](https://github.com/souliane/teatree/issues/2426), [#2663](https://github.com/souliane/teatree/issues/2663)).** A `disposition` ladder (`UNTRIAGED ‚Üí USER_SPECIFIC_KEEP | (CORE_GAP_NEEDS_TICKET ‚Üí TICKETED ‚Üí RESOLVED_RETIRED)`) makes the ledger a queue that DRAINS, not a pile that grows: a generic/teatree-core memory is a confession that core has a workflow gap, so `teatree.loops.dream.promote_memory` triages each untriaged row (injected `MemoryClassifier`, default the `durable_destination`-hint one ‚Äî a home under teatree's own skills/source/scripts/BLUEPRINT is core-generic, anything else stays user-specific memory). The promote verb no longer files a fresh `needs-triage` issue per gap ‚Äî those PILED UP because the `IssueImplementerScanner` SKIPs `needs-triage`. Instead each core gap is driven to a MERGED fix via `teatree.loops.dream.umbrella_ledger`: a CHECKBOX is upserted under the standing umbrella issue ([#2663](https://github.com/souliane/teatree/issues/2663) ‚Äî reused daily, NEVER closed), keyed on an invisible `<!-- dream-gap <cluster_key> -->` marker so the same gap never double-adds (banned-term + bare-reference scanned so the public-repo title never leaks), and the fix is SCHEDULED for a coding agent by reusing `Ticket.schedule_coding()` ‚Äî the PR merges through the SAME ¬ß17.4 keystone gated by the overlay's autonomy. The new `CodeHostBackend.update_issue(issue_url, body)` extension method edits the umbrella body in place (GitHub `PATCH .../issues/<n>`, GitLab `PUT .../issues/<iid>`). When a gap's fix Ticket reaches MERGED, `reconcile_merged_gaps` CHECKS its umbrella checkbox and retires the prose (`RESOLVED_RETIRED`, archived) through the existing `retire_resolved_memories` (driven off the Ticket's authoritative MERGED state via an injected `is_resolved` predicate, not a fragile forge re-read of a `/pull/<n>` URL). **No new model:** an in-flight gap is an existing `Ticket` row (gap/memory/umbrella linkage in `extra.dream_gap_key`/`dream_memory_cluster_key`/`dream_umbrella_url`) + its `ConsolidatedMemory` entry; the #2663 checkbox is the durable cross-night state. A BINDING row is never retired. Pass 2 schedules the fix (durable, reversible) but never hand-implements; it is **default OFF** behind the `[loops.dream] memory_promote` / `T3_DREAM_MEMORY_PROMOTE` toggle, so an unflagged pass never triages, upserts, or schedules. **Phase 3c ‚Äî the Instruction-Compliance Accountant ([#2663](https://github.com/souliane/teatree/issues/2663)).** Instruction compliance is the root KPI: a rule was PRESENT/AVAILABLE (a durable memory, a loaded-skill rule, a CLAUDE.md clause, a system gate, or an explicit in-session directive) and the agent acted against it. `teatree.loops.dream.compliance` mines one pass's extract (reusing `transcript_extract.looks_like_user_correction` as the ground-truth signal) into typed `ComplianceFinding`s ‚Äî a correction overlapping a memory-backed rule already on disk is a RECURRENCE (`rule_source=MEMORY`), one with no backing memory a first-occurrence `IN_SESSION` violation ‚Äî and persists one `InstructionComplianceSnapshot` per pass (`instructions_observed`/`violations`/`compliance_rate`/`recurrences_count`) plus one `InstructionComplianceRecord` audit row per violation. **The binding escalation rule:** a recurrence (a rule that already had a durable memory, violated AGAIN) must escalate to a gate or an eval, NEVER another memory ‚Äî `escalate_recurrences` drives ONE umbrella checkbox + a scheduled fix per recurring rule (deduped on the `compliance-recurrence-<rule_identity>` gap key) via the same `umbrella_ledger.promote_gap` path Pass 2 uses, the checkbox title prescribing a structural fix (a PreToolUse/Stop gate, a config self-check, or an anti-vacuous `under_load` eval), never more prose and no longer a fresh `needs-triage` issue; and `write_clusters` reclassifies a MEMORY_ONLY cluster whose rule already shows a recurrence off its memory destination to a core-gap home rather than re-promoting another memory. The phase runs under `t3 dream run --full`, **default OFF** behind the `[loops.dream] compliance` / `T3_DREAM_COMPLIANCE` toggle (the Pass-2 posture); `t3 dream compliance show` prints the latest snapshot (rate + recurrence count + open escalations). **Phase 3d ‚Äî the Automatable-Ask Promoter ([#2663](https://github.com/souliane/teatree/issues/2663)).** The structural "improve-with-new-stuff" sibling of the compliance accountant (the "fix-issues" half): dreaming detects recurring MANUAL user asks t3 could automate and promotes each as a fix-and-merge so the user "gets out of the loop", expressed ENTIRELY within the memory verbs (consolidate ‚Üí merge ‚Üí promote ‚Üí retire), **no new model**. `transcript_extract.looks_like_user_ask` is the keyword-blind keeper (the sibling of `looks_like_user_correction`): a USER turn reading like a directive/request (imperative cues "can you"/"please"/"let's"/"set up", or operational-ACTION cues "hotfix"/"asap"/"rollback" ‚Äî bare incident-state words "production"/"broken"/"blocker"/"wedged" are excluded, [#2732](https://github.com/souliane/teatree/issues/2732), as they describe a situation not a request); it rides `high_signal_lines` (and the narrow `user_ask_lines` sibling) so repeated asks reach the distiller and cluster into `ConsolidatedMemory` ask-clusters (same `cluster_key` upsert dedups/merges for free). `teatree.loops.dream.automation_ask` classifies each ask-cluster Bucket A (`EXISTING_GAP` ‚Äî an existing loop/skill SHOULD have handled it; the injected `AUTOMATION_CATALOG` of loop+skill names lets the classifier name the real mechanism, and the gap prescribes fixing its trigger/coverage/surfacing) or Bucket B (`NEW_WORKFLOW` ‚Äî no automation exists; prescribe a new loop/skill/gate, canonical example a hotfix lane), via an injected `AskClassifier` seam (default the deterministic catalog classifier, so the phase runs LLM-free). Each GROUNDED ask gap (the verbatim cited snippet must appear in the transcript ‚Äî reusing `engine.cluster_is_grounded`) routes through the SAME `umbrella_ledger.promote_gap` path Pass 2 / Phase 3c use ‚Äî a deduped umbrella checkbox + a scheduled coding fix, the title carrying the Bucket-A/B framing and a `dream-automation-gap` lineage tag; retirement reuses `reconcile_merged_gaps` (the ask memory retires off the gap-fix Ticket's MERGED state). The phase runs under `t3 dream run --full`, **default OFF** behind the `[loops.dream] automation_asks` / `T3_DREAM_AUTOMATION_ASKS` toggle; an unflagged pass never promotes an ask. `DreamQaProbe` is the retention/interference/monotonicity QA corpus, now POPULATED and replayed by the **¬ß 4 acceptance gates** ([#2545](https://github.com/souliane/teatree/issues/2545), `teatree.loops.dream.gates`). The cron snapshots each memory dir BEFORE and AFTER the file-side phases and runs all seven gates: **(a) retention** (every probe answerable pre-pass is still answerable post-pass), **(b) interference** (the prior-session probe pass-rate does not regress), **(c) consolidation-actually-happened** (net size reduced OR schema count grew OR clusters were recorded, AND every pruned index line has a confirmed durable home), **(d) index-budget** (`MEMORY.md` back under the real ~24 KB SESSION-LOAD byte-truncation point, not a 10x regression alarm ‚Äî so an over-budget index trips gate (d) RED while still recoverable; [#2723](https://github.com/souliane/teatree/issues/2723)), **(e) monotonicity** (two passes over a stable corpus do not lower the pass-rate), **(f) no-loss audit** (every archived entry records a source + a destination that exists), and **(g) compliance-non-regression** ([#2663](https://github.com/souliane/teatree/issues/2663) ‚Äî a recurrence remediated with another memory instead of a gate/eval FAILS the pass; an escalated recurrence passes). A probe is seeded per memory (keyed on a signature line) and replayed via an injectable, deterministic LLM-free answerer; the probe corpus is persisted (idempotent on `probe_key`, accumulating pass/run counts) so the next pass has a baseline. A **failing gate marks the pass attempted-not-succeeded** (staleness keeps firing) ‚Äî a lossy / delete-only / no-op consolidation is caught, never stamped success. Each gate has an anti-vacuous test proven in both directions, and the command's gate-gating is mutation-proven (ignoring the verdict turns the wiring tests RED). `DreamRunMarker` is the staleness marker keyed on the success timestamp (a run that keeps failing stays stale). `t3 dream run` (manual) and `t3 dream tick` (cadence-gated cron, ~04:00) take a `dream-tick` `LoopLease` and run the distillation engine (`teatree.loops.dream.engine.run_consolidation`), stamping `DreamRunMarker` only on a clean pass; `--dry-run` writes nothing. Engine v1 covers phases 1‚Äì3 (rank a bounded extract ‚Üí an injected `Distiller` clusters by root cause ‚Üí upsert by `cluster_key`, rejecting any uncited/hallucinated cluster and never overwriting a BINDING rule); the injected-distiller seam keeps the pass testable, and an SDK failure propagates rather than laundering into a fake success. **Phases 4‚Äì6 are LIVE** ([#1933](https://github.com/souliane/teatree/issues/1933) ¬ß 6): the cron runs the memory-file phases (cross-link, merge, re-index, decay) over the discovered `~/.claude` memory dirs, each fault-isolated in its own try/except so one phase (or one dir) failing never crashes the tick or stops the others, and each LIVE by default behind its own two-layer kill-switch (`[loops.dream] <phase>` toml / `T3_DREAM_<PHASE>` env, first match wins). The file-side phases run independently of the transcript extract, so they fire even on a 0-transcript-member pass (which stays attempted-not-succeeded for consolidation, but still re-indexes/decays the on-disk memory set; [#2547](https://github.com/souliane/teatree/issues/2547)). **Phase 4** (`teatree.loops.dream.cross_link`) appends symmetric `[[name]]` wiki-links between related memories ‚Äî related when their significant-token Jaccard clears a deterministic floor (no model) ‚Äî and is idempotent (a re-run on an unchanged set adds nothing). **Phase 4b** (`teatree.loops.dream.merge`, [#2723](https://github.com/souliane/teatree/issues/2723)) is the on-disk MERGE verb cross-link was missing: it collapses a near-duplicate PAIR (body-token Jaccard ‚â• 0.85 AND same frontmatter `type`/`name` family ‚Äî far above cross-link's floor) into the HIGHER-WEIGHT survivor (a BINDING rule outranks a plain restatement, so binding doctrine is never the side absorbed), appends the absorbed file's distinct lines with a provenance header, and ARCHIVES the absorbed file via `decay._archive_one` (moved, never deleted) so the re-index drops its pointer in the same pass. **Decision-3 (binding):** two BINDING near-duplicates are NEVER auto-merged ‚Äî they are cross-linked and a deduped `dream-binding-reconcile` ticket is filed via the existing `promote_memory` filer (reusing its banned-term / bare-reference withholding) for human reconciliation. Phase 4b runs AFTER cross-link, BEFORE re-index, LIVE by default behind the `[loops.dream] merge` / `T3_DREAM_MERGE` kill-switch, and is fault-isolated. The whole file-side phase orchestration (4/4b/5/6 + the ¬ß 4 gates) is a composed `teatree.loops.dream.phase_runner.MemoryPhaseRunner`. **Phase 5** (`teatree.loops.dream.reindex`) regenerates `MEMORY.md` as one tight line per memory ‚Äî a single bare `name.md` pointer (no `[name.md](name.md)` href duplication) plus a ‚â§45-char brief hook, the WHOLE line capped at 130 chars so it stays under the gate-(d) load budget ‚Äî deduped, filename-stable, content stays in the topic file, byte-identical on a no-change re-run. **Phase 6** (`teatree.loops.dream.decay`) ages a memory out by ARCHIVING it (moved to `<memory_dir>/archive/` with a provenance header, never deleted) ‚Äî gated by a NON-VACUOUS retention guard that retains a memory when ANY of: it is fresh (mtime within the window), it is still referenced (another live memory or the index `[[name]]`-links it), OR ‚Äî the **transfer-before-prune rail** ([#2546](https://github.com/souliane/teatree/issues/2546), #1933 ¬ß 2) ‚Äî its lesson has NO confirmed durable home in the `ConsolidatedMemory` ledger. Only a memory that is old AND unreferenced AND durably homed is archived; a guard-disabled probe proves each guard has teeth. The ¬ß2 rail ‚Äî *"delete an index line only after the fact has a confirmed durable home in a topic file"* ‚Äî is applied to the topic file itself: an injected `HomeResolver` seam keeps the file-side mechanics DB-free under test, while the production `ledger_durable_home_resolver` reads `ConsolidatedMemory.objects.prunable()` (the index-pruner's terminal-status + recorded-`durable_destination` surface) once per pass and maps a topic file to a row by path membership in `source_files` OR by its name appearing in a `durable_destination`. **Budget RETIRE tier ([#2723](https://github.com/souliane/teatree/issues/2723)).** The `prunable()` home-rail is structurally EMPTY for the hand-authored corpus (0 ledger rows reference on-disk memories), so it can never archive the files that bloat the index. A second, ledger-INDEPENDENT tier (opt-in via `DecayPolicy(budget_tier=BudgetTier())`, enabled in the cron) fires ONLY when `MEMORY.md` is over the gate-(d) load budget and archives the LOWEST-`_signal_score` files first ‚Äî scoring each by user/BINDING weight, inbound `[[name]]` links (+40 each), a LOGICAL `lesson_updated` recency clock (NOT raw `st_mtime`, so a cross-link / re-index rewrite does not reset it), and a per-type floor ‚Äî only as many as it takes to bring the projected hot index back under budget. Referenced entries are NOT hard-retained ([#2753](https://github.com/souliane/teatree/issues/2753)): the cross-link phase runs BEFORE decay and references most of the corpus, so a hard skip floored the tier above budget and it could never converge; the +40-per-link signal instead ranks referenced entries HIGHER so they are archived LAST, only when the budget genuinely forces it. Every archived entry stays restorable in `archive/` (full body + provenance) and recall-able via the cold `MEMORY_ARCHIVE.md` signature, so a now-dangling `[[link]]` in a surviving body is cosmetic, not data loss (the hot index uses bare `- name.md` pointers, which never dangle); the conservative stale/ledger tier keeps its reference skip. The cron re-indexes AFTER this decay so `MEMORY.md` falls back under the gate-(d) budget in the SAME pass. All phases accept an explicit `memory_dir` and never touch the real `~/.claude` under test. **Cold-tier RECALL ([#2746](https://github.com/souliane/teatree/issues/2746)).** The budget tier moves the lowest-signal memories ‚Äî ~130 BINDING/Non-Negotiable rules of a ~693-entry corpus ‚Äî to the NOT-session-loaded cold `MEMORY_ARCHIVE.md`, so they stop influencing behaviour. A `UserPromptSubmit` hook (`hooks/scripts/memory_recall.py`) closes that gap: the DB-free, stdlib-only pure core `teatree.loops.dream.recall` scores each cold-index line against the prompt's tokens (ambient `<system-reminder>`/`<command-*>` blocks stripped FIRST so an injected `MEMORY.md`/`CLAUDE.md` dump can never self-match), applies a 2-match relevance floor BEFORE any BINDING(+3)/`user_`(+2) boost (an irrelevant BINDING rule is never surfaced on one incidental token), dedupes against the hot index (pointer name + a renamed-file signature-substring guard), and injects the top ‚â§5 hits (‚â§1500 bytes, each line ‚â§200 chars) for that one turn; it is registered LAST on `UserPromptSubmit` so it never delays skill-loading, imports the core without `django.setup()`, and is crash-proof/silent on any degrade. Off-switch `[teatree] memory_recall_enabled` (default ON; self-rescue `t3 <overlay> gate memory-recall disable`), and `t3 <overlay> memory recall <query>` is the manual-lookup CLI. The recall quality and the ¬ß4 retention probe share ONE frontmatter-aware extractor `reindex.signature_text` ([#2746](https://github.com/souliane/teatree/issues/2746) nit-4) ‚Äî preferring a memory's frontmatter `description:`/`summary:` over the weak `node_type: memory` body line ‚Äî routed through by `reindex._summary_for` (hot index) and `gates._signature_line` (cold index + probe), so the hot index, cold index, and retention probe derive the SAME signature. The phase is OFF unless the `[loops.dream] derive_evals` / `T3_DREAM_DERIVE_EVALS` toggle is set, so an unflagged pass is byte-identical. The `dream` mini-loop is registered `off_live_tick` (driven only by its own cron); `t3 doctor` surfaces the 48h staleness alarm.

**Repair-loop budget + stall detection ([#2009](https://github.com/souliane/teatree/issues/2009)).** The verify‚Üídiagnose‚Üípatch‚Üíre-verify retry loop has a *visible* per-ticket-phase iteration budget, not just the time-based 24h stale-task expiry. Each `TaskAttempt` is auto-stamped on insert (the single `TaskAttempt.save` chokepoint) with `iteration` (its 1-based attempt number for the `(ticket, normalized-phase)`, spanning re-queued `Task` rows) and `error_fingerprint` (a stable hash of its terminal reason, normalized so transient noise ‚Äî timestamps, pids, hex ids, paths, bare numbers ‚Äî does not defeat the identical-failure check). `t3 <overlay> ticket show` surfaces `attempt N/max` per phase against the configurable `MAX_PHASE_ITERATIONS` cap (default 5). The re-queue chokepoint `TaskQuerySet.reclaim_orphaned_claims` consults `Task.check_requeue_allowed` (pure policy in `teatree.core.repair_loop`, model orchestration in `teatree.core.models.task_repair`) before returning a failed/orphaned task to PENDING: at the cap it raises `MaxIterationsExceeded`; two consecutive identical fingerprints on the same phase raise `IterationStalled`, which escalates to the user via a durable `DeferredQuestion` (¬ß17.1 invariant 9) and refuses the re-queue ‚Äî so a doomed phase neither re-runs forever nor burns another attempt on the identical failure.

**¬ß4 invariant ‚Äî worker enqueue pattern (load-bearing).** Transitions that own long I/O follow one rule:

- Transition body stays pure: state change + metadata only, then `transaction.on_commit(lambda: execute_X.enqueue(self.pk))`. The state change and the queued work land atomically.
- Workers take a row lock (`select_for_update()`), re-check the source state, run the runner, and on success call the next transition.
- At-least-once delivery is safe because the state guard makes redelivery a no-op.
- `post_transition` signals are reserved for lossy cross-cutting side effects (audit log, Slack reactions) ‚Äî never for the main work of the transition.

Auto-scheduling chains the phases: `start ‚Üí provision worker ‚Üí schedule_planning`, `plan ‚Üí schedule_coding` (gated on `PlanArtifact` DB record ‚Äî no artifact raises `NoPlanArtifactError`; the only path from STARTED to CODED passes through PLANNED), `code ‚Üí schedule_testing`, `test ‚Üí schedule_review`, `review ‚Üí schedule_shipping` (gated on `has_shippable_diff()`). Each forward transition threads the completing task as the next phase task's `parent_task` (PR-12), so the phase-boundary brief is warm ‚Äî `agents.prompt._parent_result_summary` reads the parent's last attempt (summary, files, next steps) instead of dispatching cold. The runner classes live in `core/runners/`; the workers in `core/tasks.py` and `core/worktree_tasks.py`. The PLANNED `plan_text` also carries an `E2E test plan / Acceptance scenarios` section for UI-visible tickets (prose-gated on `frontend_repos`, mirroring `is_ui_visible`) so behaviour-level TDD is planned up front (#2240).

**Intake landscape survey baked into the FSM ([#2541](https://github.com/souliane/teatree/issues/2541)).** The provision worker (`execute_provision`) is the ticket-intake (info-fetch) step: once the worktrees materialise and before `schedule_planning`, it surveys what is already in flight or already settled ‚Äî open PRs/MRs, in-flight local worktrees (uncommitted/unpushed), and a per-issue close/merge/supersede recommendation ‚Äî and persists it as an append-only `LandscapeArtifact` (the survey shape gathered django-free by `teatree.core.intake.landscape` and composed by `teatree.core.intake.landscape_gather`; mirrors `PlanArtifact`, latest governs). The planner CONSUMES that persisted survey rather than re-deriving it (a headless planner sees it inline in its `INTAKE LANDSCAPE SURVEY` system-context block; any planner reads `LandscapeArtifact.latest_for`). Persistence is best-effort and never a gate ‚Äî a forge outage during provision degrades to a log line and never blocks provisioning or planning; the planner then falls back to the `workspace landscape` fetch. `info artifacts` surfaces the latest survey alongside a ticket's other artifacts.

**Pre-flight attachment-fetch gate (PR-15).** The same intake step (`execute_provision`, after the landscape survey and before `schedule_planning`) refuses to hand off to the planner while any attachment the ticket references is still un-fetched ‚Äî a GitLab upload, a linked Notion file, a Slack-thread file. `teatree.core.intake.attachment_manifest` extracts every referenced attachment URL from the issue body + comments (deterministic regex, code-host text via an injected seam, fail-open), reconciles them against the on-disk cache under `<ticket_dir>/.attachments/` (ground truth: the file exists), and persists an append-only `AttachmentManifest` (mirrors `LandscapeArtifact`, latest governs). The gate is vacuous on a zero-attachment ticket; a hold is transient (re-running provision after a fetch hands off) and never a lockout ‚Äî the `attachment_gate_enabled` kill-switch, a `--fetch`, or manual placement all release it. CLI: `t3 <overlay> ticket attachments <ref> [--fetch]`; fetch transport is injected via a core registry backends populate at app-ready (Notion wired, other kinds report a manual-download hint). Companion intake-source check: `teatree.verification.url_check.check_url` (HEAD-first, ranged-GET fallback, typed `OK`/`UNRESOLVED`/`NETWORK_ERROR`) gates `PendingArticleSuggestion.record_candidate`, dropping a fabricated/404 candidate URL and recording-anyway on a transient network error.

**Who drains the queue.** No always-on `db_worker` is assumed; a won-owner tick drains a bounded batch of `DBTaskResult` jobs in-process (`teatree.loop.queue_drain`), standing down while a live worker holds the `worker` (`WORKER_SINGLETON`) flock ‚Äî the one singleton every worker entry point acquires after PR-28 retired the legacy `teatree-worker` name ‚Äî and first expires READY jobs older than `T3_QUEUE_STALE_HOURS` (default 24h) to the reversible `FAILED` state so a fresh drainer never fires stale heavy jobs.

**Clean-worktree preflight (#884).** `code()`, `test()`, `review()`, `ship()` refuse the transition if any worktree has tracked uncommitted changes. The refusal raises a `DirtyWorktreeError` (an `InvalidTransitionError` subclass, **not** `TransitionNotAllowed`) inside the caller's outer atomic block, so the FSM state change is rolled back and the lease reaper returns the CLAIMED task to PENDING on the next tick. No auto-stash ‚Äî worktrees share one `.git`, so a stash is repo-global and could clobber an unrelated branch's work.

**Per-(repo, ticket) open-PR budget gate (north-star PR-2).** `core/gates/pr_budget_gate.check_pr_budget` runs at the two core PR-creation seams ‚Äî `_run_ship_gates` (`pr create`, right after the shipping gate) and the orphan-branch path (`_ensure_pr.create_or_defer_pr`) ‚Äî refusing to open a PR when the ticket already has `max_open_prs_per_repo_per_ticket` open (not-merged) PRs in that repo. **Net-new-tech-debt merge gate (north-star PR-3).** `core/gates/debt_delta_gate.check_debt_delta` runs at the same two core PR-creation seams as the budget gate ‚Äî `_run_ship_gates` (`pr create`, right after the budget gate) and the orphan-branch path (`_ensure_pr.create_or_defer_pr`), both through one `evaluate_debt_delta` orchestration ‚Äî refusing a ship whose diff introduces **net-new** tech-debt suppression: a new `noqa` / `type-ignore` / `pragma-no-cover` comment, an unreferenced `pytest.mark.skip`/`xfail`, a new `per-file-ignores` entry, or a lowered `fail_under` coverage floor. **Sync writers honor the same gate ([#1426](https://github.com/souliane/teatree/issues/1426)).** The DoD decision is factored into `dod_gate.sync_gate_allows` so every sync writer ‚Äî not just `ship()` ‚Äî respects it, closing the bypass where sync wrote a post-ship state directly. A **post-ship** state is at or past `SHIPPED`. A *workflow* state inferred from a live open PR is *capped* ‚Äî a gate-refused state demotes to `STARTED`; a *terminal* state reflecting an external fact (merged PR ‚Üí `MERGED`, board "Done" ‚Üí `DELIVERED`) is **never demoted** but records a durable `dod_e2e_violation` audit marker and logs loudly, so the gap is auditable not silent.

**Concurrent-stack cap (#1397).** `max_concurrent_local_stacks` caps the distinct tickets whose worktrees can be in `services_up`/`ready` at once per overlay. The gate (`local_stack_gate.check_local_stack_limit`) runs before `Worktree.start_services()` and raises `LocalStackLimitExceededError` naming the blockers. Default `0` is unbounded; sibling worktrees of one ticket count as one stack; each blocker is reconciled against docker first (a zero-container row is a phantom and excluded; a mid-restart row stays counted, fail-safe).

**Provisioning time-box + loud alert ([#2220](https://github.com/souliane/teatree/issues/2220)).** A long-blocking provisioning subprocess (DSLR restore, `migrate`, `--create-db`) must fail loud, never grind silently. `provision_timebox.run_timeboxed_step` bounds each step by a *fast* (`provision_fast_step_timeout_seconds`, default 120) or *heavy* (`provision_step_timeout_seconds`, default 1800) ceiling ‚Äî `ProvisionStep.heavy` opts a step (a DB import, a frontend build) into the long one; every other step (symlinks, settings, a compose override) defaults fast, so a hang on an ordinary step surfaces in seconds instead of burning half an hour. On the ceiling it aborts and DMs the user, while a heartbeat distinguishes slow-but-moving from a hang. `detect_migration_conflict` matches a running `migrate`'s conflict output and surfaces the `makemigrations --merge` remedy. Best-effort.

**Parallel provisioning + resource-aware admission ([#2949](https://github.com/souliane/teatree/issues/2949)).** `workspace provision` runs a ticket's worktrees under a bounded subprocess pool (`_workspace_provision_parallel.run_worktree_provisions_in_parallel`) instead of one serial `for` loop ‚Äî each worktree's provision runs as its OWN `python -m teatree worktree provision` subprocess, never in-process threads, because `WorktreeProvisionRunner`'s DB-import path mutates process-wide `os.environ`. The pool size defaults to an nCPU-derived cap (`teatree.utils.ram_probe.default_provision_concurrency`, halved core count) unless `provision_max_concurrency` pins one. Before each new subprocess is admitted, `provision_admission_gate.check_provision_admission` HOLDS it when host RAM is at/above `provision_ram_ceiling_percent` (default 85); a held request re-checks on the next poll and drains automatically once RAM frees, bounded by `max_hold_seconds` (default 600s) past which admission is overridden (logged loud) rather than blocking the CLI invocation forever. Within one worktree, `ProvisionStep` is a typed DAG node (PR-27): `requires`/`produces` tokens order the steps and steps with no dependency path between them run concurrently (the `subprocess_only` ones on a bounded thread pool, ORM steps serially in-process) via `run_provision_steps`; a `requires` token no step `produces` fails the run loud (fail-closed). `ProvisionStep.post_condition` is a truth check evaluated after the callable ‚Äî a step that "ran" but did not produce its resource is recorded FAILED ‚Äî and its aggregate (`teatree.core.provision.provision_postconditions.aggregate_provision_post_conditions`) is what `worktree status` evaluates to prove a worktree is really provisioned. `ProvisionStep.skip_probe` lets a step short-circuit to a near-zero `StepResult(skipped=True)` when its precondition is already satisfied. Every provision persists a `ProvisionReport` (steps + total duration + success) to `Worktree.extra['provision_report']` ‚Äî no schema change ‚Äî rendered by `workspace provision --report` and `worktree status`; a total exceeding `provision_slow_threshold_seconds` (default 600) fires the same best-effort user alert as a single step timeout.

**Wrong-repo worktree guard ([#2276](https://github.com/souliane/teatree/issues/2276)).** `find_clone_path` resolves a ticket's repo by basename, so a sibling clone of the same name (different `origin`) could be cut silently into the worktree. `WorktreeProvisioner._create` calls `git_guard.guard_repo_remote_slug` before `git worktree add` whenever `repo_name` is an `owner/repo` slug: a mismatch between the clone's `origin` slug and the expected slug raises (a missing `origin` fails closed). A bare basename carries no canonical slug, so the legitimate `--repos <basename>` flow is untouched.

**Per-repo branch provisioning ([#33](https://github.com/souliane/teatree/issues/33)).** A ticket whose repos live on DIFFERENT branches (an unmerged multi-repo change ‚Äî one repo on a feature branch, a sibling on another) maps each repo to its own branch in `Ticket.extra['branches']` (repo ‚Üí branch); the `WorktreeProvisioner` (#2589) checks each repo out on its mapped branch while the ticket DIR stays the single `extra['branch']`, so the repos provision as SIBLINGS in one dir. The CLI surface is the `--repos` token: a `repo:branch` form (e.g. `--repos 'backend:fix/x,frontend:main'`) supplies the override ‚Äî `resolve_repo_names` strips the suffix to the bare name and `parse_repo_branch_map` extracts the map from the same string; a bare `repo` token (no `:branch`) falls back to the shared ticket branch. The map merges across `workspace ticket` reruns so adding a sibling repo's branch never drops the others.

**Idle-stack reaper + acquisition queue ([#2190](https://github.com/souliane/teatree/issues/2190), [#44](https://github.com/souliane/teatree/issues/44)).** Four composed parts replace the hard `SystemExit` at the cap; all are global, mechanical-only, and individually kill-switched.

- **Reaper** (`IdleStackReaperScanner` ‚Üí `reap_idle_stack`): demotes an IDLE running stack to `provisioned` via `stop_services` (reversible ‚Äî DB + worktree preserved). Idle (`core.gates.idle_stack.preserve_reason`) = no live `Session`/`Task`, `last_used_at` past `idle_stack_idle_minutes`, not the CWD, no live external-delivery lease, no recent E2E run, not `reaper_pinned`; fail-safe ‚áí KEEP. `classify_running_worktrees` logs preserved-vs-reaped so a reap is never silent (#2227).
- **Queue**: at the cap `acquire_or_enqueue` reaps ‚Üí retries ‚Üí enqueues a `LocalStackQueueItem`; `LocalStackQueueDrainerScanner` re-checks the cap each tick with Fibonacci-minute backoff and re-fires `start` once a slot frees ‚Äî never tearing down another ticket's stack.
- **Stale-stack reaper** ([#2207](https://github.com/souliane/teatree/issues/2207)): unowned compose projects (no `Worktree` row) older than `stale_stack_min_age_minutes` (default off) are downed via `workspace reap-stale`; unknown age ‚áí KEEP.
- **Safe disk reclaim** ([#2246](https://github.com/souliane/teatree/issues/2246)): `workspace reclaim-disk` runs only three zero-data-loss prunes (build cache + dangling-only images + unreferenced-only volumes) via a fixed argv, so `-a`/`system prune` can never enter the set ‚Äî the sanctioned path so the agent stops hand-rolling raw `docker image prune -af`.

## 5. Agent Execution

The agent layer is `teatree.agents` (headless executor + prompt + skill bundle + structured result schema) and `teatree.loop` (the /loop topology). The runner and judge run **virgin** (`isolated_claude_env` + clean-room `ClaudeAgentOptions`: `setting_sources=[]`, empty `settings`, a neutral `HOME`/cwd) so the developer's `CLAUDE.md`, auto-memory, and project config never bias a result. **Prompt caching** stays on by the bundled CLI's content-based default (5-min TTL; the API-key switch does not disable it ‚Äî confirmed via the `cache_creation_input_tokens`/`cache_read_input_tokens` usage fields the harness already surfaces). The stable `SKILL_BUNDLE_FRAMING` leads the system prompt and the polluted preamble lives in the USER prompt, so the cacheable prefix is already maximal; `ClaudeAgentOptions` exposes no 1h-TTL knob, so teatree injects none (no raw `cache_control` headers). **Sub-agent-spawning isolation ([#2642](https://github.com/souliane/teatree/issues/2642)).** A spawning scenario (`scenario_exposes_subagent_spawn` ‚Äî `orchestrator_delegates_*`, `full_speed_*`, `team_mate_*`, `delegates_under_load_*`) makes the SDK sub-agent do REAL, destructive git work; a neutral cwd does NOT protect the developer's clone, because the sub-agent locates it via the editable install's `.pth` (`import teatree` ‚Üí real `src/teatree`) and the shared `.git`. So `ApiInProcessRunner._resolve_eval_target` runs each spawning scenario against a per-run EPHEMERAL CHECKOUT (`teatree.eval.ephemeral_checkout`): a throwaway `git worktree --detach` whose own working tree + detached `HEAD` absorb every write, with `cwd` + `add_dirs` pointed at it and `ephemeral_checkout_env` overlaying `PYTHONPATH` (so `import teatree` resolves into the throwaway) and clearing the inherited `GIT_*` pins (so `git` rediscovers from the ephemeral cwd). The checkout is torn down after the run; a non-spawning scenario keeps the existing clean-room shape unchanged. When the real root cannot be located the spawning scenario REFUSES (`EphemeralCheckoutError`) rather than fall back to the real clone. **Transient missing CLI ([#2642](https://github.com/souliane/teatree/issues/2642)).** `shutil.which("claude")` momentarily returns `None` when the bundled CLI auto-updates mid-run (the nvm symlink is swapped); `resolve_claude_path` re-probes with a bounded backoff so one transient miss never reds the rest of the batch, while a genuinely-absent binary still fails after the bounded attempts. **All-skipped guard ([#1811](https://github.com/souliane/teatree/issues/1811)).** A run that never happened (`claude` absent/unauthenticated) would SKIP as passed; `--require-executed` exits non-zero when `collected > 0` but `executed == 0`. The standalone metered workflow (`.github/workflows/eval.yml`, weekly cron + manual) passes `--require-executed` unconditionally (never key-gated) and is no-PR-guarded so a window with nothing merged skips cleanly.

**Structured result schema (`agents/result_schema.py`).** Agents return JSON: `summary`, `files_modified`, `tests_run`, `tests_passed`, `tests_failed`, `decisions`, `needs_user_input`, `user_input_reason`, `next_steps`, `commands_executed`. `additionalProperties: false`. Validated without the `jsonschema` library to keep the dep tree small.

**Skill bundle (`agents/skill_bundle.py`) + delegation map (`skill_support/map.py`).** A phase ‚Üí companion-skills map (e.g. `coding ‚Üí test-driven-development`), topo-sorted `requires:` resolution, per-overlay `companion_skills`. Reviewer-dispatch review skills come from `OverlayConfig.get_review_companion_skills()` (default `code-review`); on the `reviewing` phase `build_system_context` embeds them in full, and an orchestrator-built reviewer dispatch goes through the single shared `build_reviewer_dispatch_prompt()`, which prepends a REQUIRED Skill-tool load block (`t3:review` + overlay review skills) so conventions reach the reviewer structurally, not via orchestrator memory. The **coding phase** force-loads a symmetric contract: a REQUIRED load block (`/t3:architecture-design`+`/t3:code`), `architecture-design` embedded in full, and behavior-preservation + no-AI-signature clauses, so the discipline reaches a skill-stripped builder. It also injects an explicit stack+overlay load block computed from the resolved bundle (framework skill `/ac-django`/`/ac-python`/`fastapi` plus the active overlay skill), so a code-touching dispatch never relies on auto-detect; an unresolved stack falls back to a "load your stack's coding skill yourself" instruction ([#1368](https://github.com/souliane/teatree/issues/1368)). The SKILL.md reading + rendering primitives (`_read_skill_contents{,_scoped}`, `_find_skill_md`) live in the Django-free `agents/skill_injection.py`, shared by `build_system_context` and a **raw-Agent-tool spawn** preamble: an orchestrator that spawns an e2e/coder/reviewer sub-agent with a bare prompt string bypasses every skill-injection path, so `skill_injection.build_subagent_skill_preamble` concatenates the requested skills' `SKILL.md` bodies (resolved across the framework AND active-overlay skills dirs) and `t3 <overlay> skill-preamble --skills t3:rules,t3:e2e[,<overlay-skill>]` emits it for the orchestrator to prepend to every brief (fails loud, non-zero, on an unresolved skill). `skills/rules/SKILL.md` ¬ß Sub-Agent Limitations makes the carry mandatory, pinned by `evals/scenarios/orchestrator_embeds_skills_in_subagent_brief.yaml`.

**The single model constant ‚Äî abstract tiers.** Every concrete Claude model id in teatree lives in exactly ONE place: `agents/model_tiering.TIER_MODELS` (`{"frontier": "claude-opus-4-8", "balanced": "claude-sonnet-5", "cheap": "claude-haiku-4-5"}`). Everything else ‚Äî production phase dispatch, eval scenarios, the benchmark, and the tests ‚Äî references an abstract TIER (`frontier`/`balanced`/`cheap`), never a concrete id. `resolve_tier(tier)` reads `TIER_MODELS`, overridable per tier via `[agent.tier_models]` in `~/.teatree.toml` (merged OVER the default, mirroring `skill_models`). Adopting a new model is one edit to `TIER_MODELS` or one config line ‚Äî zero scenario, test, or dispatch edits. This is a pure in-memory constant lookup (no I/O, no DB) so the agent-spawn hot path stays trivially safe.

**Model tiering.** `agents/model_tiering.resolve_phase_model(phase)` resolves phase ‚Üí tier ‚Üí concrete model id. `DEFAULT_PHASE_MODELS` maps each FSM phase to a TIER: the genuine-reasoning phases (`planning`/`coding`/`debugging`/`reviewing`/`retrospecting`) ‚Üí `frontier`; the mechanical-but-non-trivial phases (`testing`/`shipping`) ‚Üí `balanced`; the pure-handoff phase (`requesting_review`) ‚Üí `cheap`. A phase NOT in the dict falls back to `DEFAULT_TIER` (`balanced`). Per-phase overrides via `[agent] phase_models` (a TIER name, a concrete id, or a sentinel that inherits the user's default ‚Üí `None`).

**Model capability order.** `core/cost.tier_rank(model)` ranks a model id / tier name along a *capability* axis `cheap < balanced < frontier` (the abstract tiers). An old short-name (`opus`) or a concrete dated id (`claude-opus-4-8`) ranks identically to the abstract tier it belongs to (`frontier`). An unknown full id ranks *above* every known tier (most-capable, so a lower floor never silently downgrades an unknown spawn target), the inverse of the conservative pricing fallback. This order is the comparator for the most-capable-wins floor merge below.

**Cache-multiplier SSOT ([#2401](https://github.com/souliane/teatree/issues/2401)).** The Anthropic cache-pricing multipliers ‚Äî a cache *read* bills 0.1x and a 5-minute cache *write* 1.25x the model's base input rate ‚Äî are a fixed property of the API, not of any model, so they live in **one** foundation-layer leaf `src/teatree/pricing.py` (`CACHE_READ_MULTIPLIER` / `CACHE_WRITE_MULTIPLIER`). Every cost path imports them from there ‚Äî `core/cost` (per-attempt SDK cost), `eval/cost_fit` (the warm-equivalent benchmark fit), and `eval/models` (the billed-input regressor) ‚Äî rather than each keeping a private copy that could silently drift (the single source of truth). `teatree.pricing` is a `tach` `foundation` module with no dependencies; `tests/test_pricing_ssot.py` greps `src/teatree` and fails RED if the multipliers are ever defined in more than one place.

**Per-skill model floor + spawn-model merge.** `model_tiering.resolve_spawn_model(phase, *, skills)` is what both spawn paths call (headless SDK `model=` and interactive Agent-tool dispatch). It starts from the per-phase tier (resolved to a concrete model id) and raises capability for each loaded companion skill carrying a `[agent.skill_models]` MODEL floor (most-capable-wins via `tier_rank`, order-independent; a winning tier-name floor is resolved through `resolve_tier`). The floor is MODEL only ‚Äî no per-skill effort axis ‚Äî and returns `None` (inherit) only for a sentinel `phase_models` override with no floor; every other phase resolves to a concrete model id.

**The single effort constant + per-tier spawn effort.** The reasoning-effort axis has its own single source of truth `agents/model_tiering.TIER_EFFORT` (`{"frontier": "xhigh", "balanced": "high"}`), the effort parallel of `TIER_MODELS` ‚Äî only the reasoning tiers carry an effort; `cheap` (Haiku, which rejects the lever) is absent. `resolve_spawn_effort(phase)` resolves phase ‚Üí tier ‚Üí effort exactly as `resolve_phase_model` resolves phase ‚Üí tier ‚Üí model (same `[agent] phase_models` override mechanism, so a phase downgraded to a cheaper tier drops model AND effort in lock-step), overridable per tier via `[agent.tier_effort]` (validated against the effort scale). The headless SDK builder pins it as `ClaudeAgentOptions.effort=` for the SAME phase the model resolved from ‚Äî `xhigh` for a frontier phase, `high` for a balanced one, unset for the cheap/Haiku phases and a sentinel-opted-out phase. Skill floors and the honesty escalation raise only the MODEL, never the phase's effort tier.

**Session-level pins (interactive main agent).** `[agent] session_effort` (`low|medium|high|xhigh|max`) and `[agent] session_model` are the interactive main-agent pins: injected as `--effort`/`--model` into the interactive `claude` spawn argv, so the user never runs `/model`. This is a SEPARATE axis from the per-tier `TIER_EFFORT`/`resolve_spawn_effort` spawn effort above ‚Äî the interactive `session_effort` pin never leaks into a headless SDK sub-agent (whose effort comes from its phase tier), and the in-session Agent tool still has no effort param. `t3 doctor` validates the values (a bad effort FAILs, an unknown model WARNs) and `_check_stale_path_t3` ([#2281](https://github.com/souliane/teatree/issues/2281)) FAILs when a `t3` binary precedes `$UV_TOOL_BIN_DIR` on `PATH`, masking `uv tool install` updates.

**Selective per-phase fan-out ([#2229](https://github.com/souliane/teatree/issues/2229)).** A declarative, default-OFF way for an opted-in phase to fan work out (adversarial-verify on review, judge-panel on planning) without the blanket session-wide `ultracode`. The `core.phases.FANOUT_BY_PHASE` registry maps `(role, phase) ‚Üí PhaseFanout`, parallel to `SUBAGENT_BY_PHASE`; the opt-in is `[agent.phase_fanout]` keyed `"role:phase"` (`true` = default width, int overrides). `resolve_fanout_directive` renders `""` with no entry/opt-in (the byte-identical spine) else a **plain-text** directive (¬ß2 SDK/API-portable, [#2236](https://github.com/souliane/teatree/issues/2236)) threaded as the `fanout_directive` dispatch field. Ships **route (b)** only (rigor inside the one in-session sub-agent); real top-level parallel fan-out is deferred.

**Situational honesty-critical escalation ([#2263](https://github.com/souliane/teatree/issues/2263)).** Routes a *verification* spawn to the most-honest configured model (`[agent] honesty_model`, default Opus) when the agent's honesty is in question ‚Äî situational, NOT a phase floor. Four triggers write a session/task-scoped `HonestyEscalation` row (mostly agent-fired via the `skills/rules` directive `t3 <overlay> honesty escalate`; `shipped_incomplete` also has a deterministic backstop in the rubric done-gate). `resolve_spawn_model` raises the winner to `honesty_model` when a verification phase has an active escalation ‚Äî the LAST step of resolution, so nothing downgrades it afterward. `is_active` auto-clears at read-time (`cleared_at IS NULL AND expires_at > now()`, 6h TTL backstop); the primary clear is a fully-passed `rubric-grade`.

### 5.6 Loop Topology

**Track A vs Track B ([#1838](https://github.com/souliane/teatree/issues/1838)) ‚Äî read this first.** Two orthogonal axes restructure the loop, and "panes" belongs to only one of them:

- **Track A ‚Äî restructure the loop on the lead session, NO panes.** Arm the dormant `orchestrate_phase` (the DB-home `orchestrate_claim_enabled` setting, `t3 <overlay> config_setting set orchestrate_claim_enabled true`, [#2315](https://github.com/souliane/teatree/issues/2315)) and per-loop ownership ([#2318](https://github.com/souliane/teatree/issues/2318)). The original `dedicated_loops` toggle ([#2319](https://github.com/souliane/teatree/issues/2319) ‚Äî split one fat `/loop` into N dedicated scoped-tick slots) is **retired**: [#2650](https://github.com/souliane/teatree/issues/2650) made one native Claude `/loop` per enabled `Loop` row the single, default loop shape (below), so the fat-`/loop`-vs-dedicated-slots toggle is gone ‚Äî the per-loop split is now unconditional, not a default-off setting. Track A's live remainder is the `orchestrate_phase` arming + per-loop ownership, on the lead session, with no separate sessions involved.
- **Track B ‚Äî pane-backed teammates.** A pane is a separate long-lived Claude session executing claimed `team:<role>` work. CORE-MAKER + OVERLAY-MAKER are built; REVIEWER panes are prohibited in code (a code-level raise pinned by a fitness test). Track B is the only axis that introduces panes.

`[teams] enabled = false` (default) is the master agent-teams off switch ‚áí the loop runs the classic in-session `Agent(run_in_background)` sub-agent fan-out, no panes. Flip it via `t3 teams on` / `t3 teams off` (`t3 teams status` to read). The two axes compose independently: Track A's `orchestrate_phase` arming reshapes the lead loop whether or not Track B's panes are enabled.

TeaTree drives the day from one long-lived Claude Code session that registers one native Claude `/loop` per enabled `Loop` row (#2650; every loop ships paused, so a fresh install registers none), each firing on its own DB-configured cadence (default 12 min). The tick body `teatree.loop.tick.run_tick` is code ‚Äî tested, typed, version-controlled ‚Äî composing named single-responsibility phases ([#1796](https://github.com/souliane/teatree/issues/1796)): `scan_phase` (parallel world-scan), `sweep_phase` (maintenance scanners split out of the scan), `act_phase` (dispatch + mechanical + persist), and `orchestrate_phase` (the wip-driven autonomous fan-out ‚Äî a no-op at default `medium`; `slow` targets one live worker, `full` targets the summed per-overlay `max_concurrent_auto_starts` budget, and `boost` targets `boost_concurrency = N` live workers via a **pool-refill driver** (clamped by the PR-01 resource concurrency ceiling `provision_max_concurrency`; `boost_concurrency = 0` keeps `full`'s summed-overlay target), admitting a queued TODO/followup before a new-ticket auto-start ‚Äî the admission-priority ordering lives in `teatree.loop.queue_drain`). `orchestrate_phase` is wired as a **read-only budget PLANNER** ‚Äî it computes the standing live-worker *target* (and the marginal `cap = target ‚àí in_flight` it would admit this pass) but never claims a `Task` (the single claim point stays the live `claim_next` CAS, so the tick can never orphan a claim). When the opt-in, default-OFF `orchestrate_claim_enabled` toggle is ON and the wip clamps, the tick persists the **target** as an admit-budget ceiling to the `tick-meta.json` sidecar (`teatree.loop.admit_budget`); the live claimer reads it and refuses once in-flight CLAIMED WIP reaches the target, so claimed ‚â° spawned ‚Äî and, the pool-refill invariant, when a worker exits below the target the next tick admits the shortfall back to N. At `medium` or with the toggle OFF no budget key is written (absence = unclamped = byte-identical throughput), and the reader fails open on an absent/stale/corrupt budget so a dead loop never throttles live dispatch ([#1838](https://github.com/souliane/teatree/issues/1838), [#1917](https://github.com/souliane/teatree/issues/1917)).

**Agent-teams Track-B ‚Äî the inert maker-only pane layer ([#1838](https://github.com/souliane/teatree/issues/1838)).** A maker pane is a long-lived `team:<role>` claim of the existing `Task` + lease (no new model). `teatree.teams` houses the lifecycle FSM (`spawn ‚Üí active ‚Üí idle ‚Üí stopped`, state derived from the live `Task` claim + lease + `Session`), the idle-pane reaper (demote a pane with no live `Session` past `[teams] idle_minutes`, fail-safe KEEP), and the t3-master collision guardrails (a pane claims ONLY `team:<role>`, never `t3-master` or a `loop:` slot ‚Äî proven-disjoint key spaces). Budget knobs `[teams] max_panes` (1) and `idle_minutes` (30) on `UserSettings` are fail-safe to their defaults at every tier, so the safety bound cannot be configured away; a pane heartbeats via `Task.renew_lease`, so the existing orphan-reclaim recovers a dead pane. **Default-OFF** (`teams_enabled` false): the maker path and reaper are no-ops, and the spawn primitive `TeammatePane.spawn` itself **fails closed** ‚Äî it reads the effective `teams_enabled` (env > overlay-row > global-row > default, via `get_effective_settings`) and raises `AgentTeamsDisabledError` before claiming any `team:<role>` slot, so the setting is the single source of truth at the spawn seam and a teammate can never be spawned while agent teams is off ‚Äî even if a caller reaches the primitive directly, bypassing `claim_maker_pane`. The only live-path importer of `teatree.teams` is the sanctioned, `None`-when-off pane-reaper scanner, pinned by the [#2320](https://github.com/souliane/teatree/issues/2320) AST inertness scan. On `SessionStart(source=compact)` the lead synchronously orphans the stale `t3-master` lease so its next tick re-anchors uncontested and no pane can win the compaction-window CAS race. REVIEWER panes are never spawnable (a code-level raise pinned by a fitness test); reviewing stays on the fresh-spawn-per-head_sha AutoReviewDispatch path. **Opus floor on a maker pane.** `build_pane_options` floors the resolved spawn model up to opus (`_floor_teammate_model` + `_TEAMMATE_MODEL_FLOOR`): a maker pane runs a long-lived, multi-task session and a sub-opus mate auto-compacts mid-task and loses its working context. An inherited (`None`/empty) or below-opus resolution is pinned to opus ‚Äî closing the inherit hole a plain most-capable-wins merge leaves open (`tier_rank(None) == tier_rank("opus")`) ‚Äî while an at-or-above-opus model passes through unchanged (the floor only raises, never lowers). The native Agent-Team roster carries the same rule in prose (`skills/wip` ¬ß "Fixed roster in Agent-Team mode") with the co-located eval `team_mate_spawned_opus_never_sonnet`, so the host roster and the pane layer agree: a team mate is opus or stronger. **Pane DISPLAY layer (WI-5, presentation-only).** `pane_display` optionally renders a pane's in-process SDK session in a visible terminal via plain `tmux split-window -- claude --resume <session_id> ‚Ä¶` (the resumed id is the SAME SDK session, the single SOT); native iTerm2 splitting is just a property of how the user attached (`tmux -CC`). It never replaces the SDK run ‚Äî no tmux/TTY/any failure falls back to the in-process path ‚Äî and the in-pane `claude` is plain interactive (never `claude -p`), so billing stays on the interactive lane. Teardown is DB-driven: demote-to-STOPPED triggers `reconcile_orphan_panes` to `kill-pane` every orphaned `team:*` pane. Mode `[teams] display = auto|tmux|none` (default `none`, fail-safe); `none` + headless/CI ‚áí byte-identical to today.

**#786 epic ‚Äî the immortal-singleton roster model is fully retired (WS1‚ÄìWS5 + #54, all merged).** The original model ‚Äî a coordinator spawning a fixed roster of long-lived loop sub-agents it had to keep alive and re-spawn on death/compaction ‚Äî was the root cause of the recurring "loop died on compaction / re-spawned" toil and the duplicate-on-restart hazard. It is **fully retired**: no roster, no `spawn_brief`, no takeover-respawn, no resume-by-agentId. The replacement satisfies three acceptance-contract invariants, each delivered by a specific workstream and detailed in the appendix:

- **Invariant 1 ‚Äî 0 sessions ‚áí nothing runs.** The loop is session-bound; zero open sessions ‚áí dormant, by design (WS3). Two in-session siblings ride `SessionStart` / `t3 doctor check` / the recovery path: `account_switch` ([#1916](https://github.com/souliane/teatree/issues/1916)) compares `oauthAccount.accountUuid` and, on a `/login`, invalidates the backend token cache so DMs do not route to the old workspace; `mcp_connectivity` ([#2282](https://github.com/souliane/teatree/issues/2282)) live-probes every enabled MCP server via `claude mcp list` and validates each against its overlay-declared provider (`get_mcp_provider_expectations()`), making a disconnected/mismatched MCP a loud finding. A third sibling, `connector_manifest` (PR-19), lets each overlay declare its required-vs-optional claude.ai connectors by NAME (`get_connector_manifest()`); `check_connector_manifest` FAILs `t3 doctor check` on a down **required** connector with mode-correct guidance (first-install vs post-account-switch, from `claudeAiMcpEverConnected`) + a `RECONNECT` line, `t3 mcp reconnect [--open]` prints those lines, and `run_connector_preflight` refuses to start the loop when a required declared connector is down. A separate in-process `scope_cache` (PR-19) records known-missing `(token_id, scope)` pairs so a backend call short-circuits pre-HTTP (`ScopeMissingError(cached=True)`, one banner per pair) instead of re-failing every call site ‚Äî consulted by the Slack transport, reset per tick, cleared by `t3 doctor authorizations`.
- **Invariant 2 ‚Äî at most one tick per enabled loop.** Under [#2650](https://github.com/souliane/teatree/issues/2650) / [PR-28] each enabled loop is driven by its OWN self-rescheduling `loop_timer` chain that the singleton `t3 worker` drains, firing `t3 loops tick --loop <name>` on that loop's cadence; there is no single machine-wide tick. The per-loop executor mutex is the WS2 `LoopLease` DB row (backend-agnostic conditional-UPDATE CAS, expiry-reapable) claimed under the disjoint `loop:<name>` slot, so two sessions never double-drive the same loop while different loops run in parallel. Atomic per-unit claim is WS1 `t3 loop claim-next` (claim == spawn boundary). A second concurrent tick of the SAME loop loses the CAS and SKIPs. **Durable per-loop control plane ([#1913](https://github.com/souliane/teatree/issues/1913)).** `LoopState` is the canonical (and only) loop enable/disable control tier (mirrors `ConfigSetting`): one row per loop name carries `enabled`/`paused`/`disabled`, an absent row resolves to `enabled` (empty table is a provable no-op), and a `paused`/`disabled` row stops even the core `dispatch` loop. `t3 loop {pause,resume,disable,enable} <name>` are atomic, idempotent transitions via the `loop_state` management command (`resume`/`enable` are the one "make it run again" verb clearing either hold), and `t3 loop loop-state <name>` is the strictly READ-ONLY status probe ‚Äî it reports the current state and writes nothing, with output phrased as a read (`status: <STATUS>`) rather than the mutation verbs' "is now", so inspecting a loop can never be mistaken for a pause/enable that just ran. `enable`/`disable`/`resume` move BOTH source-of-truth planes the #2584 verdict reads in one transaction ‚Äî the `LoopState` upsert AND the row-level `Loop.enabled` column the tick gates on ‚Äî so the verb can never report "now enabled" while the tick keeps skipping the loop (the bug where `enable` wrote only `LoopState` and left `Loop.enabled=False`). `pause` is the reversible control-plane hold only: it leaves `Loop.enabled` untouched so `resume` returns it to running without re-enabling a row that was deliberately disabled. The hold is consulted first by the tick gate (`LoopsConfig.is_enabled`), the review-claim chokepoint (`review_claim_signals.review_loop_enabled`), AND the in-session Stop self-pump on the `dispatch` loop ‚Äî the restart-surviving "pause everything" (2026-06-03 incident). It is the SINGLE loop-control plane: loop control is `/loops` (`t3 loop enable/disable/pause/resume`) + the DB only ‚Äî there is no env kill-switch (the former in-process `T3_LOOPS_DISABLED` is gone) and no `[loops]` toml disabled-state tier. The shared read `loop.loop_state_db.loop_held_in_db` is a `domain` leaf and every DB read fails open/safe so a hiccup never silently disables a loop or crashes the Stop hook. **Toward autonomous DB-configured loops ([#1796](https://github.com/souliane/teatree/issues/1796)).** A `Loop` row (`teatree_loop`: `name`/`prompt`/`delay_seconds`/`enabled`/`last_run_at`) is the durable definition of one autonomous loop ‚Äî the DB-configured replacement for the code-cadence tick, runnable on its own `delay` (a never-run loop is due immediately). The `prompt` is the instruction to run the loop's work each tick (e.g. "run a sub-agent to call the work scan"): the domain scanners under `teatree.loops` stay as the Python code the loop invokes ‚Äî they are scan units, not separate loops. A data migration seeds one autonomous row per loop, each with its own cadence ‚Äî a fixed `delay_seconds`, or a `daily_at` wall-clock time for once-per-day loops (`news` at 08:00, `dream` at night); `t3 setup` re-runs an idempotent `get_or_create` seed (`seed_loops`) so a fresh or squashed-migration install also has the defaults. Loops are not only the registry scanners ‚Äî the reactive Slack-answer loop is one too. A loop's `prompt` is a first-class reusable `Prompt` row (`teatree_prompt`, FK from `Loop.prompt`): a named instruction with declared templated `params` (`render` substitutes only declared `{name}` tokens, leaving other braces literal) and a `PromptVersion` edit history (`revise` snapshots the superseded body+params, keyed `(prompt, version)`). Prompts are triggerable on their own via `t3 prompts list|render` (the `/prompts` skill); loops are read and triggered via `t3 loops list|tick` and `t3 loop list` (the `/loops` skill). Rows are managed from the Django admin. Each default loop carries a real one-line `description` (what it does and when) ‚Äî the install seed (`teatree.loops.seed.DEFAULT_LOOPS`) is the source of truth that populates `Loop.description` (and the prompt-backed loop's `Prompt.description`), a data migration backfills it onto existing rows, and `t3 loops list` renders it as a continuation line under each loop's status row. **The `Loop` table carries cadence + the enable toggle for the live tick (#2513 cutover), gated by the unified verdict (#2584).** The live loop tick selects its scanner jobs from the `Loop` rows via `build_loop_table_jobs` ‚Äî a loop runs iff its row is `enabled` AND `is_due` AND `LoopsConfig.is_enabled` agrees (the DB `LoopState` control tier), bumping `last_run_at` only for the loops it admits ‚Äî and the statusline + `t3 loop list` read the `Loop` table too. The legacy code-cadence path (a registry fan-out gated by a `MiniLoopMarker` cadence ledger) is **deleted** (LOOP-PR-A) ‚Äî `Loop.last_run_at` is the single cadence ledger and the `Loop` table the single source of truth for the live tick + live status. **Each `Loop` row carries its OWN on-disk entry point ‚Äî the `script`/`prompt` column is load-bearing (#2513).** A script-backed row's `script` is its OWN module `src/teatree/loops/<name>/loop.py` (the file exposing that loop's `MINI_LOOP`), never a value shared across rows ‚Äî the retired shared `run.py` runner is gone. `build_loop_table_jobs` reads each admitted row's column to decide what to dispatch (a script row resolves its `script` path UP to the loop name via the one normalization seam `teatree.loops.run.parse_script_loop_name` + the live registry; a prompt row dispatches its own loop), so which behaviour fans out is decided by the row's column, not by a name-only registry lookup. A row whose `script` does not resolve to a real registered loop module raises loudly and is logged + skipped ‚Äî never a silent no-op. `arch_review` is the one prompt-backed default; its prompt instructs a sub-agent to run an architectural review with the `ac-reviewing-codebase` skill. #2584 closes the #2513 gap where `build_loop_table_jobs` gated on `Loop.enabled` alone and silently ignored `t3 loop pause`/`disable`: the loop tick now routes through the SAME `LoopsConfig.is_enabled` (`LoopState`) verdict the review-claim chokepoint reads, checked BEFORE `mark_run` so a held loop's cadence anchor is preserved. The inverse gap is closed too ‚Äî the per-loop runner (`t3 loops tick --loop <name>`, #2650) scopes the SAME `build_loop_table_jobs` gate to one row, so it honours `Loop.enabled` symmetrically. The cutover is plumbing only: it ships every loop **PAUSED** (the default-loops seed lands every row `enabled=False` ‚Äî folded into the squashed `0001_initial` and re-run by the idempotent `seed_loops` install-seed) so nothing ticks until an operator deliberately enables a row ‚Äî a re-enabled row is preserved by the idempotent `get_or_create` seed. `LoopState` (pause/disable) beats the row across the gating planes. There is no `[loops.<name>]` toml cadence tier and no `MiniLoopMarker` ledger (LOOP-PR-A deleted both) ‚Äî `Loop.last_run_at` is the single cadence ledger for BOTH the live tick and the off-live-tick `dream` cron (which gates on the dream row's `is_due`/`last_run_at` + the unified enable verdict). The seeded `Loop`-table set equals `iter_loops()` (#2584: the orphan `slack_answer` row is never seeded ‚Äî it has no registry `MiniLoop` and runs only via its dedicated `loop-slack-answer` `/loop` slot). **The worker owns the per-loop tick cadence ([#2650](https://github.com/souliane/teatree/issues/2650) / [PR-28]) ‚Äî one timer chain per enabled row, no master runner.** There is no single shared-tick cron and no `t3 loops run` interval loop: the singleton `t3 worker` drains one self-rescheduling `loop_timer(name)` chain per ENABLED `Loop` row (PR-28 default-ON), each firing `t3 loops tick --loop <name>` on that loop's own cadence (the per-loop primitive scopes the tick to that one enabled, due row, fans it out through the shared `run_tick` pipeline gated by `build_loop_table_jobs` on the row's `is_due` and bumping `last_run_at`, and claims the disjoint per-loop `loop:<name>` lease so the loops run in parallel). Running `t3 loops tick` with no `--loop` is a hard error. PR-28 retired the native Claude `/loop` cron mirror (`claude_specs`, `t3 loop claude-spec`/`verify-cron`, the `handle_enforce_loop_registration` nudge gate): the worker is the sole DB-loop cadence owner, so there is no `CronCreate`-per-enabled-loop registration and no fallback plane ‚Äî flipping `loop_runner_enabled` OFF stops the loops entirely. Enabling/disabling a loop moves the DB planes (`t3 loop enable|disable <name>`) and the reconciler adds/prunes that loop's timer at the enable/disable chokepoint at once. **Self-owned worker ‚Äî `t3 worker` + self-rescheduling timer chains ([#1796](https://github.com/souliane/teatree/issues/1796), default ON ‚Äî [PR-28]).** The singleton `t3 worker` owns the tick cadence by default: with `loop_runner_enabled` ON (the shipped default since PR-28) `SessionStart` re-spawns a detached worker whenever the `worker` flock is free (the SessionStart resurrector `hooks/scripts/worker_supervisor.py`, whose Django-free cold-read default is pinned equal to the dataclass default so a fresh install actually spawns), and the worker is the sole cadence owner. `loop_runner_enabled` is now the **kill-switch**: flipping it OFF is the instant runtime escape that stops the loops entirely ‚Äî there is no fallback plane (PR-28 retired the legacy native-`/loop` cron mirror), though the `default`-queue drain still runs under OFF via the reactive drain loop, so OFF halts loop ticks without stranding queued FSM/headless work. The cadence is durable, crash-surviving state ‚Äî exactly one pending `loop_timer(name)` django-tasks row per enabled `Loop` row on the dedicated `loops` queue is a scheduled `run_after` timer; the DB row IS the fire, so a crash never loses a beat. When a worker executor drains a timer it runs a five-step body that re-schedules its own successor BEFORE doing the tick work (self-dedup ‚Üí successor-first re-enqueue ‚Üí live `admitted_loop_names` admission check ‚Üí a deadlined subprocess tick in its own process group, deadline `max(300s, 3√ócadence)`, whole-group kill on expiry ‚Üí post-tick cadence refinement from the fresh `mark_run_if_unchanged` anchor), so a crash mid-tick always leaves a queued successor and the chain never stalls. The `mark_run_if_unchanged` CAS stays the sole idempotency guard (an at-least-once double delivery is a no-op). A deterministic zero-token reconciler `ensure_loop_timers()` keeps the "exactly one pending timer per enabled loop" invariant true ‚Äî adding a missing chain head, pruning a surplus timer, repairing a chain stuck RUNNING past its deadline, deleting a disabled/unknown loop's timers ‚Äî and runs at worker startup, on its own ~5-minute self-rescheduling chain, and from the loop enable/disable chokepoint; a daily `prune_task_results` chain caps DBTaskResult table growth, and an hourly `expire_stale_jobs` chain keeps the `default`-queue backlog swept for a long-lived worker. Worker startup ALSO expires the stale `default`-queue backlog once BEFORE spawning any executor (`expire_stale_default_jobs`, scoped to `default` so the reconciler stays the sole owner of `loops`-queue timer staleness), so a box that queued days-old provision/ship jobs while no worker ran never blind-fires them the instant the default-ON worker spawns (the flip's load-jam class). The worker runs K=4 programmatic `django_tasks_db` executor threads ‚Äî 2 pinned to `loops`, 2 to `default` ‚Äî so a heavy headless job can never starve a reactive timer; a supervisor thread re-reads `loop_runner_enabled` every ~5s and stops every executor on a flip-off or SIGTERM. At-most-one comes from the `worker` flock singleton; at-least-one from the worker supervisor + reconciler (NO OS scheduler ‚Äî no cron/launchd/systemd) plus the `SessionStart` resurrector (`hooks/scripts/worker_supervisor.py`), which re-launches the worker whenever the flock is free; a fully-headless box starts `t3 worker` once from a login profile. ### 5.7 Self-Improving Monitor

A detector swarm that rides the same tick the regular `/loop` runs. It watches for smells the rest of the loop cannot self-report ‚Äî dispatcher silently skipping a phase, a `MergeClear` issued but never reconciled, a statusline entry whose evidence has gone stale ‚Äî and converts each into a `SelfImproveFiring` row plus a graduated action (`log ‚Üí statusline ‚Üí slack ‚Üí ticket ‚Üí auto_fix`, monotonic ladder). It is the legibility substrate ¬ß¬ß17.4‚Äì17.8 relies on. Auto-fix is whitelisted: today only `StaleStatuslineEntryDetector` carries `auto_fix = True`, and its self-heal is wired for real ‚Äî the prior `_default_rerender` no-op is retired (#2625) in favour of the idle render seam `teatree.loop.phases.render.rerender_statusline`, which re-renders the zones file so a merged-PR / terminal-ticket URL drops out. Because the heal is idempotent and side-effect-free, the detector declares `max_rung = auto_fix` and the action ladder sends an `auto_fix` report straight to its ceiling on the FIRST observation ‚Äî a graduated climb would be pointless (surfacing a stale statusline on the statusline it is about to re-render) and would in fact never reach `auto_fix`, since a persistent smell keeps the same `state_hash` and the cool-down would pin it at rung 1 forever (#2625 Part B). The detector stays in the domain layer and never reaches up into the orchestration render seam (tach-enforced): instead the orchestration entry point that drives the cheap self-improve tier ‚Äî the dedicated `loop_self_improve` slot ‚Äî injects `teatree.loop.phases.render.self_improve_rerender` as the ladder's `auto_fix_callable`; the schedule's `_detector_auto_fix` adapter (routing a detector's own `rerender`) is the fallback for a directly-constructed detector with no injected seam. This covers stale *entries*; whole-render *age* (a dead loop, when no tick runs at all) is caught instead by the read-path freshness banner since the detector cannot fire when the loop is not ticking. The shipped detector set (`detectors/registry.py`) is `DispatchGapDetector`, `ForgottenMergeDetector`, `StaleStatuslineEntryDetector`; additional `auto_fix` slots land with their own structural whitelist test.

| `IssueDispositionScanner` ([#2122](https://github.com/souliane/teatree/issues/2122)) | `needs-triage` dead noise | Lists `needs-triage` open issues and auto-closes ONLY high-confidence DEAD ones via three deterministic buckets ‚Äî `already_shipped` (a delivered ticket exists, falsified by a live in-flight ticket), `exact_duplicate` (title fingerprint matches another open issue, falsified by a unique fingerprint), `obsolete` (every body-referenced path is gone, falsified by one still-existing path). Conservative bar: ANY uncertainty emits nothing. Emits `issue_disposition.close_candidate` ‚Üí the idempotent `close_dead_issue` mechanical handler closes with an audit-trail comment; bounded `auto_disposition_max_closes_per_tick`. **Default-OFF** behind `auto_disposition_enabled` (per-overlay overridable). It can CLOSE noise but is physically unable to enqueue work ‚Äî no `Task`, no claim, never an agent route ‚Äî so the burn-down doctrine (done-signal = open-bug count, never an empty queue) is structurally preserved. |
| `SnapshotWarmerScanner` ([#2949](https://github.com/souliane/teatree/issues/2949)) | reference-DB DSLR snapshot rot | Checks every overlay-declared `DjangoDbImportConfig` (`OverlayBase.get_snapshot_warmer_configs`, default empty) for staleness ‚Äî a DSLR snapshot's own `YYYYMMDD_<tenant>`-embedded date doubles as its "last refreshed" timestamp, no separate marker. A stale/missing snapshot emits `snapshot_warmer.refresh_needed`; the `refresh_snapshot` mechanical handler restores the newest snapshot, migrates, and re-snapshots only when migrations were applied (`teatree.utils.django_db.refresh_reference_snapshot`) ‚Äî absorbing the slow restore+migrate path out-of-band so a ticket-critical-path provision never pays it. Daily cadence (its own mini-loop, `snapshot_warmer_max_age_days`); `snapshot_warmer_disabled` kill-switch; mechanical-only. |

`CodexReviewScanner`'s and `PrSweepScanner`'s auto-dispatch is gated on the fleet doctrine (`mode = "auto"` + `require_human_approval_to_merge = false`); every other overlay stays manual. All scanner knobs are per-overlay overridable.

The monitor never auto-merges substrate, never auto-edits memory / skills / `BLUEPRINT.md`, and never bypasses the ¬ß17.4 `MergeClear` reviewer-attestation **except** on a solo overlay the user declared end-to-end-trusted (`mode = "auto"` + `require_human_approval_to_merge = false`) ‚Äî where the per-diff CLEAR cannot be issued because maker and reviewer are the same human identity, so an unrelaxed gate would silently no-op every green PR ([#1309](https://github.com/souliane/teatree/issues/1309)). The carve-out is *minimal*: every precondition gate (draft, changes-requested, CI, uv-audit escalation) stays in force, the per-diff CLEAR is replaced by the SHA-bound `merge_pr_squash_bound` fallback (#1985, re-checking live SHA-bind/draft/CI), and the **cold-review floor still holds** ‚Äî the bypass requires a recorded INDEPENDENT `merge_safe` `ReviewVerdict` at the live head (`reviewer != maker`) ([#68](https://github.com/souliane/teatree/issues/68)). Non-opted-in overlays keep the CLEAR requirement.

**Auto-review dispatch closes the no-review loop ([#68](https://github.com/souliane/teatree/issues/68)).** On `flag_no_review` for a green+clean own PR the scanner enqueues ONE claimable `Task(phase=reviewing)` (gated on `auto_review_dispatch = solo_overlay and not require_human_approval_to_merge`); the reviewer's recorded `merge_safe` verdict is the artifact the sweep merges on. Because a reviewing-phase headless dispatch is denied the shell (PR-11), the reviewer cannot run `review record` ‚Äî it **RETURNS a typed `review_verdict` in its result envelope**, and the orchestrator (`attempt_recorder.record_result_envelope` ‚Äî a DIFFERENT actor, so maker‚âÝchecker holds by construction) records the `ReviewVerdict` server-side from it, resolving the per-MR `MRReviewLock` ([#1405](https://github.com/souliane/teatree/issues/1405)) so the chain never stalls with the lock held; the interactive/CLI path still uses `review record`. The merge is **event-driven, not cadence-bound** ([#2026](https://github.com/souliane/teatree/issues/2026)): the CLI `review record` calls `trigger_sweep_for_verdict` the moment a `merge_safe` verdict lands, rebuilding the overlay sweep scanner and running its single-PR `evaluate_one` ‚Äî the same decision ladder `scan` runs, so the two paths cannot drift; the periodic sweep is the backstop that merges the server-recorded verdict on the headless path. Dedup/contract mechanics live in `AutoReviewDispatch`'s docstring.

**Review loop FSM ‚Äî the bounded author‚Üîreviewer ping-pong ([#2298](https://github.com/souliane/teatree/issues/2298)).** `ReviewLoop` is an independent FSM (`AUTHORING ‚Üí REVIEWING ‚Üí {PASSED, EXHAUSTED}`, NOT a `Ticket.State` transition) that encodes iterate-then-terminate review: each verdict-guarded transition reads the *recorded verdict*, never the caller, so a HOLD re-arms a fresh author leg carrying the reviewer punch-list (bounded by `max_rounds`; `exhaust` surfaces `needs_user_input`). The verdict recording drives it (`_advance_review_loop`, shared by the CLI `review record` and the headless orchestrator's server-side recording of a returned `review_verdict` envelope), closing the pre-#2298 gap where a HOLD was inert. Two variants: the **SELF** variant records an internal verdict and posts nothing (no `ReviewVerdict`, no egress); the **EXTERNAL** variant binds a real `ReviewVerdict` through the cold-review path and `pass_` sets the `passed` proceed flag (it does NOT itself merge ‚Äî that stays the sweep's job), and runs the `e2e`/`e2e_reviewing` phases (author `t3:e2e`, reviewer `t3:e2e-review`). Leg dispatch is idempotent per `(review_loop, round, leg)` via the `ReviewLoopRound` unique constraint.

**Delivery-ownership predicate stops double-scheduling externally-delivered units ([#2104](https://github.com/souliane/teatree/issues/2104), [#2217](https://github.com/souliane/teatree/issues/2217)).** A unit delivered out-of-band by a hand-dispatched delivery agent (per `/teatree-batch`) is implemented directly with no planning phase and no loop-armed reviewer, so the loop must not re-derive that work each tick. `external_delivery.under_external_delivery(ticket)` is true while a TTL'd `external_delivery` lease on `Ticket.extra` is live; it is claimed at the `workspace ticket` entry the delivery agent runs (never by the loop's own FSM) and refreshed on each external-owner seam (`ticket plan`/`transition`), so a hand delivery cannot lapse mid-flight while a crashed owner still self-reaps on the 1h TTL. The lease is consulted at **three** chokepoints: `execute_provision` skips `schedule_planning`; the `pr_sweep` review-arm no-ops (resolving the PR back to its author/delivery ticket via `resolve_author_ticket`, never by `issue_url == pr_url`); and the global dispatch chokepoint `orchestrate._dispatchable_filter` excludes live-lease tickets at the DB layer via `not_under_external_delivery_q` (held in lockstep with the Python predicate by a parity test). Without the third, hand-advancing STARTED ‚Üí PLANNED made any non-planning phase dispatchable and a second coder was claimed (the #2217 incident).

**Filesystem-evidence double-dispatch guard ([#2217](https://github.com/souliane/teatree/issues/2217)).** The lease above only protects a unit the DB knows about; the #2217 incident produced a worktree on disk with **no `Ticket` row**. `worktree_collision.find_foreign_issue_worktrees` keys on the issue number instead ‚Äî the dir for issue `N` is deterministically `<N>-<slug>`, so globbing `<N>-*` finds any existing worktree regardless of DB state. The single chokepoint is `workspace ticket`: inside the same `atomic()` that creates the ticket, a foreign `<N>-*` dir raises `ForeignIssueWorktreeRefusedError`, rolling the ticket back so a refusal leaves zero DB trace; the CLI prints the colliding path with a `--take-over` opt-out (the ticket's own dir is excluded, re-provision is idempotent). It is the filesystem-keyed complement to the DB lease.

**Adopt outside-originated work ([#2275](https://github.com/souliane/teatree/issues/2275)).** `workspace ticket --adopt` / `--adopt-branch <branch>` registers a Ticket + Worktree against an EXISTING branch instead of deriving `<number>-<slug>` ‚Äî for work that originated outside the derive-a-branch flow. Run from inside the checkout, the branch auto-detects (`resolve_adopt_context` reads the branch, the worktree root, and the repo slug from git; `--adopt-branch` names it explicitly, a detached HEAD is refused). The provisioner records the existing `worktree_path` verbatim (`Ticket.extra['adopt']` maps repo ‚Üí path, the backing clone derived from the checkout's own shared git dir when it lives outside the clone root) instead of `git worktree add`, so no second worktree dir is created; the `create_branch=False` fallback still covers checking a pre-existing branch out into a fresh dir. The foreign-dir guard is skipped like `--take-over` (adopt is an explicit "use this checkout" intent).

**Lightweight trivial-work plan-skip carve-out.** Some AUTHOR tickets are trivial mechanical edits (a typo, a one-line constant) where planning is pure overhead. `trivial_plan_skip.mark_trivial_plan_skip(ticket, reason=‚Ä¶)` stamps a durable marker (no TTL) carrying a MANDATORY reason (a malformed/empty marker is treated as absent, fail-safe to "planning required"). Consumed at the same two seams ‚Äî `execute_provision` skips `schedule_planning`, and `check_plan_artifact` accepts the marker so the ticket advances STARTED ‚Üí PLANNED with no `PlanArtifact`. Recorded via `t3 <overlay> ticket skip-planning <id> --reason`. The lightweight sibling of the heavyweight `plan-bypass` (`--human-authorize`, which fabricates a `PlanArtifact`); both stay intact and scoped to a ticket that explicitly carries the marker.

### 5.8 Reactive Slack-Answer Loop

A tight-cadence (default 20s), token-cheap reactive `/loop` slot that answers user DMs out-of-band so a quick ack / status question gets a reply in seconds, not at the next slower per-loop tick. Coalesces consecutive same-user messages into one logical turn, classifies (pure Python) into `ACK_ONLY` / `SIMPLE` / `NEEDS_WORK`, and either reacts, posts a threaded reply, or delegates to the `t3:answerer` sub-agent.

## 6. Overlay System

An overlay is a downstream Django project that customizes teatree for a specific project/organization.

**¬ß6.0 Overlay Thinness Principle (Non-Negotiable).** Generic workflow logic belongs in core, not in overlays. Before adding logic to an overlay, ask: "Would a different project using the same framework need the same logic?" If yes, it belongs in core ‚Äî parameterized and configurable. Overlays provide only: (1) configuration values, (2) project-specific glue, (3) truly unique workflows. Everything else ‚Äî DB provisioning strategies, migration runners, symlink management, service orchestration ‚Äî is a configurable engine in core; the overlay configures it, never reimplements it. An overlay method exceeding ~30 lines of non-configuration code likely contains generic logic to extract.

**OverlayBase ABC (`teatree.core.overlay`).** Composition over inheritance: `overlay.config` is an `OverlayConfig` dataclass; `overlay.metadata` an `OverlayMetadata`. All methods take a `worktree` for context.

| Required | Returns | Purpose |
|---|---|---|
| `get_repos()` | `list[str]` | Repositories to provision |
| `get_provision_steps(worktree)` | `list[ProvisionStep]` | Ordered setup steps |

Optional hooks cover env, services, Docker base images, DB import strategy, post-DB steps, symlinks, PR validation, sync targets, skill metadata, CI config, E2E config, variant resolution, tool subcommands, visual QA targets, E2E env extras + preflight, the per-worktree task commands (`get_test_command` ‚Üí `run tests`, `get_lint_command` ‚Üí `run lint`), the auto-merge guard (`can_auto_merge` ‚Üí `MergeGuard`), the snapshot-warmer configs (`get_snapshot_warmer_configs()` ‚Üí `list[DjangoDbImportConfig]`, default empty ‚Äî one per DSLR-backed variant/tenant the warmer loop keeps current out-of-band, [#2949](https://github.com/souliane/teatree/issues/2949)), and the ¬ß17.4 cross-repo merge probe's working-repo declaration (`get_merge_candidate_repo_slugs()` ‚Üí `list[str]` of `owner/repo` slugs for repos the overlay operates merges on but does not package, so a CLEAR whose PR lives in one of them can be bound ‚Äî [#2323](https://github.com/souliane/teatree/issues/2323)). All default to empty / permissive so existing overlays keep working ‚Äî core enforcement only activates for overlays that opt in.

**Typed provisioning substrate ‚Äî `ProvisionStep` DAG + `Variant` (PR-27, [#787](https://github.com/souliane/teatree/issues/787)/[#1385](https://github.com/souliane/teatree/issues/1385)).** `ProvisionStep` is a typed DAG node: `requires`/`produces` tokens order the steps (a `requires` token no step `produces` fails the run loud, fail-closed) and independent steps run concurrently, while `post_condition` is a truth check evaluated after the callable so a step that "ran" without producing its resource is recorded FAILED. The aggregate of every step's `post_condition` plus the core invariants (worktree dir, env cache, DB present) is `teatree.core.provision.provision_postconditions.aggregate_provision_post_conditions` ‚Äî the truth test `worktree status` runs on a `provisioned`/`services_up`/`ready` worktree, so deleting `.t3-cache/.t3-env.cache` or the DB out from under it makes `status` refuse green with a non-zero exit. **Variant** is a first-class frozen dataclass (`teatree.core.provision.variant.Variant`: `name`, `canonical_tenant`, `default_language`, `dslr_snapshot_name`, `e2e_credentials_key`) resolved through `OverlayBase.resolve_variant(name) -> Variant` ‚Äî no overlay hook takes a raw `variant: str`, so a tenant/snapshot/credentials name is never reconstructed from a bare string at a call site.

**Repo SCOPE axis (`owned_repos`).** `OverlayConfig.owned_repos` is a forge-host-keyed dict of the namespace patterns the overlay legitimately works on ‚Äî the **scope** axis, orthogonal to **visibility** (`private_repos`/leak-prevention, fails open) and **collaboration** (`author_is_self`). Classification (`teatree.core.intake.repo_scope`) gates host equality before the namespace half, so a `gitlab.com` repo can never match a `github.com` scope. The unknown-repo gate (`owned_repo_guard`) holds a push or keystone merge to a repo no opted-in overlay owns ‚Äî fails **CLOSED** on a clean unknown verdict (opposite the visibility gate) and **OPEN** on a resolver exception (never-lockout). It ships **INERT** (`require_owned_repo_approval` default `false`); enabling it requires first declaring the FULL owned host/namespace list in private `~/.teatree.toml`, since a partial list would hold the operator's own private merges as unknown. A path-only TOML overlay cannot opt itself in (its repos are declared under an instantiable overlay). The PreToolUse hook bootstraps Django before resolving overlays so the gate actually fires in production. Owned does NOT imply auto-merge: a shared product repo is in scope yet still needs colleague review.

**Overlay API version pin + `teatree.overlay_sdk` (PR-27).** `teatree.__overlay_api_version__` (now `"1"`) is bumped on any breaking change to the overlay-facing API. Overlays assert this at import to fail loudly when teatree diverges from what they were built against. `teatree.overlay_sdk` is the surface-frozen namespace an overlay imports from ‚Äî a single documented facade (`OverlayBase`, `OverlayConfig`, `ProvisionStep`, `Probe`, `Variant`, the config/util helpers) whose `__all__` and whose `OverlayBase` extension-hook signatures are snapshotted by `tests/teatree_overlay_sdk/test_overlay_sdk_surface.py` in **core's** CI, so an internal rename or a signature change fails core *locally* instead of breaking an overlay's own CI asynchronously.

**Docker base-image sharing (¬ß6.2a).** Teatree builds each `BaseImageConfig` once from the master clone's lockfile and tags it with the single master tag `{image_name}:base` (NOT per-lockfile) ‚Äî shared across every worktree. `ensure_base_image` (re)builds only when the image is **absent** (`docker image inspect` rc‚âÝ0) or **broken** (the tag resolves but its config Id does not ‚Äî a corrupt/interrupted build ‚Üí `rmi -f` then rebuild); a worktree provision otherwise never triggers a build. Code isolation is volume-mount level (`.:/app:rw`); dependency drift between the branch's lockfile and the baked master deps is reconciled at container start by the overlay's entrypoint (`uv sync`), so no per-lockfile image tag is needed.

**`t3 startoverlay <name> <dest>`** scaffolds a lightweight overlay package: `src/<name>/{__init__,overlay,apps}.py`, `skills/overlay/SKILL.md`, `pyproject.toml`. No `manage.py`/`settings.py`/`urls.py` ‚Äî teatree is the Django project.

**Discovery:** `~/.teatree.toml [overlays.<name>]` wins over `teatree.overlays` entry points on name conflicts. `discover_active_overlay()` returns the unique overlay if one exists, else the one whose `manage.py` is in cwd ancestors.

## 7. Backend Protocols and ABCs

Every external API concern is a `@runtime_checkable Protocol` in `teatree.core.backend_protocols` (the domain owns the abstractions it consumes; `backends` implements them). PR is the canonical term in core; GitLab MRs are translated at the API edge.

| Protocol | Implementations |
|---|---|
| `CodeHostBackend` ‚Äî PR/issue/comment (incl. `list`/`update_issue_comment`)/upload/review-state + the ¬ß17.4.3 merge-RPC surface (`fetch_live_head_sha`, `fetch_pr_merge_state`, `fetch_pr_is_draft`, `fetch_required_checks_rollup`, `merge_pr_squash_bound` ‚Äî raw forge I/O resolved via `core.backend_registry`; `merge.execution`/`merge.ci_rollup` keep the verdict/error classification) | `GitHubCodeHost`, `GitLabCodeHost` |
| `CIService` ‚Äî pipeline cancel/trigger/quality-check | `GitLabCIService` |
| `MessagingBackend` ‚Äî mentions/DMs/post/reply/react | `SlackBotBackend`, `NoopMessagingBackend` |

Request parameters are grouped into frozen `slots=True` dataclasses (`PullRequestSpec`, `MessageSpec`). `repo + pr_iid` is the natural unit on both code hosts ‚Äî protocol methods never accept free-form PR URLs.

**Selection.** Per-overlay configuration (`~/.teatree.toml`) declares `code_host = "github" | "gitlab"` and `messaging_backend = "slack" | "noop"`. The loader (`backends/loader.py`) resolves the overlay's selected backend with no platform branches in caller code, cached `lru_cache(maxsize=1)` per overlay identity.

**Inbound events.** `t3 slack listen` runs a Socket Mode receiver that writes events to append-only JSONL queues (`slack-events.jsonl`, `slack-reactions.jsonl`) so scanners can drain atomically without racing.

**Reaction surface (#1281).** FSM `reactions.add` failures (`missing_scope`, `not_in_channel`, `mcp_externally_shared_channel_restricted`, ‚Ä¶) raise `SlackReactionError` from `backends/slack/react_errors.py` ‚Äî never silently return `False` ‚Äî so callers cannot fall back to a `chat.postMessage(text=":emoji:")` thread reply. `SlackBotBackend.post_message` / `post_reply` reject bodies matching `^:[a-z0-9_+\-]+:$` with `SingleEmojiBodyRefusedError`, foreclosing the failure-mode shape at the backend boundary. FSM-side wrappers (`add_reactions_for_transition`, `add_approval_reaction`) catch the raise locally so Slack auth gaps cannot roll back FSM transitions.

**Destination-routed post/react ([#1750](https://github.com/souliane/teatree/issues/1750)).** Token picked by *destination* via `SlackBotBackend._route_token`: user's own DM ‚Üí bot; colleague/channel ‚Üí personal `xoxp`. Every colleague-surface post/react under the user's identity routes through the one gate+route+audit chokepoint `core/on_behalf_egress.OnBehalfSlackEgress` (self-DM acks ungated); the call-site authorization is pinned by the `on-behalf-routed-egress` / `on-behalf-colleague-primitives` entries in the chokepoint registry (`quality/chokepoints.yaml`, enforced by `scripts/hooks/check_chokepoints.py`); see [`skills/rules/SKILL.md`](skills/rules/SKILL.md).

**Deterministic reference linkifier.** The clickable-references rule (a bare `#N` / `!N` on a user-facing surface must become a clickable link) is enforced *in code* by `core/reference_linkifier.py`, not by asking the model. `ReferenceResolver` resolves DB-first (`PullRequest`, `Ticket.issue_url`) then constructs from the active repo context; `linkify` (idempotent; skips linked refs, inline code, fenced blocks) runs at the Slack chokepoint `SlackReplier._deliver`. Forge posts are NOT linkified (forge auto-links bare ids). It is the sole mechanism for the rule ‚Äî the former bare-reference *blocking* gate was removed (it over-blocked and asked the model to rewrite refs non-deterministically); the linkifier rewrites in code with no block and no model round-trip.

**Sync ABC (`core/sync.py`).** `SyncBackend` is an ABC with `is_configured(overlay)` and `sync(overlay) ‚Üí SyncResult`. Implementations: `GitHubSyncBackend`, `GitLabSyncBackend`. Both consume `CodeHostBackend` ‚Äî platform-specific code lives only in the Protocol implementation, not in sync logic.

## 8. Command Tiers

| Tier | Tool | Needs Django? | Examples |
|---|---|---|---|
| Runtime | django-typer management commands | Yes | `worktree provision`, `tasks work-next-sdk`, `followup sync` |
| Bootstrap | Typer CLI (`t3`) | No | `t3 startoverlay`, `t3 info`, `t3 ci cancel` |
| Overlay | Typer CLI delegating to `manage.py` via subprocess | Indirectly | `t3 <overlay> start-ticket`, `t3 <overlay> worktree start` |

Internal utilities (`utils/`) are Python modules, not a CLI tier.

**Runtime commands** (`core/management/commands/`): `lifecycle`, `tasks`, `followup`, `workspace`, `worktree`, `db`, `env`, `run`, `pr`, `ticket`, `tool`, `e2e`, `overlay`, `standup`, `signals`, `mr_reminder`, `checking`, `availability`, `retro`, `loop_tick`, `generate_*_docs`. Each is a django-typer command group with subcommands. `db query` and `db shell` enforce read-only at two layers (leading-keyword filter + transaction `READ ONLY` / `query_only=ON`).

**Retro enforcement tooling** (`t3 <overlay> retro review-findings`, [#1573](https://github.com/souliane/teatree/issues/1573)): the scaffold behind invariant 6 / ¬ß17.6. Fingerprints a PR's review comments, records the **supplied** A/B/C verdict (never auto-guessed), and files one deduped enforcement issue per class-C finding. The untrusted comment text is a leak vector, so its bare refs are neutralized and the rendered body banned-term scanned before filing ‚Äî the `gh api` stdin path bypasses the PreToolUse publish gate, so this is the only guard.

**Gate-failure feedback loop** (`t3 <overlay> retro gate-failures`, [#2024](https://github.com/souliane/teatree/issues/2024)): the review-findings sibling whose input is a **gate firing on the agent's own output**. A teatree gate BLOCK lands in the session transcript as a `hook_blocking_error` attachment whose text leads with a `TEATREE GATE ‚Äî <phrase>` marker. `eval.gate_failures.extract_gate_failures` reads the single transcript chokepoint, keys on the attachment type + marker (never `exitCode`), and reduces the gate identity to a bounded slug; `classify_gate_failure` routes each as `environmental` (a `hook_non_blocking_error` ‚Äî missing plugin dir, hook-runner traceback) or `preventable` (anything agent-output-shaped, and any unknown BLOCK ‚Äî fail toward an eval), and `--escalate` files one deduped enforcement issue per *recurring preventable* failure. Privacy by construction: `GateFailure` stores only the slug + session id, never the message/`stderr`/`command`. It lives in `teatree.eval` (a `core ‚Üí eval` import would be a backwards tach edge; `retro` calls in on a forward edge). Auto-invoking it from the loop tick is orchestrator-only ([#837](https://github.com/souliane/teatree/issues/837)), a separate change.

**Intake landscape survey** (`t3 <overlay> workspace landscape`, [#2541](https://github.com/souliane/teatree/issues/2541)): the read-only, gather-once half of ticket intake. Before the planner designs a plan, `teatree.core.intake.landscape.survey_landscape` composes three deterministic signals through narrow consumer-defined code-host Protocols (`PrLister`, `MergedPrLister`) ‚Äî the local worktree state (dirty / unpushed checkouts, both failing *open* so in-flight work is never silently hidden), the operator's open PRs, and a per-issue `done|partial|superseded|open` disposition with a `close|merge|supersede|keep` action. The classification is a mechanical floor (a *merged* PR naming the issue ‚áí done/close ‚Äî the resolved-but-open path; an *open* PR ‚áí partial/merge; otherwise open/keep); the planner agent refines the verdict from the prose. The `run_landscape` command renders a JSON `LandscapeReport`; every forge probe failure degrades to a `warnings` entry rather than aborting intake. Distinct from ¬ß9's checking report (a mid-loop catch-up) ‚Äî this is the pre-plan landscape the `/t3:ticket` step surfaces.

**Checking report** (`t3 <overlay> checking show`, #1529): a terse, read-only "what did I miss" catch-up for when the user checks in mid-loop. By default aggregates ALL configured overlays into one `AllOverlaysReport`; `--this-overlay` restores single-overlay scope. Each overlay has its own `checking_checkpoint_<overlay>.json` marker (atomic `tmp.replace` write, tolerant read); each marker advances independently AFTER gathering so a second run sees an empty window. Three groups: Merged / In-flight / Needs you, every reference clickable, capped at 5. `DeferredQuestion` is queried ONCE for the whole report; overlay-scoped items carry an `[overlay]` inline tag. A window start at/after `now` falls back to the 24h lookback; advance is monotonic. `--since` and `--no-advance` inspect without advancing. The needs-you group is overlay-extensible via `OverlayBase.get_checking_sources()` (default `[]`); core makes no live forge calls.

**Factory signals** (`core/factory_signals.py`, `t3 <overlay> signals`, MCP `factory_signals`): five derived-on-read quality/velocity signals over the merge/review/CI/repair ledgers already recorded ‚Äî mirrors `core/standup.py`'s read-only-aggregator doctrine (no new models, no migrations, no LLM calls). Each signal is a **provider-shaped** function returning `SignalReading(value, sample_size, window_days, status)` with `status ‚àà {ok, insufficient_data, instrumentation_gap}` ‚Äî fail-loud, never fabricating a green (sample `< MIN_SAMPLE=5` ‚Üí `insufficient_data`; a provably-silent upstream recorder ‚Üí `instrumentation_gap`, never 100%). The five: **S1 first_try_green** (`MergeAudit` merges with ZERO `RedMrFixAttempt` rows, joined via `core/merge/pr_slug_resolution.normalize_repo_slug` + `pr_ref_from_url`; a dead `my_prs` recorder ‚Üí `instrumentation_gap`, plus the `re_ci_count` companion); **S2 defect_escape** (fix-kind tickets + `RedCardSignal` rows in the window over the PRECEDING window's merges ‚Äî window-level, not per-PR); **S3 review_catch** (fraction of merged PRs whose `ReviewVerdict`/`for_pr` history holds or surfaces a blocker/major `Finding`; ‚â•5 merges at catch-rate 0 ‚Üí hard RED "vacuous review lane" ‚Äî the rubber-stamp detector); **S4 merge_latency** (median `MergeClear.issued_at`‚Üí`MergeAudit.merged_at` exact FK join + the stale-actionable-CLEAR age, RED above 48h even in a zero-merge window); **S5 repair_burn** (mean terminal `TaskAttempt.iteration` per succeeded `(ticket, phase)` + the failed-attempt fraction). `compute_factory_signals(window_days=28)` composes the readings against the immediately-preceding window as a rolling baseline into a `FactorySignalsReport` (per-signal red-floor `tripped` + rolling-baseline `regressing` verdicts, one top-level verdict ‚àà {ok, regressing, red}); starving a signal can only lower the score ‚Äî an `instrumentation_gap` is RED, never a free pass. The five providers are a star-shaped `SIGNALS` registry the recipe (`evals/recipe.yaml`, ¬ßFactory score below) names verbatim; the module produces signal only ‚Äî no gate, no merge lockout.

**Factory score** (`core/factory_recipe.py` + `core/factory_score.py`, `t3 <overlay> recipe score|approve`, MCP `factory_score`, flag-gated behind the DARK `factory_score_enabled` ‚Äî SIG-PR-2): the recipe-weighted aggregate the T4 outer loop optimises against, composed over the ¬ßFactory signals `SIGNALS` seam READ-ONLY (`compute_factory_signals` + the `SignalReading`/`SignalRow` dataclasses ‚Äî never re-deriving a query). The committed `evals/recipe.yaml` names each SIGNALS id with a `weight` (the five sum to 1.0), an optional `red_when` floor, and a normalisation `cap` for the two magnitude signals (`merge_latency` hours, `repair_burn` iterations); `load_recipe()` fails loud (`RecipeError`) on drift ‚Äî a missing file, an unknown/missing id, weights off 1.0, or a missing/forbidden cap ‚Äî so the file can never diverge from the registry (git is the audit log for weight/floor changes). `score()` folds worst-wins honesty structurally: any `instrumentation_gap` or RED row ‚Üí verdict RED; a covered reading below its `red_when` floor ‚Üí RED; an uncapped rate outside `[0,1]` ‚Üí RED (never clamped); coverage below `coverage_floor` ‚Üí `aggregate=None` + RED ‚Äî an untrustworthy score is never a number, and a RED verdict always carries a `None` aggregate. Each scored read persists one `FactoryScoreSnapshot` (`recipe_sha` + `tree_sha` provenance, per-signal evidence) whose `delta_vs_previous` and `delta_vs_last_different_recipe_sha` give the outer loop its time-series ‚Äî the cross-recipe-sha delta is what stops a re-weighting from hiding a regression behind its own sha. Recipe evolution is doubly human-gated: the committed-file PR diff, plus `approved_recipe_sha` written ONLY by the human-run `t3 <overlay> recipe approve`; an unapproved sha stamps `recipe_approved=false` on every payload and snapshot and queues exactly one deduped `DeferredQuestion` per new sha. Ships flag-gated OFF: `recipe score` COMPUTES read-only (to calibrate weights against real ledger data pre-enable), but `--record` refuses, zero snapshot rows are written, zero questions are queued, and `build_server()` does not register the MCP `factory_score` tool ‚Äî the empty migrated table is the only footprint (the `ConfigSetting` empty-table doctrine). Per the repo doc-alignment rule, this section moves with the code.

**Autoresearch outer loop** (`core/models/outer_loop_experiment.py` + `loops/outer_loop/`, `t3 <overlay> outer status|propose|history|tick` ‚Äî T4-PR-3): the self-improving runtime that turns the factory score into an optimisation target. It is a two-loop frame ‚Äî the INNER maker pipeline builds one ticket into a merged PR; the OUTER loop advances at most one `OuterLoopExperiment` one FSM step per tick through **propose ‚Üí ratify ‚Üí implement ‚Üí measure ‚Üí keep-only-if-better**, consuming `factory_score.score()` (the T4-PR-2 metric-to-beat) as its measure. Implement rides the exact `schedule_gap_fix` shape ‚Äî a synthetic-issue `Ticket` + `Ticket.schedule_coding()` ‚Äî so an experiment's PR merges through the SAME critic-gated keystone every ticket uses; the outer loop gains ZERO new merge authority. The keep rule is pure and anti-Goodhart (`loops/outer_loop/decide.py`): KEEP iff the target signal improved beyond its own `regress_band` AND nothing else regressed vs the admission baseline ‚Äî a non-improving or collateral-damaging experiment is never kept, it moves to `REVERT_PENDING`. The human is embedded STRUCTURALLY: `admit()` is the only writer of the `ADMITTED` state and RAISES unless a consumed (answered) ratify `DeferredQuestion` exists ‚Äî there is no auto-admit code path. Revert is likewise human-ratified and drives to a TERMINAL state (no dead-end, no permanent slot hold): a `REVERT_PENDING` tick records a revert `DeferredQuestion` (`loops/outer_loop/revert.py:ask_revert`), the operator performs the git revert and runs `t3 outer resolve-revert <id>`, which consumes that question and calls `record_reverted()` ‚Üí `REVERTED`, freeing the max-concurrent slot for the next experiment (full AUTO-revert is a later PR). **Ships QUADRUPLE-OFF** ‚Äî a full tick is a proven no-op at defaults: (1) the DARK `outer_loop_enabled` feature flag is off (G1); (2) the seeded `outer_loop` `Loop` row lands disabled (`t3 loops enable outer_loop` required); (3) `off_live_tick=True` keeps it off the live fan-out (its own cron drives it, like `dream`); (4) the guards that matter are CODE not config ‚Äî G2 (`critic-live`) refuses every tick until a `CriticVerdict`-backed critic gate is registered and has `>= MIN_CRITIC_SAMPLE` verdicts (a defensive lazy lookup that fails CLOSED when it cannot confirm a live critic), and G3 (`signal-trust`) refuses while ANY factory signal reports `instrumentation_gap` (never optimise an untrustworthy score). G1b additionally refuses while `factory_score_enabled` is off ‚Äî the loop has no metric and must not write a `FactoryScoreSnapshot` (T4-PR-2's empty-table-when-off doctrine). G4 bounds the run (`precheck_budget` + max-concurrent=1 + weekly cap + a convergence brake that parks after N consecutive non-kept decisions). The ledger row binds hypothesis ‚Üí shas ‚Üí both score snapshots ‚Üí the human decisions; kept = a merged sha, rejected = a closed PR, reverted = a revert commit ‚Äî git is the audit log; the operator-facing liveness surface is `t3 outer status`/`history` (the typed tick result names the refusal reason). **Known weakest link:** MEASURE is a time+merge-count window, NOT causal attribution ‚Äî a horizon-window delta is confounded by unrelated merges, so a KEPT decision is "correlated better", not "proven caused"; the no-regression-anywhere rule and the human-ratified revert bound the risk. Per the repo doc-alignment rule, this section moves with the code.

**Overlay contract check** (`t3 overlay contract-check --compose <paths>`) reads every `${VAR}` reference in compose files and fails if any is neither defaulted nor declared by core (`_declared_core_keys()`) or the active overlay (`OverlayBase.declared_env_keys()`).

**Teatree source resolution in overlays.** `[tool.uv.sources] teatree = { path = "../../souliane/teatree", editable = true }` is the committed default ‚Äî no SHA pinning, no mode switching. CI clones teatree at the same relative path before `uv sync`. Local dev uses whatever is checked out locally.

**Machine interface ‚Äî `--json` / stable exit codes ([PR-30](https://github.com/souliane/teatree/issues/2565)).** `t3` is a machine interface a front-end (Pi, a CI runner, another agent) drives by shelling to `t3 ‚Ä¶ --json` and parsing stdout. The contract: under `--json` stdout is a **pure JSON channel** (valid `json.loads`, zero human bytes); the human view (tables, banners, diagnostics) always goes to **stderr**. `teatree.core.machine_output.emit(payload, *, json_output, out, err, human)` is the single seam every record/query command routes through ‚Äî it serializes the typed return via `json.dumps` on stdout under `--json` and renders the human view on stderr otherwise. A converted command sets `print_result = False` so django-typer does not additionally repr the return onto stdout, while `call_command` consumers still receive the typed object (the same typed return PR-18's `print_table` renderer consumes ‚Äî one seam, not two conversions). **Exit codes:** subcommands raise `SystemExit(N)` (never `typer.Exit`, swallowed under `call_command`); the overlay bridge (`cli/overlay.py` `managepy`/`managepy_core`) catches `CommandFailedError` and re-raises `SystemExit(returncode)`, so a bridged child's exit code propagates faithfully with no traceback instead of collapsing to a stack-trace exit 1. `t3 capabilities --json` publishes the machine-readable capability registry ‚Äî each command's `--json` support and exit-code contract ‚Äî so a front-end discovers the interface instead of scraping `--help` (the registry, not BLUEPRINT prose, is the SSOT for the converted-leaf set: `queue status`, `tasks list`/`create`, `followup sync`, `workspace emit`, `worktree status`/`diagnose`, `availability show`, `questions list`, plus the pre-existing JSON commands). A record/query command that returns a human string keeps the `checking show` precedent (`json.dumps` under `--json`, human string otherwise, both on stdout); a command that returns a typed object routes through the `emit` seam (JSON stdout / human stderr) ‚Äî both keep stdout a pure JSON channel under `--json`.

**Golden-path lifecycle wrapper ‚Äî `t3 <overlay> do <ticket-ref>` ([PR-31](https://github.com/souliane/teatree/issues/2565)).** One command walks a ticket through the lifecycle (intake ‚Üí provision ‚Üí plan ‚Üí code ‚Üí test ‚Üí review ‚Üí ship) so a front-end (or a human) never has to remember the sequence. It is a **thin** front-end-seam wrapper over the EXISTING chokepoints ‚Äî it SEQUENCES and REPORTS, it adds no gate and never touches the FSM directly. The step order and the `Ticket.state`‚Üístatus mapping live in `teatree.core.lifecycle_pipeline` (`PIPELINE` + `drive`, kept pure and side-effect-free against injected `DriveSeams`). Two step kinds split the walk: `auto` steps (intake=`workspace ticket`, provision=`workspace provision`, ship=`pr create`) are invoked in-process via `call_command` (child streams redirected to stderr so `do`'s stdout stays a pure JSON channel); `agent` steps (plan/code/test/review) are reported `pending` and the walk stops ‚Äî dispatching the phase agent is the loop's job (¬ß5.2), not this wrapper's. **Resumable + idempotent:** it re-reads `Ticket.state` each step and resumes at the live phase, never redoing a completed one; a gate-refused auto step surfaces the blocker and stops. `--json` emits the per-step typed status through the `emit` seam (steps array + `stopped_at`/`stopped_reason`); `--plan` prints the sequence with no side effects. **Exit codes:** `0` = progress/pending/done, `1` = a gate blocked an auto step (or the ticket is `ignored`).

## 9. Code Host Sync

`teatree.core.sync.sync_followup() ‚Üí SyncResult` is platform-agnostic. Per-overlay it resolves the overlay's `CodeHostBackend`, fetches open PRs authored by the current user (incremental via cached `updated_after`), upserts tickets by `issue_url` (or PR URL if no issue linked), enriches non-draft PRs with pipeline + approvals + review threads, infers ticket state from PR data (`infer_state_from_prs()` advances forward only, never regresses ‚Äî and a post-ship inferred state is routed through the shared DoD decision per ¬ß "Sync writers honor the same gate"), and detects merged PRs. It also re-checks the user's OWN open authored PRs for merge conflicts every sweep ‚Äî a non-incremental fetch (always `updated_after=None`, so a re-conflict that did not bump the PR's timestamp as `main` advances is still seen) whose conflicted PRs land in `SyncResult.conflicted_mrs` (the overlay-agnostic `ConflictedMR` shape: iid, repo, web_url, title). The GitLab backend reads `has_conflicts` / `merge_status == cannot_be_merged` / `detailed_merge_status == conflict`; `followup sync` surfaces them in a loud WARNING block ([#78](https://github.com/souliane/teatree/issues/78)). Detection only ‚Äî never an auto-rebase or auto-push.

| PR data | Inferred state |
|---|---|
| Draft | `started` |
| Non-draft | `shipped` |
| Non-draft + review-requested or approvals > 0 | `in_review` |

Review threads are classified `waiting_reviewer` / `needs_reply` / `addressed`. Draft notes (GitLab) / pending reviews (GitHub) surface as a statusline `review_draft` prompt to publish.

Posting discipline (#1207): `t3 review post-comment` defaults to creating a DRAFT and DMs the user the link; the colleague-visible `--live` path is gated on a single-use, MR-URL-scoped `LivePostApproval` minted by `t3 review approve-live-post <mr-url> --slack-ts <ts>` after the Slack DM at that timestamp is verified (from the user, recent within 15 min, contains an explicit approval phrase). The historical immediate-post default is retired; CLI enforces draft-by-default rather than relying on prose discipline.

One-step authorization (#126): `t3 review authorize <repo>!<mr> --approver <id>` collapses the former two-command on-behalf dance ‚Äî it records the durable `OnBehalfApproval` AND mints the matching single-use `LivePostApproval` in one call. The `post-comment --live` path consults one consolidated read-only decision (`review_authorize.resolve_live_authorization`); the two original commands remain for the Slack-ts verification path.

Published-note removal: `t3 review delete-discussion <repo> <mr> <note_id>` removes a published MR note; `t3 review delete-issue-note <repo> <issue_iid> <note_id>` is the issue/work-item twin (`DELETE /issues/{iid}/notes/{id}`). Both are colleague-visible on-behalf mutations routed through the recorded-approval gate (#960) ‚Äî the issue path scopes the approval to `<repo>#<issue>` (distinct from the MR `<repo>!<mr>` scope, via `issue_gate_target`) and verifies the delete took (#2081, `verify_issue_note_deleted`). These CLIs are the only sanctioned path: the `block-raw-review-post` hook (#1164) hard-denies a raw `glab api`/`gh api` DELETE to `‚Ä¶/issues/<iid>/notes/<id>`.

Verified-delivery notify wrapper ([#1181](https://github.com/souliane/teatree/issues/1181)): `teatree.messaging.notify_with_fallback` tries `notify_user` and, on a transport `FAILED`, falls back to a round-trip-verified backend send (`BotPing.transport` records the path). Loop/CLI sites route through the wrapper; `teatree.core` callers stay on `notify_user` (no `core ‚Üí messaging` edge). A `NOOP`/`FAILED` INFO `BotPing` is durably parked, and the always-on `undelivered_notify` scanner re-attempts parked rows each tick under their `idempotency_key`; the drain is bounded ([#2064](https://github.com/souliane/teatree/issues/2064)) ‚Äî a row past `MAX_REDELIVERY_ATTEMPTS` or 72h is `EXPIRED`, so the backlog never grinds and stale DMs never surface late.

Tabular DMs ([PR-18](https://github.com/souliane/teatree/issues/1777)): `teatree.backends.slack.table_format` is the pure formatter turning headers + rows + per-column alignment into a native Block Kit `table` block (`rich_text` cells, alignment via `column_settings`, capped 100 rows total incl. the header / 20 cols) plus a space-aligned monospace fence fallback (`slack_table_fence`; over-budget cells ellipsis-truncated widest-column-first, never wrapped, rows capped to match the block with an `‚Ä¶ and N more` trailer, empty ‚Üí `(no rows)`). `notify_user` takes an opaque `blocks` argument threaded to `post_message` so a bot‚Üíuser DM posts the block with the fence as the `text` degradation; the blocks stay opaque JSON so `teatree.core` never imports the Slack backend (no `core ‚Üí backends.slack` edge). The CLI twin is `teatree.core.table_output.print_table` (¬ß "Machine interface"), so both surfaces render tabular data through one seam each; the `slack-formatting` reference skill (an overlay companion) documents the formatter and the mrkdwn gotchas.

Review-shape audit (#1206): `t3 review run <MR_URL>` is the read-only entry point reviewer sub-agents call before scanning a diff. It fetches MR metadata, classifies complexity, counts existing-review state (open discussions + draft notes + approvals), and emits a structured JSON summary so every reviewer starts from the same shape rather than improvising. The command never publishes ‚Äî it stays outside the on-behalf surface. A merged/closed MR (live `state` field from `/changes`) short-circuits to a `skipped_merged` / `skipped_closed` verdict (#2081) ‚Äî a note can never land on one, so a mid-flight close aborts the review. GitHub PR URLs return `unsupported_forge` (exit 2) deterministically until a parallel GitHub backend lands.

Structured-evidence gate (#1280): `t3 review post-comment` and `post-draft-note` refuse a finding whose body matches an "X is missing/wrong/broken/stale" pattern unless an accompanying `FindingEvidence` record (`--evidence-json '{...}'`) carries verified receipts ‚Äî it passes only when `confidence='verified'` AND at least one verified-path field is non-empty (full schema in code). Implemented in `teatree.cli.review.evidence_gate`; runs alongside the on-behalf (#960), colleague-MR shape (#1114), comment-bloat (#2663), and TODO-anchor (#1186) sibling gates inside `ReviewService._run_pre_publish_gates`.

Comment-bloat gate (#2663): the same publishing path refuses a review note that drags in *project chatter* ‚Äî a stakeholder `@handle`, a Slack thread timestamp, or a tracker id (`#1234`/`!567`) paired with a coordinate-with-people directive (`ping the author`, `sync with the team`, `in standup`). A review comment is about the diff, not the tracker. The note-LENGTH dimension of bloat stays owned by the colleague-MR shape gate (#1114, 3-paragraph/200-word cap), so this gate adds no parallel length cap; a *bare* `tracked at #1234` non-blocker pointer (the TODO-gate remediation form) is NOT refused. `--allow-bloat` is the #126 per-call escape for a genuinely load-bearing reference. Implemented in `teatree.cli.review.bloat_gate`.

Close-trailer scanner (#1398): `ban_close_trailers_on_namespaces` lists fnmatch patterns over `namespace/repo`. DB-home (#1775); set via `t3 <overlay> config_setting set ban_close_trailers_on_namespaces`; the TOML value is ignored on read. When the target PR/MR's repo matches and the body carries a `Closes|Fixes|Resolves` trailer (`part of` and full-URL variants too), `ShipExecutor._build_pr_spec` silently strips those lines before opening the PR (`teatree.core.intake.close_trailer_scanner`). Distinct from the overlay-scoped `forbid_close_keywords` gate (#1012), which refuses the publish; this scanner cleans the body and proceeds.

Open-questions warn (#1933): any open question (solved or not) and any non-explicit assumption must be listed in BOTH the commit body AND the PR description under an `Open questions & assumptions` section (per-item status `decided-by-user` / `assumed` / `open`). `teatree.core.gates.open_questions_gate` runs at both PR-creation chokepoints (`ShipExecutor._build_pr_spec`, orphan-branch `create_or_defer_pr`): a body lacking the heading warns and proceeds. Warn-only per the "a gate without a reliable heuristic warns" rule ‚Äî the heading wording is not separable enough to block on. Canonical doctrine in `skills/ship/SKILL.md` ¬ß 5.

Required description sections (#312): the MR-description generator emits a standard body ‚Äî `## What`, `## Why`, plus any section an overlay declares mandatory via `OverlayMetadata.get_required_description_sections()` (e.g. `["Configuration"]`). `teatree.core.review.mr_metadata.ensure_standard_body` appends every missing section (with the overlay's `get_description_section_defaults()` body when supplied, so a thin commit still ships a meaningful default) at both PR-creation chokepoints (`ShipExecutor._build_pr_spec`, `ship_preview`); never duplicating a present section and preserving the first line. The gate side ‚Äî `OverlayMetadata.validate_pr` ‚Üí `validate_mr_metadata(required_sections=‚Ä¶)` ‚Äî flags a description missing a declared section, so an explicitly-authored body that bypasses the generator is still rejected. Core is overlay-agnostic: it renders only what the overlay declares, hard-coding no section names or wording.

Egress-leak gate family ‚Äî one doctrine, several entry points, each named here with its load-bearing invariant; detector lists, exit codes, thresholds, and exempt sets live in the code:

- **Public-repo diff privacy-scan** (#685, #730): pre-push hook `refuse-public-push-with-leak.sh` runs `t3 tool privacy-scan` over the pushed diff + commit messages when `origin` is PUBLIC, blocking emails / home paths / private IPs / keys / internal hostnames / banned terms. It fails CLOSED on a genuine finding (a dedicated exit code) but fails OPEN on any other non-zero so a scanner crash cannot wedge every push (#126).
- **Proactive public-repo identity gate** (#762, #785, #2655): the proactive sibling of the reactive backstop above. `teatree.core.public_identity.is_public_github_remote` drives the clone-local stamping of the canonical GitHub `users.noreply.github.com` author identity onto any worktree off a PUBLIC GitHub clone (the worktree provisioner and `workspace stamp-identity`), so a public-repo commit never carries the inherited real-domain email the leak backstop would later block. It is HOST-AWARE (#2655): the gate short-circuits any non-github host to `False` BEFORE the `gh repo view` visibility probe (`is_github_host`, recognising `github.com` and `github.com-<ssh-alias>`), so a non-github remote (e.g. a `gitlab.com` SSH URL) whose host-stripped `owner/repo` slug collides with a real public github.com repo is never mis-stamped with the github identity. Callers pass the full remote URL ‚Äî host intact ‚Äî never the bare slug.
- **Foreign-open-MR push guard** (#2211): pre-push hook `refuse-push-to-foreign-mr.sh` ‚Äî a collaboration-safety sibling (not a leak gate). For each pushed ref it resolves the branch's backing OPEN PR via `gh pr list --head <branch> --state open` and BLOCKS, naming the author + PR number, when the PR's author login != our `gh api user` login ‚Äî a teammate's open MR our changes would silently modify (our work belongs on our own branch; a worktree opened to INSPECT a colleague's MR is read-only). Our own open MR, a branch with no open MR, and a foreign CLOSED/merged MR (which `--state open` excludes) all pass; an explicit `[push-to-foreign-mr-ok: <reason>]` commit-message token allows a genuine co-authoring push. Fails OPEN on a `gh` failure / gh-unavailable / unresolved login so a transient forge-API error never bricks a legitimate push.
- **Banned-terms posting gate** (#1415, `teatree.hooks.banned_terms_scanner`): the `PreToolUse` non-commit sibling. It and the #1213 quote-scanner share `extract_bash_payload` and a per-segment override (#2031/#2034): the `--allow-banned-term`/`--quote-ok` flag token OR a leading `ALLOW_BANNED_TERM=1`/`QUOTE_OK=1` env-prefix clears only on the publish segment, so a chained-segment decoy cannot vouch. The deny message names the **env-prefix** form (`ALLOW_BANNED_TERM=1 <command>` / `QUOTE_OK=1 <command>`), not the `--‚Ä¶` flag: the flag is consumed by the gate's parser, never the posting command, so a `t3 review post-comment` subcommand would reject it as an unknown option ‚Äî the env prefix is the spelling that actually works on every command. A `> path <<EOF` heredoc is scanned only when its path is posted via a body-file flag; a stdin `--body-file -` heredoc always is. A `t3 review post-comment`/`post-draft-note` segment carries the published body in any of three sources ‚Äî the positional `NOTE`, `-m`/`--body`, or `--body-file <path>` (#32) ‚Äî all of which are scanned, while the `--file` argument (a diff ANCHOR, a SOURCE path, not a body-file) is deliberately skipped (`teatree.hooks._t3_review_post`). An inline `--body`/`-m`/positional-NOTE body whose `$(...)` command-substitution sits inside a SINGLE-quoted span is inert literal text bash passes verbatim, so `resolve_inline_body_value` reads the value token's source span (`raw`) and SCANS such a body rather than fail-closing on "publish body could not be read" ‚Äî a body bash WOULD expand (unquoted / double-quoted `$(...)`) still fails closed since its real content is unknown before the command runs (#1415) ‚Äî though for a LOCAL `git commit` that unreadable-body sentinel then DOWNGRADES to a warn regardless of landing-repo visibility (`command_targets_local_commit`: a commit is local and the pre-push gate, `refuse-public-push-with-leak.sh` #703, re-scans every commit message before a public push), while a `gh`/`glab` POST keeps hard-blocking; a `git commit -F -` reads its message from STDIN, so a heredoc / `printf|git commit -F -` writer body is RESOLVED and scanned (a real term in a readable commit body still blocks) rather than fail-closed as an unreadable file named `-`. The raw-aware `command_has_opaque_forge_transport` mirrors this so a non-forge segment that merely mentions a single-quoted `$(...)` is not treated as a hidden forge transport. The gate delegates matching to `check-banned-terms.sh`, whose exit-code contract is 0=clean / 1=banned-term-found / 2=scanner-could-not-run; when the shell fallback resolves to an old system `python3` (below the >= 3.13 floor) the matcher import crashes, so the fallback probe-imports the module first and exits 2 (fail CLOSED, loud) rather than letting the crash exit 1 (which collides with "found") become ALLOW. `scan_text` mirrors this: a crash, timeout, unexpected exit, or exit-1-with-no-report returns `SCANNER_UNAVAILABLE_MARKER` and BLOCKS ‚Äî a security gate that fails open on a crash is the bug class (#1954). **Company-identifier carve-out** (#1415 over-block): an optional `[teatree].banned_terms_allowlist` array names the org's OWN compound identifiers (`<org>-engineering`, `<org>-product`, internal-URL namespaces) that are never customer PII; `term_match._strip_allowlisted_tokens` blanks each allow-listed identifier's whole-token run BEFORE matching (mirroring the email carve-out), so a SHORTER single-token banned term (a bare org slug) can no longer surface inside a longer company-owned identifier or an internal-URL path ‚Äî while a genuine customer codename NOT on the allow-list is still blocked, and a bare standalone org-slug token still fires. The carve-out flows through the SINGLE shared matcher, so the shell hook (`banned_terms_cli`), the posting gate (`banned_terms_scanner`, incl. report attribution), and the diff-only commit scan all honour it identically; empty (default) is a no-op preserving prior behaviour.
- **Pre-dispatch quote-scanner** (#1401, `handle_dispatch_prompt_quote_scanner`): the dispatch-boundary companion to #1213 ‚Äî denies only a HIGH user-voice/PII match in an `Agent`/`Task` prompt, with a `[quote-ok: <reason>]` opt-out, so a verbatim quote cannot leak downstream.
- **Diff comment detectors** (added-lines-only): `code_comment_self_reference` (#1465) is a **blocking** `privacy_scan.py` diff detector (**fail-open per detector**, #1536) for bookkeeping self-references. `code_comment_density` (#1538) is the commit-side half of the comments-as-code rule (#1532; names + types are the docs, a long comment is a code smell ‚Üí refactor); it is **content-aware** ‚Äî beyond the line-count ratio / consecutive-run passes it flags a comment whose words merely **restate** the next code line and a docstring opening that merely **echoes the signature** (a single such line suffices), while leaving a genuine non-obvious-why comment and a justified multi-line docstring ALLOWED. It is **advisory** ‚Äî the standalone check (#1369, `t3 tool comment-density`), the pre-push hook, and the `comment-density-warning` CI job warn and **exit 0**, and it is NOT a `privacy_scan.py` blocking detector (#1844 ‚Äî no content-blind "overly long prose" heuristic that would flag legit long comments). A golden corpus (`tests/test_comment_density_gate.py`) + an eval pin must-DENY (restatement + signature-echo) symmetric with must-ALLOW (a docstring carrying a real why). The push-stage gates (`doc-update-gate`, `ensure-pr`, the leak gate) are `stages: [push]` ‚Äî skipped by a bare `prek run --all-files` but re-run by CI ‚Äî so `t3 tool verify-gates` runs both stages and returns the combined exit code, so local green == CI green.
- **Full-tree banned-brand backstop** (#1570/#1591, `core.banned_terms_tree`, CLI `t3 banned-terms scan-tree`, CI job `banned-terms-tree`): catches what the diff/payload gates structurally cannot ‚Äî a brand ALREADY committed, invisible to any post-landing diff ‚Äî by scanning every tracked file's **committed blob** (`git show HEAD:<path>`, working-tree fallback only for an uncommitted file) with the shared whole-token matcher (`teatree.hooks.term_match`, pinned by the parity test). The brand list (`[teatree].banned_brands`) is a curated high-confidence subset distinct from the egress `banned_terms`; unpopulated, the scan is INERT and says so loudly rather than returning a silent clean 0. The CI step passes `--require-brands`, upgrading that inert state to a hard MISCONFIGURED failure (exit 2, distinct from exit 1 = findings) so a missing secret reds the job loudly instead of a fake-green no-op. Local dev omits the flag.
- **Programmatic on-behalf egress privacy-scan** (#1295 capability J): the autonomous-events reply transport (`core.reply_transport`) wires the publication privacy gate (`core.gates.privacy_gate.scan_outbound_text`, the egress wrapper of the pure `scan_for_publication`) into its single on-behalf chokepoint ‚Äî `_BaseReplier._send` and `redeliver`, via `_enforce_privacy`, **scoped to code-host events** (`_FORGE_BY_SOURCE`: GitHub/GitLab only ‚Äî a Slack channel ref is not a repo target and is out of scope here). A GitHub PR comment / GitLab MR note posted under the user's identity is body-scanned for the overlay's `privacy_redact_terms`/`privacy_block_patterns` plus the built-in quote anchors BEFORE the wire call ‚Äî closing the gap the Bash-hook banned-terms/quote gates structurally miss (those scan raw `gh`/`glab` command text, not a programmatic API post). **Public-ness is derived from the SAME visibility axis the Bash gates use** (`hooks.publish_destination.is_public_destination` ‚Üí `_target_is_public`), NOT a per-overlay `public_repos` list (which is empty by default and would make the gate inert): a repo is scanned unless provably private (`private_repos` / `internal_publish_namespaces` allowlists or a cached `gh`/`glab` probe), so it actually protects a real public repo like `souliane/teatree`. It fails CLOSED on a finding (`PublicationPrivacyBlockedError` ‚Üí FAILED `ReplyDispatch`, never delivered; on `redeliver` it propagates to the retry sweep) and on a classification error (treats the target as public ‚Üí scanned); a provably-private repo is a clean pass and a bot‚Üíuser `post_dm` is never scanned. Overlay redact-rule resolution is a best-effort ADD on top of the always-on built-in quote anchors ‚Äî when no overlay resolves, the built-in detectors are the floor, so the gate never goes inert.

This guard is the prerequisite that lets a later slice move the secret lists themselves into the DB (closing the last TOML carve-out) without a shared export ever leaking them. The pre-push scanner alignment still holds: `scripts/privacy_scan.py` resolves via the shared `teatree.hooks.banned_terms_cli.resolve_banned_terms` (env override ‚Üí `[teatree].banned_terms` TOML, fail-CLOSED ‚Äî a present-but-unset/unreadable config reports the detector INERT on stderr rather than degrading to a silent empty ban list), and the banned-terms GATE flag (`banned_terms_gate_enabled`) is already DB-first via the cold reader (PR3).

**Feature flags vs. settings (T4-PR-1).** A *setting* is a durable knob the operator tunes and keeps; a *feature flag* is a TEMPORARY switch that gates in-flight code and is meant to DIE ‚Äî born with the code it guards, removed once that code is trusted (or abandoned). Both are `UserSettings` bool fields and resolve their VALUE identically through the untouched `env ‚Üí ConfigSetting(overlay) ‚Üí ConfigSetting(global) ‚Üí dataclass-default` chain; the code-level registry `teatree.config.feature_flags.FEATURE_FLAGS` only DISCRIMINATES a flag from a setting, GOVERNS its lifecycle, and AUDITS dead toggles ‚Äî it never becomes a second value tier (the value lives in the `ConfigSetting` store; the registry lives in code, because a flag is born and dies with the code it gates). Each `FeatureFlag` carries a `stage` (`FlagStage`: `DARK` ‚Üí `SETTLING` ‚Üí `REMOVE`), a `tracking_issue`, a `summary`, and an `off_value` (`False` for a positive-sense `*_enabled` flag, `True` for an inverted-sense `*_disabled` one). The **lifecycle** runs `DARK` (gated code in flight, ships OFF, enabled per-install for a trial) ‚Üí `SETTLING` (trusted, default-enable imminent, flag survives as an escape hatch) ‚Üí `REMOVE` (gated code permanent, the toggle is dead weight to delete). This is the typed-registry-plus-fitness-test idiom of `SETTING_HOMES` / `COLD_HOOK_SETTINGS`: the conformance suite `tests/config/test_feature_flags.py` keeps every entry honest ‚Äî each names a real `bool` `UserSettings` field registered in `OVERLAY_OVERRIDABLE_SETTINGS` (registration-drift closed like `cold_hook_settings`), carries a non-empty `tracking_issue` and a valid `stage`, and ‚Äî the load-bearing Goodhart guard ‚Äî a `DARK` flag's dataclass default MUST equal its `off_value`, so a dark feature can never ship default-ON without a code-reviewed stage demotion. Surfaces: `t3 config show` partitions flags OUT of the user-facing `intent` dump into a stage-labelled `flags` section (a temporary switch never reads as a knob to keep); `t3 <overlay> config_setting set|get <flag>` appends a `[feature flag, stage=‚Ä¶, tracking ‚Ä¶]` trailer; and `t3 <overlay> config_setting flags` is the read-only dead-toggle audit report that surfaces a `REMOVE`-stage flag LOUD. Seeded with `outer_loop_enabled` (DARK ‚Äî the OFF switch the T4 autoresearch outer loop ships behind; behaviorally inert, nothing reads it yet) and `teams_enabled` (DARK), among others. `loop_runner_enabled` was graduated OUT of the registry by [PR-28] once the worker owned the cadence by default ‚Äî it became a durable operational kill-switch, not a dying flag (the DELETE the lifecycle prescribes, not a `REMOVE` stage: the *setting* survives, only its flag-hood ends). The live registry is therefore all-`DARK`, so the stage-discrimination machinery (`dark_flags`, `render_flags_audit`) is proven non-vacuously over a mixed fixture. `dark_flags()` is the query hook a later self-catching critic uses to tie a dark flag to the done-means-merged status of the code it gates.

Local text-to-speech (#2060): `teatree.core.speak.deliver_user_dm()` is the one bot‚Üíuser-DM chokepoint, gated on macOS `say`, driven by the per-overlay `[teatree.speak]` sub-table. `local` controls speaker playback (`off`/`dm`/`all`) and the independent `slack` bool attaches audio to the DM; local playback is silenced at the playback call site when the resolved availability defers questions (`away` or `autonomous_away` ‚Äî the user is not at the keyboard, #2544), preserving `slack` and the configured `local` value (exception-safe). Playback is mutually exclusive via a bounded-wait cross-process `flock` so fan-out reads never overlap or backlog (#2152/#2156), and `clean_for_speech()` strips markdown/URLs/emoji and drops status/log lines so `say` reads prose, not "Info" preamble (#277). **Meeting-aware mute + question-mirror parity (#2171):** the same playback call site also silences local `say` while a configured presence backend (`[teatree.speak] presence_backend`, `teatree.core.presence` + `teatree.backends.msteams.presence` over MS Graph `/me/presence`) reports the user in a meeting ‚Äî Slack arm and configured `local` untouched, unknown/unreachable never suppresses; and the AskUserQuestion Slack mirror now carries audio to the phone too (the `hook_router` injects an enricher into the `slack_mirror` leaf that spawns `t3 speak-dm` detached after the text question lands). ## 11. Skills & Plugin Architecture

Skills live in `skills/*/` ‚Äî one `SKILL.md` + optional `references/` per skill. When installed as a plugin, skills are namespaced under `t3:` (e.g. `/t3:code`). The lifecycle skill set is `code`, `debug`, `test`, `review`, `review-request`, `ship`, `ticket`, `workspace`, `followup`, `handover`, `next`, `retro`, `contribute`, `setup`, `platforms`, `rules`. Read-only auxiliary skills sit alongside it: `checking` (what-did-I-miss) and `todos` (the session's task list, via `tasks list --session`, plus the in-session harness-TODO maintenance discipline ‚Äî the agent reconciles its live `TaskList`/`TaskUpdate`/`TaskCreate` list against the conversation each turn, aided by the read-only `tasks reconcile-checklist` emitter; teatree cannot write the live harness list from a hook/subprocess, so this maintenance is an in-session discipline, not a background job).

Skills declare dependencies via a single YAML frontmatter `requires:` field (transitive, topo-sorted, cycle-checked; a required skill with no `SKILL.md` ‚Äî an external methodology skill ‚Äî warns and passes through so the `Skill` tool still loads it). Third-party skill frameworks (e.g. superpowers) are absorbed into the `rules` skill rather than delegated, to avoid context duplication.

**Sub-agents (`agents/`).** Phase agents wrap skill bundles (`orchestrator`, `coder`, `tester`, `e2e`, `e2e-review`, `reviewer`, `review-request`, `shipper`, `debugger`, `followup`, `planner`, `answerer`, `scanning-news`) ‚Äî each a YAML+description wrapper referencing skills via `skills:` frontmatter, no content duplication. `e2e-review` ([#2298](https://github.com/souliane/teatree/issues/2298)) is the reviewer leg of the EXTERNAL review loop (dispatched for the `e2e_reviewing` phase), making the skill‚Üîagent pairing 1:1 for `e2e`/`e2e-review`. `review-request` (`(author, requesting_review) ‚Üí t3:review-request` in `SUBAGENT_BY_PHASE`) is the spawnable agent for the review-request phase ‚Äî read-only on the codebase, it discovers the ready PR and posts the request under the on-behalf policy. They are invoked by lifecycle skills, by the headless executor (¬ß5.2) on a claimed phase task, and by the loop tick (¬ß5.6) when a scanner signal calls for agent judgment. Interactive-only skills (no agent): `retro`, `next`, `contribute`, `setup`.

**Worktree-scoped dispatch resolution ([PR-12](https://github.com/souliane/teatree/issues/2564)).** A dispatched task runs in its OWN worktree, so the overlay and framework-skill detection root are resolved from the task's ticket ‚Äî its `overlay` field (via `get_overlay_for_ticket`) and `dispatch_worktree_path(ticket)` (the first materialised worktree whose recorded path still exists on disk) ‚Äî never the orchestrator's ambient cwd, which is the loop's clone rather than the ticket's checkout. `resolve_skill_bundle(..., worktree_path=‚Ä¶)` threads that path into `SkillLoadingPolicy.select_for_runtime_phase`'s detection `cwd` at every dispatch consumer (headless SDK run, `loop_dispatch pending-spawn`, the interactive `claude` command, and the teams maker pane); a missing/pre-provision path falls back to the ambient cwd. The same seam (`agents.dispatch_preflight`) injects a **target head-state** block into a coding/maker brief ‚Äî the worktree's HEAD commit (sha, subject, committed-at) classified against the task's triggering timestamp ‚Äî so the maker builds on any commit already on the branch instead of restarting; robust by construction (any missing worktree / non-git path / git failure yields no block, never a crash on the dispatch path). Every dispatched spawn carries a type-prefixed **display name** `t3-<type>-<id>` (`dispatch_gates.spawn_display_name`) the `pending-spawn` payload hands the `/loop` slot as the Agent tool `description`, so every spawn is attributable at a glance; an untyped task has no registered sub-agent and is already excluded upstream by the dispatchable claim filter. Long-running maker briefs auto-inject a heartbeat-DM cue; verification (review) briefs carry the anti-rubber-stamp block (reproduce the proof first, grade every quality dimension).

**Distribution.** Two install paths, one source of truth:

- **APM**: `apm install souliane/teatree`
- **CLI-first**: `git clone ‚Ä¶ && uv tool install --editable . && t3 setup` ‚Äî also registers the plugin in `~/.claude/plugins/installed_plugins.json` with `installPath` pointing at the main clone (no `~/.claude/plugins/t3` symlink; always live)

On every `t3 setup` run, `dep_drift` checks `[project].dependencies` against the editable install and reinstalls + `execv`-restarts if a declared dep is missing. The same run re-syncs runtime skill links and **prunes stale ones** ‚Äî a teatree-managed link whose skill was removed or renamed upstream is removed so the dropped skill stops resolving, while contribute-mode workspace links and a user's own real skill directories are left untouched. Because `t3 update` re-runs `t3 setup`, updating teatree auto-cleans skills dropped upstream.

**¬ß11.4 Bash Permissions.** The plugin's `settings.json` ships a **broad allow, narrow deny** `permissions` list ‚Äî every tool family the workflow touches is allowed, with load-bearing denies (push to default branches, `--force`, `--no-verify`, root `rm -rf`, `curl|bash`, `gh repo delete`) taking precedence; the `t3` CLI is the workflow's safety wrapper, so blocking inside the CLI is the wrong layer. Plugin config is **not self-modifiable** ‚Äî edits to the allow-list are rejected by Claude Code's autonomy guardrail, so a mid-workflow classifier denial means the agent stops and asks via `AskUserQuestion` (`skills/rules/SKILL.md` ¬ß "Classifier Denial Protocol"). `t3 doctor authorizations` is read-only ‚Äî it reports which generic recommended auto-mode authorizations are absent from `~/.claude/settings.json`; teatree ships **no** classifier whitelist of its own.

**¬ß11.5 Explicit skill loading.** Skill loading is fully explicit ‚Äî slash commands (`/t3:code`), phase mapping (`t3 agent --phase coding`), ticket status, the transitive `requires:` chain, and cwd/overlay context. There is no free-text keyword scan of the prompt or a task description: `SkillLoadingPolicy` resolves from those explicit signals only, and the `UserPromptSubmit` hook surfaces only the framework / overlay / companion skills a prompt's cwd context implies (`PreToolUse` blocks Python code edits until those load). Session-end retro fires from the `SessionEnd` hook keyed on the lifecycle skills loaded during the session; ticket-URL intake resolves through `t3 <overlay> workspace ticket <url>` (overlay-resolved, provider-agnostic), not a frontmatter URL pattern. The `SKILL.md` schema (`skill_support/schema.py`) rejects a stale `triggers:` / `search_hints:` / `companions:` key loud, so a skill file never silently carries a removed mechanism.

**¬ß11.6 Never Slim Skills.** A SKILL.md's load-bearing rules stay inline in the SKILL.md ‚Äî never extracted into `references/` to save tokens, because reference-file loads are unreliable at agent runtime (an agent does not reliably fetch a `references/*.md` on demand, so a rule moved there is silently ignored). Context reduction is achieved by **phase-scoped loading** (only the skills a phase needs are embedded) and **summary demotion** (a non-primary skill in a dispatched bundle gets a one-line "available ‚Äî load if needed" summary rather than its full body), never by shrinking an individual SKILL.md. A `references/` file is for genuinely optional deep-dive material only. This is the design rule the `/t3:rules` ¬ß "Never Slim Skills" prose enforces on authors.

**¬ß11.7 Configuration review cadence.** `CLAUDE.md`, `skills/*/SKILL.md`, and the hook/permission config (`hooks/hooks.json`, plugin `settings.json`) get a deliberate, deletion-first review every 3‚Äì6 months ‚Äî the accumulation of gates, rules, and skill prose is pruned back to what still earns its place, rather than only ever growing. The dreaming pipeline (`/t3:dreaming`, `t3 dream run`) is the executor: its memory-consolidation + core-gap pass is the standing cadence that surfaces stale config and drives each removal to a merged change.

**>90% branch coverage, non-negotiable** (`fail_under = 93, branch = true`). Omits only migrations.

- `conftest.py` monkeypatches `HOME`/`XDG_*` to `tmp_path`, strips `GIT_*`, isolates overlay env, resets backend + overlay caches between tests

- Core has no Playwright suite (no UI). Overlays declare their own via `get_e2e_config()`; `t3 <overlay> e2e {run,external,project}` runs them. The `external` runner resolves its Playwright working dir in order ([#2620](https://github.com/souliane/teatree/issues/2620)): an explicit `--repo` (a `[e2e_repos.<name>]` clone) wins; else, when `get_e2e_config()` supplies BOTH a `url` and a `ref` (`overlay_e2e_repo` lifts them into an `E2ERepo`, `e2e_dir` default `e2e`), the overlay's own repo is cloned at that ref (a `--branch`/`--ref` override wins) ‚Äî so a migrated suite on an unmerged branch runs without a separate `T3_PRIVATE_TESTS` checkout; else the legacy `T3_PRIVATE_TESTS` directory. A green/red `e2e run` records run provenance on the durable recipe (`Ticket.extra['e2e_recipe'].last_run`, `core/e2e_workitem.record_run`): result + per-repo SHAs + env, plus (for vanilla-spec suites whose specs live in another repo, [#272](https://github.com/souliane/teatree/issues/272)) the exact `spec_path` and the overlay-resolved `manifest_entry` id (its CI lane) via the `get_e2e_run_provenance(spec_path)` overlay hook ‚Äî so a run is reproducible from the DB alone. Core stays overlay-agnostic (default hook returns `""`; empty provenance fields are omitted). For a multi-config Playwright suite (one config per lane ‚Äî api-flow vs contrib vs portal), the overlay maps a spec to its `-c <config>` (and any other per-spec Playwright flags) via the `get_e2e_playwright_args(spec_path)` overlay hook; the `external` runner threads those into the `npx playwright test` command ahead of any explicit `--playwright-args` (default `[]` ‚Äî no flags, prior command shape preserved). The per-feature **acceptance scenarios** a templated test plan renders from come through the sibling `get_e2e_scenarios(spec_path)` overlay hook (default `()`): each element is an overlay-defined frozen `Scenario` (`surface`/`title`/`preconditions`/`steps`/`expected`/`modality` `ui|api`/`captures` of `Capture{slot,caption}`), so the overlay owns the scenario manifest and core just threads the tuple to the renderer. `t3 <overlay> e2e post-test-plan --manifest` ([#1409](https://github.com/souliane/teatree/issues/1409)) maintains ONE structured test-plan note per **ticket** (never the MR), idempotent on a hidden marker carrying a JSON state blob (the merge SOT); each run merges its manifest's env(s) over the prior state, freezing the side it omits. The note renders a test plan ‚Äî a multi-repo header with clickable commit-provenance and a `Dev ¬± Local` reconciliation, then bodies in one of four templates ([#2304](https://github.com/souliane/teatree/issues/2304)): the three workflow-shaped ones (`capture-matrix` default side-by-side table, `browser-click-first` screenshots, `link-api` links + code) and `scenario-plan`, which renders ordered acceptance scenarios ‚Äî each a `### Scenario N ‚Äî <surface>` block with Preconditions / numbered Steps / Expected / Actual (a ‚úÖ pass marker or a blocked-or-omitted note) and either captioned inline screenshots (`ui` modality) or a contract-check marker (`api` modality) ‚Äî separated by `---` rules with an `**Environment:**` footer. `--body-file` posts a pre-authored body verbatim (mutually exclusive with `--manifest`). Pure render/merge lives in `_test_plan_render`; the `scenario-plan` shape in `_test_plan_scenario`; the two non-default workflow renderers in `_test_plan_workflow_templates`; host orchestration in `_test_plan`.
  - **Upload integrity.** Artifacts embed the relative `/uploads/<secret>/<file>` GitLab claims on save and upload to the project that owns the note's issue URL; a post-upload `verify_upload` gate re-fetches each via the token-authed API and magic-byte-checks it, so "posted" means "verified renderable" (`TestPlanMediaError` refuses otherwise).
  - **Image preflight** (`core/test_plan_validation`, before any upload): every image must carry a `highlightAndShoot` red box, none may be byte-identical (md5 dedup), and a manifest with no media is refused (`TestPlanValidationError`); `--skip-validation` is the user-only bypass.
  - **Video preflight** (`core/video_evidence`, the video counterpart to the image preflight): an evidence video is refused when its opening frames are blank (the dead-pre-roll defect that shipped a 40s-blank walkthrough) or the clip is blank throughout ‚Äî `scripts/analyze_video.py` yields the verdict and the `e2e post-test-plan` path plus the `/t3:e2e` / `/t3:e2e-review` skills reject on it. Pinned by the `e2e_review` eval scenario (HOLD on a blank-pre-roll video).
  - **Honesty gates.** A colleague-repo body gate (`core/test_plan_blocked_gate`) refuses a `--body-file` post whose free text signals incomplete verification when the issue URL matches `colleague_repo_url_pattern` (`BlockedTestPlanPostError`); the structured `--manifest` render is exempt because its `**Blocked:** <reason>` disclosure is the honest channel. Self-evidence on the user's own ticket is un-gated by default (`post_e2e_evidence` in `on_behalf_auto_actions`, #960). `t3 <overlay> e2e retract-evidence` withdraws the note. A sibling field-context evidence probe (`core/doc_evidence`, [#2296](https://github.com/souliane/teatree/issues/2296)) asserts a generated-doc term in its expected structured field/column, never a page-wide substring ‚Äî an available primitive, not yet a globally-wired gate.

## 14. Django Project Workflows

Reference DB architecture, the import fallback chain (`DjangoDbImportConfig` strategy + selective fake-migration retry + renumber-record reconcile + post-import steps), DSLR integration, the worktree provisioning workflow (`worktree provision`), the server startup workflow (`worktree start`), and the state reconciler (`t3 workspace doctor`) are implementation details ‚Äî they live in code under `teatree.core.runners`, `teatree.utils.db`, `teatree.utils.django_db`, `teatree.utils.django_db_reconcile`, `teatree.core.worktree.reconcile`, and the `lifecycle` / `worktree` / `workspace` management commands. The architectural contract:

- **One reference DB per overlay** (canonical control DB at `teatree.paths.CANONICAL_DB`; worktree-aware code auto-isolates onto a per-worktree DB under `teatree.paths.auto_isolated_worktrees_dir()` = `~/.local/share/teatree-worktrees/<slug>/`, the `<slug>` deterministic from the checkout path via `teatree.paths.isolated_slug`). `t3 workspace clean-all` reaps the DB-unreferenced orphan dirs under that root ‚Äî those whose slug matches no live `Worktree` row's checkout ‚Äî while keeping any dir holding a `.git` checkout (the #706/#835 data-loss guard)
- **Analyze-then-wipe done-worktree reaper** (`teatree.core.worktree.worktree_done`, surfaced by `workspace clean-all` / `clean-merged`) ‚Äî a worktree is auto-wiped ONLY when its ticket is FSM-done (`MERGED`/`DELIVERED`/`IGNORED`, or a forge squash-merge) AND every unpushed commit and uncommitted change is PROVEN content-redundant against `origin/main` (`teatree.core.worktree.branch_classification.branch_redundancy`: patch-id `git cherry` zero, synthetic-squash `commit-tree` tree-delta capture, or `branch --merged`). A forge merged-PR signal alone never authorizes deletion (post-merge work is kept). There is **no recovery snapshot** (#1770 removed): unproven work is kept, never destroyed. Two fail-safe pre-gates run ahead of done-detection ‚Äî a **liveness guard** (`cleanup_liveness`: live `Session` / active `Task` / process CWD / git `index.lock` / recent HEAD commit ‚áí skip-and-report) and an **ownership guard** (`cleanup_ownership`: data-driven via `colleague_repo_url_pattern` + `user_identity_aliases`, a colleague-authored branch on a shared product repo is excluded up front; a no-op for the owner's solo repos). Everything NOT auto-deleted is surfaced as a machine-readable **structured EMIT** record (`workspace emit` ‚Üí JSON array, schema in `cleanup_emit`: path/branch/kind/`unique_commit_shas`/`merged_with_post_merge_work`/`banned_terms_status`/liveness/`last_commit_date`/owner) for a separate judgment skill to route. The **salvage primitive** (`workspace salvage`, `cleanup_salvage`) captures an emitted item's banned-terms-clean unique content onto a fresh branch + PR, verifies the forge has it, then deletes the source ONLY after a verified landing ‚Äî a failed push/open/verify leaves the source intact
- **Configurable import strategy per overlay** (`get_db_import_strategy(worktree) ‚Üí DbImportStrategy | None`) ‚Äî overlays declare *what* to import; core runs the engine
- **Migration retry with selective faking** for known-stuck migrations, declared per overlay
- **Renumber-record reconcile** ([#1038](https://github.com/souliane/teatree/issues/1038)) ‚Äî when a snapshot DB carries OLD migration numbers and the base branch has since renumbered (a migration inserted earlier bumped later numbers), `django_db_reconcile` detects Django's `InconsistentMigrationHistory` and renames the stale `django_migrations` record to the new name so `check_consistent_history` passes and the forward migrate proceeds. It fires ONLY on a *provable* pure renumber (identical descriptive suffix, exactly one lower-numbered stale record, unapplied readable on-disk dependency); a genuine divergence is left untouched so real schema drift surfaces as a failure, never masked
- **Post-DB steps** run after import (password reset, fixtures, ‚Ä¶) ‚Äî declared per overlay
- **State reconciler** (`t3 workspace doctor`) reconciles DB ‚Üî disk ‚Üî docker drift on demand. Alongside the infra stores it surfaces the **work-tracking-truth** drift classes (SELFCATCH-1) ‚Äî `UnpushedWork` (commits absent from all remotes), `DoneButUnmerged` (a done-claiming ticket with no `MergeAudit` and a non-upstream branch), `DuplicateScope` (>1 worktree dir per issue) ‚Äî each built from the existing `commits_absent_from_all_remotes` / `branch_redundancy` / `find_foreign_issue_worktrees` probes with fail-closed semantics (an inconclusive probe is a finding, never a silent pass). The global `work_state` loop scanner runs the same probes each tick and emits `workstate.drift` into the action-needed statusline zone, so committed-but-unpushed / done-but-unmerged / duplicate-scope work surfaces autonomously instead of when a human asks. Read-only ‚Äî it surfaces drift; destructive remediation stays gated
- **Redis is a normal per-stack compose service** ‚Äî there is no teatree-owned shared Redis container or DB-index pool. An overlay whose app needs Redis runs it as an ordinary `docker-compose` service in the worktree's own stack (its own container, isolated network), reached by the app at the in-stack hostname. One-stack-at-a-time provisioning makes a shared pool pointless; the per-stack container IS the isolation. `worktree start` heals a `provisioned` worktree whose DB was never created (interrupted provision), and `ServiceLauncher` is single-flight per `(worktree, service)` so concurrent frontend rebuilds never race the output dir

## 17. The Self-Improving Factory Architecture

Teatree is a durable self-healing **and** self-improving development factory. This section is the lasting architectural reference ‚Äî the umbrella under [#836](https://github.com/souliane/teatree/issues/836); each component below is a separately tracked ticket implemented as deterministic teatree code, not skill prose.

Why this architecture exists, observed repeatedly: durability comes from **enforcement encoded in code/structure**, not prose that decays. A rule kept in memory/skills and relied on by vigilance recurs anyway; the same rule encoded as a gate/test/hook does not. The invariants below are the structural form of that lesson ‚Äî load-bearing, binding every change to teatree itself.

### 17.1 Invariants

1. **Two layers, never conflated.** *Self-healing* (independent review, draft-locks, recovery, gates) is the substrate. *Self-improvement* runs on top: each caught failure-class becomes the smallest enforcement artifact that makes the class structurally impossible. Self-improvement is **gated by** self-healing ‚Äî the system is never changed in a way the healing layer cannot catch or roll back.

2. **The flywheel.** A defect (from diff review, OR the code-health loop on un-changed code, OR an orchestrator-noticed near-miss) ‚Üí the orchestrator synthesises ‚Üí output is the *smallest enforcement* (gate/test/hook), never a prose rule ‚Üí the failure-class is extinct. A repeat failure whose only output is memory/prose is a flywheel failure.

3. **Topology.** The orchestrator is the synthesis brain (retro synthesis, code-health triage, enforcement escalation, merge/clear decisions). Sub-agents are sensors/hands emitting structured signal into durable state, never self-judging. Skills carry judgment/methodology; teatree code carries the deterministic loops, gates, and intake. Corollary: mechanics ‚Üí code, judgment ‚Üí skill.

4. **Blast-radius rule.** Substrate changes are draft-locked and need a recorded human sign-off, satisfied by EITHER a per-PR `MergeClear.human_authorizer` OR the overlay carrying a STANDING merge-approval grant ‚Äî `autonomy = full`, OR an explicit `require_human_approval_to_merge = false` on a non-collaborative tier ([#2666](https://github.com/souliane/teatree/issues/2666): `mode = auto` + `require_human_approval_to_merge = false` is the same owner statement as `autonomy = full`; the collaborative `notify` tier is excluded since it merges only after a colleague approval). Either standing grant lets the overlay merge substrate without a per-PR sign-off. Either way the agent runs the `t3 <overlay> ticket merge` keystone (¬ß17.4.3); the human never performs the merge. The carve-out removes ONLY the per-PR sign-off ‚Äî the floor (cold-review `reviewer != maker`, SHA-bind, CI-green, not-draft, never-lockout, privacy/leak scan) holds on every substrate merge; with the merge-approval gate ON on a below-full tier the `human_authorizer` stays mandatory.

5. **Durability discipline is load-bearing.** Durable task/state plus pre-compaction snapshots let the orchestrator brain survive compaction/restart; keep them.

6. **Enforcement over prose, as a standing audit.** Invariant 2 makes the flywheel's *output* a gate; this invariant makes the *standing posture* explicit. Every user behavioural directive MUST be (a) codified in teatree and (b) enforced by deterministic code/gates wherever mechanizable, so skill prose shrinks to the judgment that genuinely cannot be mechanized and skills get **lighter** over time. This is a recurring retro/review responsibility: the enforcement gate (¬ß17.6 / [#850](https://github.com/souliane/teatree/issues/850)) turns a mechanizable rule into a gate, and the recurring audit keeps reclassifying prose rules ‚Üí code gates.

7. **Consolidation over drift.** Behavior encoded outside the teatree framework ‚Äî personal `settings.json` hooks, dotfiles automation, overlay-local ad-hoc config, personal memory guardrails ‚Äî must be considered for promotion into teatree on every retro/review pass. Genuine per-instance variance must be modelled as a documented teatree setting or config knob; undocumented divergence silently drifts and violates invariant 2.

8. **All FSM state transitions go through the `t3` CLI.** The pre-condition and pre/post transition hooks are the coherence mechanism (ledger update, attestation-binding to the HEAD/workstream the phase was earned against, privacy/AI-signature scan, `mark_merged()`). Out-of-band state mutation ‚Äî raw `gh pr merge` / `glab mr merge`, or hand-editing the phase ledger / FSM state ‚Äî is prohibited and **mechanically guarded** (`hook_router._BLOCKED_COMMANDS`, the same hook layer as the draft-lock and structured-question gates). The keystone IN_REVIEW ‚Üí MERGED transition is ¬ß17.4; the non-bypassable gate placement is ¬ß17.6.3. The two sanctioned escapes for stale state ‚Äî `lifecycle clear-ledger` and the hardened `lifecycle visit-phase ‚Ä¶ reviewing --agent-id` ‚Äî are themselves `t3` commands, never manual edits.

9. **Every user-directed question is captured ‚Äî sync or durable ‚Äî and reaches Slack.** A user-directed question must either (a) call `AskUserQuestion` with the user reachable this turn, or (b) be recorded as a `DeferredQuestion` row when the resolved availability mode defers questions (`away` or `autonomous_away`). Mode resolution is a single deterministic precedence (exposed by `t3 availability`): unexpired manual override ‚Üí live presence (a recent `UserPromptSubmit`, upgrade-only) ‚Üí `[teatree.availability]` cron-window ‚Üí `present` (default). There are three availability modes ([#2544](https://github.com/souliane/teatree/issues/2544)): `present` (interactive, self-pump runs), `away` (holiday ‚Äî questions defer AND the Stop self-pump pauses), and `autonomous_away` (unattended ‚Äî questions defer like `away` yet the self-pump keeps the factory running like `present`). The two behaviours `away` conflated are split by `Resolution.defers_questions` (away + autonomous_away) and `Resolution.pauses_self_pump` (away only), so a long unattended run no longer has to choose between deferring questions and stopping the loop. The away path is a *sanctioned destination* for the same `AskUserQuestion` call (converted at the `PreToolUse` layer, never bypassing the ¬ß807 structured-question gate); since the user reads Slack, every question is mirrored to their DM (away mirrors before denying) and away‚Üípresent auto-drains the backlog. A live user turn (a same-session `UserPromptSubmit` within a 90 s freshness window) renders in-client even under manual-away so `/checking` works without an availability flip, and present/away capture unify on one generation-stamped, mirror-linked `DeferredQuestion` whose *answer* is delivered back ‚Äî a Slack reply binds the live generation and the next prompt injects it once. The presence/window-slide mechanics live in `docs/blueprint/loop-topology.md`. Component: ¬ß17.3 C3.

10. **Orchestrator never executes work directly ‚Äî every implementation, review, test, debug, and ship action is dispatched to a sub-agent.** The orchestrator's role is synthesis, classification, dispatch, and CLEAR issuance (invariant 3, ¬ß17.4.1, ¬ß17.8 clause 3); the *hands* are sub-agents (`t3:code`, `t3:review`, `t3:test`, `t3:debug`, `t3:shipper`, `/teatree-batch`'s delivery sub-agent) and the durable loop. Inlining implementation work ‚Äî even a "trivial" typo Edit, a Bash run redoing a sub-agent's job, a local test cycle dodging `t3:test` ‚Äî conflates judgment with execution, denies maker‚âÝchecker independence, and concentrates the compaction risk the topology spreads across durable handoffs. **Narrow exceptions:** read-only orientation in the orchestrator's own session (`Read`/`Grep`, `gh pr view`, `git status`, `AskUserQuestion`, sanctioned messaging), the `t3 ‚Ä¶` invocations it owns (`MergeClear`, attestation, next-dispatch), and conversational replies with no repo mutation. Mechanized as gate 2 in ¬ß17.6.4 (`handle_enforce_orchestrator_boundary`): a `PreToolUse` deny on a main-agent LONG/HEAVY foreground `Bash` command, with `run_in_background: true` the escape hatch and a payload `agent_id` distinguishing sub-agent from main ([#115](https://github.com/souliane/teatree/issues/115)). It shares one `_fail_open_or_deny` chokepoint (with always-available `t3 <overlay> gate ‚Ä¶ disable` kill-switches and the `self_rescue` allowlist) with the sibling over-deny gates; the PUBLIC-egress leak gate is excluded and stays fail-CLOSED. Detail in ¬ß17.6.4.

11. **Any interactive Claude Code session that mounts this teatree install MAY drain the `PendingChatInjection` queue.** The inbound-Slack bridge (#1014) records each user DM as a `PendingChatInjection` row; the `handle_inject_pending_chat` `UserPromptSubmit` hook drains unconsumed rows into the next prompt's `additionalContext`. Drain eligibility is **decoupled from loop ownership**: the autonomous `t3 loop start` session holding `_OWNER_LOOP` never receives `UserPromptSubmit` events, so a t3-master gate here would block *every* user reply from reaching *any* interactive session. At-most-once delivery rides primitives orthogonal to ownership ‚Äî the single-use `PendingChatInjection.consume()` (`UPDATE ‚Ä¶ WHERE consumed_at IS NULL`) and the `(overlay, slack_ts)` `UniqueConstraint` ‚Äî so the scanner can over-poll safely. The t3-master gate stays on the ¬ß5.6 self-pump (singleton), not on inbound message drains.

12. **An outcome-claiming completion carries a resolvable artifact pointer, fail-closed.** The out-of-band completion surface (`tasks complete --note`) records externally-landed work. `teatree.core.review.completion_evidence` makes two separate judgments: a note *asserts an outcome* when an `OUTCOME_CLAIM_KINDS` verb (`merged`/`posted`/`shipped`/`deployed` + synonyms) co-occurs with a context cue (internal idioms like `merge conflict` are stripped first), and an asserting note must carry a *resolvable pointer* an auditor can follow ‚Äî a URL, a cued git SHA, an MR/PR/issue ref, a note id, a Slack ts, or a real/dotted-module path ‚Äî else `check_completion_evidence` refuses (`CompletionEvidenceError`). A completion with no note, or asserting no outcome, is never gated (the structural form of invariant 6's "done claims require evidence"); the exact matchers live in `completion_evidence.py`. A `failed ‚Üí completed` resolution requires a mandatory `--note` ([#1949](https://github.com/souliane/teatree/issues/1949)); `claimed ‚Üí completed` stays note-optional.

13. **Post-write side effects are confirmed by re-reading, not assumed from the write's own response** ([#1192](https://github.com/souliane/teatree/issues/1192)). A write's own success response (`ok: true`, a harness tool call returning without error) is not proof its effect is actually visible ‚Äî eventual consistency, a stale replica, or a race can make a claimed write unobservable moments later. `teatree.core.verify_by_reread.verify_by_reread` is the small core contract: given a `reread` callable that independently re-observes the target system, it returns a `RereadOutcome` (`confirmed` / `not_confirmed(reason)`) and never raises ‚Äî a verification failure is a signal to log or retry, never a reason to crash the write path. Call sites: Slack `reactions.add` (`teatree.backends.slack.reactions.add_reaction_verified`, wired into `add_reactions_for_transition` / `add_approval_reaction`) re-reads via `reactions.get` before trusting the write; and `create_pr` ([#1194](https://github.com/souliane/teatree/issues/1194), `teatree.core.merge.pr_create_verify.verify_pr_exists`) re-reads a just-created PR's open-state on the ship / `ensure-pr` path before recording the URL, so a phantom PR (a 404 re-read) is reported failed and never advances the FSM. Inbound-event ingestion applies the sibling shape ‚Äî a failed drain records the error and retries with backoff, then dead-letters (`IncomingEvent.record_failure`, [#673](https://github.com/souliane/teatree/issues/673)) so a poison event never silently drops nor blocks the queue. Direct MCP Slack writes are refused at the `PreToolUse` boundary (`handle_block_mcp_slack_write`, [#1196](https://github.com/souliane/teatree/issues/1196), kill-switch `[teatree] mcp_slack_write_gate_enabled`) so every Slack write routes through the `t3` CLI egress chokepoint under this contract.

**Admin-dashboard HTML snapshot ([#12](https://github.com/souliane/teatree/issues/12)).** `src/teatree/core/dashboard_snapshot.py` renders the teatree Django admin index ‚Äî frozen to a dedicated `AdminSite` carrying only `teatree.core`'s registered model admins, with output-affecting settings (`ROOT_URLCONF`, `STATIC_URL`, `LANGUAGE_CODE`, `TIME_ZONE`, `DEBUG`) pinned and the CSRF token stripped ‚Äî to a byte-stable HTML string via Django's test client; `scripts/hooks/generate_dashboard_snapshot.py` stands up a throwaway sqlite test DB and writes the canonical [docs/generated/dashboard/admin-index.html](docs/generated/dashboard/admin-index.html). The `docs-drift` job regenerates it (NO_STAGE) before the same `git diff --exit-code docs/generated` gate the diagrams use, and `tests/teatree_core/test_dashboard_snapshot.py` is the loud in-suite gate (it owns a DB, so it plays the `check_fsm_diagrams_sync.py` role) ‚Äî so the dashboard "screenshot" can never drift from the registered models.

**Representative CLI-output snapshot ([#12](https://github.com/souliane/teatree/issues/12)).** `src/teatree/cli/cli_output_snapshot.py` renders the front-door `t3` command output (`t3 --help`, `t3 loop --help`) to byte-stable markdown via `teatree.cli.command_tree.render_help_blocks` ‚Äî the same #2599 determinism seam (pinned width, no env-derived sizing, home-rooted defaults folded to `~`, overlay surface pinned to `t3-teatree`) the CLI reference uses, so no timestamp/duration/PID/path can leak; `scripts/hooks/generate_cli_output_snapshot.py` writes the canonical [docs/generated/cli/representative-output.md](docs/generated/cli/representative-output.md). The `docs-drift` job regenerates it (NO_STAGE) before the same `git diff --exit-code docs/generated` gate, and `tests/teatree_cli/test_cli_output_snapshot.py` is the loud in-suite gate ‚Äî the curated, always-fresh complement to the exhaustive CLI reference (#67).

**Scoped mutation testing ([#131](https://github.com/souliane/teatree/issues/131)).** `t3 mutation run` runs mutmut over only diff-touched modules in the NARROW `[tool.teatree.mutation]` safety registry; `tests/quality/test_mutation*.py` carry the ledger + kill-proof. `BaselineRatchet` ([#1926](https://github.com/souliane/teatree/issues/1926)) makes the recorded `baseline_surviving` count a ratchet ‚Äî a run above it fails in BOTH `warn` and `block` mode (count may only shrink), the prerequisite to flipping `mode` to `block`.

**Test-path-mirror forward-guard.** `t3 tool test-path-mirror` enforces *tests mirror production code*: a test file's first-party imports (parsed by AST, never executed) must place it under the mirror dir `tests/teatree_<pkg>/...` of at least one imported module; a top-level src *module* maps to the `tests` root as an EXACT-only expectation. The verdict is a per-path **ledger**, not a scalar count: `[tool.teatree.test_path_mirror] baseline_file` points at a committed sorted grandfathered-path list (`tests/test_path_mirror_grandfathered.txt`). The gate is RED when any LIVE violation is NOT in the ledger (a NEW mis-pathed file, named) AND RED when any ledger entry no longer violates (forced banking ‚Äî closing the old count-baseline's headroom hole where unbanked reductions let new mis-pathed files ride in green). `--update-baseline` rewrites the ledger to the exact live set but refuses to ADD an entry without `--allow-regression`. A path list merges as a git **set-union**, so two disjoint PRs never collide ‚Äî the semantic non-mergeability of a single integer (git cannot union two +1s) is gone. A small reviewed exemption set (`tests/integration/`, `tests/e2e*`, `conftest.py`, a `# test-path: cross-cutting` pragma, ‚Ä¶) keeps legitimate multi-package tests out of the count.

**Merge-safe ratchet ledgers + CI parity.** The two intra-package deferred-import ratchets follow the same per-item ledger shape: `tests/quality/deferred_import_pegs.toml` (`[intra_core]` / `[intra_loop]`) pegs each source file's function-scoped `teatree.core` / `teatree.loop` imports (unlisted files peg at 0), counted by the shared `tests/quality/_deferred_imports.py` walker. Over-peg blocks (names the file), under-peg banks (lower the entry); disjoint per-file bumps merge as set-union, same-file contention surfaces as a git textual conflict. The local/CI predicate gap is closed two ways without violating the no-full-suite-on-push invariant (`tests/test_no_full_suite_on_pre_push.py`): a push-stage `ci-critical-parity` prek hook runs the PATH-SCOPED fast lane (`tests/quality` + the never-lockout contract + `--doctest-modules src/teatree`), and `dev/ci-parity.sh` chains the EXACT full CI predicate (prek all-files, `makemigrations --check`, `t3 tool test-path-mirror`, `dev/test-cov.sh`, `t3 ci coverage`) as the one documented pre-push command for src-touching PRs ‚Äî opt-in by workflow, never a push hook, because the 93% whole-tree coverage floor is a whole-tree property no diff-scoped subset can prove.

**Behavioral eval harness ([#1160](https://github.com/souliane/teatree/issues/1160)).** `src/teatree/eval/` grades agent behaviour across free deterministic lanes (real-code-path regression corpus, skill-coverage, negative-control, transcript-replay, corpus-grade, skill-command-validity) and a metered AI lane (in-process `claude-agent-sdk`, weekly standalone `eval.yml`, authed on the credential the `eval_credential` knob selects ‚Äî DEFAULT the subscription OAuth token, reversing [#2707](https://github.com/souliane/teatree/issues/2707), with the metered `ANTHROPIC_API_KEY` still selectable via config). Bare `t3 eval` runs the whole suite in one table; subcommands target one lane. The architecture:

- **#550 3-tier skill model.** Tier-1 `skill-command-validity` (free) fails on a backticked `t3 ‚Ä¶` in a SKILL.md that no longer resolves against the live CLI registry; Tier-2 is the metered behavioral scenarios; Tier-3 `skill-prose-judge` is **advisory** (scores SKILL.md prose, never fails the suite).
- **Under-load behavioural-drift lane ([#2346](https://github.com/souliane/teatree/issues/2346)).** `EvalSpec.lane` selects the harness mode: `clean_room` (default) loads one skill into an empty context; `under_load` reproduces real-session instruction drift ‚Äî the skill bundle as system prompt plus an 8k‚Äì20k-token polluted `context_preamble` folded into the user prompt. The bundle is capped at a deterministic char budget (the scenario's own `agent_path` skill, the `rules` skill, and every small canonical-source skill always kept; only the largest peripheral skills shed) so a full bundle + a realistic preamble + a multi-tool scenario fits the model's input window instead of overflowing with "Prompt too long" before the model runs. A `--lane` filter slices the one catalog so the cheap PR-path anti-vacuity gate and the weekly metered lane run the right subset. There is **no known-red allowance and no shrink-only ratchet**: every `under_load` scenario must be GREEN, and a failing one reds the run **outright**, exactly like a failing `clean_room` scenario (`t3 eval run --trials k` exits non-zero on any red scenario, in either lane). The behavioural fix lever is the **skill prose** ‚Äî a drifting rule must be made unmissable in the loaded `skills/*/SKILL.md` the lane sends the model, never added to a tolerated-red list. Each scenario's matchers are proven anti-vacuous on every PR by `tests/eval_replay/test_scenarios_anti_vacuous.py` (its `_fail` fixture grades RED, `_pass` GREEN, a `_noop` transcript cannot satisfy it) before the metered lane (weekly `.gitlab-ci.yml` run + on-demand `eval.yml` workflow, off the per-PR path) ever runs.
- **Model resolution by abstract tier ([eval redesign]).** Eval scenarios reference models by abstract TIER, not a concrete id: `EvalSpec.tier` (`frontier`/`balanced`/`cheap`) or `EvalSpec.phase` (an FSM phase name) resolve through the single `agents/model_tiering.TIER_MODELS` constant (`eval/model_resolution.resolve_eval_model`). The resolution precedence is `model:` (an explicit concrete-id escape hatch, `model@effort` preserved) > `tier` ‚Üí `resolve_tier` > `phase` ‚Üí `DEFAULT_PHASE_MODELS[phase]` ‚Üí `resolve_tier` > `DEFAULT_TIER` (`balanced`). Adopting a new model is one edit to `TIER_MODELS` (or one `[agent.tier_models]` line) ‚Äî every scenario and test follows, no edit. NO concrete model-id string literal appears in any scenario or test.
- **Selective-PR eval + weekly benchmark publish ([#2704](https://github.com/souliane/teatree/issues/2704)).** A PR runs the metered suite ONLY for the scenarios its diff touched: `scripts/eval/scenarios_for_changed.py` maps changed paths (STDIN) to the names of every spec whose `source_path` matches, and the PR workflow loops `t3 eval run <name>` per changed scenario (the positional NAME filter; the multi-name `--only` flag is removed). `.github/workflows/eval-pr.yml` (`on: pull_request`) detects-then-runs the changed scenarios at the DEFAULT per-tier single-trial lane. Because that lane runs the selection SEQUENTIALLY in ONE job, the selector caps it at `MAX_SELECTIVE_PR_SCENARIOS` (= `lane_shard.MAX_SCENARIOS_PER_SHARD`): a corpus-wide mechanical edit (a `model:`‚Üí`tier:` backfill, a mass rename) that touches every scenario file is truncated to a deterministic sorted-name prefix so it stays inside the 80-min step cap ‚Äî full-catalog coverage of such a change is the weekly SHARDED lane's job, not the PR lane's. When the cap bites, `changed_scenarios.ScenarioSelection.truncation_note()` emits a `selected N ‚Ä¶ capped to K ‚Ä¶ deferred M to the weekly sharded lane` line to stderr from both `t3 eval changed-scenarios` and `scripts/eval/scenarios_for_changed.py` ([#2737](https://github.com/souliane/teatree/issues/2737)), so a corpus-wide PR's truncated coverage is visible in the CI log instead of silently dropped. Every metered run (PR + weekly) can emit two artifacts from THIS run's results: the PRIVATE per-trial `--transcript-html` (reasoning + tool calls), and the PUBLISH-SAFE `--summary-md` (`report.render_summary_markdown` ‚Äî total counts + total cost + model + a `scenario | lane | verdict | trials | cost` table built ONLY from spec identity + verdict + pass/trial counts + per-scenario `cost_usd`, NEVER a transcript; an anti-disclosure test pins a transcript sentinel absent). The weekly `eval.yml` uploads each shard's sanitized summary (`eval-summary-*`), and a `publish` job merges them (`scripts/eval/merge_summaries.py` ‚Äî summing per-scenario cost into a dashboard total) into `docs/evals/index.md`, pushing to `main` so `docs.yml` serves the dashboard on Pages. A red weekly run is surfaced by GitHub's native failed-run email (no extra secret). The single-trial execution shape lives in `cli/eval/single_trial.py`, the multi-trial/matrix shapes in `cli/eval/multi_trial.py`.
- **Reusable eval-CI primitives + `workflow_call` workflows.** The selection / scheduled-guard / dashboard-merge logic is extracted into pure modules (`teatree.eval.{changed_scenarios,merged_prs,summaries}`) exposed both as reusable CLI subcommands (`t3 eval changed-scenarios` selects a PR's scenarios from a STDIN diff; `t3 eval merged-prs-since` is the weekly no-PR-merged cron guard; `t3 eval merge-summaries` builds the public dashboard) and consumed by the host's `scripts/eval/*.py` (now thin shims over the same core). The two reusable workflows `.github/workflows/{eval-pr-reusable,eval-weekly-reusable}.yml` (`on: workflow_call`) own the selective-PR and weekly metered flows once ‚Äî with overlay host-checkout + scenario-assertion inputs and the no-silent-green enforcement ‚Äî so a downstream overlay's eval CI becomes a thin caller of teatree's primitives instead of a duplicate. The anti-leak invariant is unchanged: the summary merger and the publish job read only the sanitized verdict rows, never transcripts.
- **`--benchmark` (the 3-tier matrix) + `--model` (force one model).** `t3 eval run --benchmark` runs every (filtered) scenario against ALL THREE tier models (`frontier`/`balanced`/`cheap`, each resolved through the single `TIER_MODELS` constant via `app_helpers.benchmark_models`) on the metered matrix lane and renders the comparison matrix + a self-contained HTML dashboard (`eval/matrix.render_matrix_html`) ‚Äî the canonical weekly CI benchmark entry (`eval-weekly-reusable.yml`); adopting a new model needs no flag edit. `t3 eval run --model <m>` forces the whole suite onto one model (single-trial metered). `--benchmark`/`--model`/`--models` are mutually exclusive. The DEFAULT `t3 eval run` is per-scenario tier/phase, one trial, no pass@k, no escalation. **Escalate-on-fail (`t3 eval run --escalate-on-fail --escalate-trials N`, default N=3).** Still available off the default path for the PR lane: a single noisy LLM trial is not yet a hard red ‚Äî `teatree.cli.eval.escalate.escalate_failures` re-runs ONLY the failed scenarios at N higher trials and classifies each `flaky` (recovered) vs `confirmed` (the lane reds). Single-trial only. **Effort matrix (`scripts/eval/lane_matrix.py --efforts low,medium,high`).** An optional axis that fans each `{lane, shard}` leg across the three reasoning-effort tiers, carried through `t3 eval run --effort <tier>`. The `t3 eval run` lane dispatch (matrix / `--model` force / pass@k / single-trial) lives in `cli/eval/run_dispatch.py`.
- **Ground-truth corpus + audit layer ([#2192](https://github.com/souliane/teatree/issues/2192), [#1861](https://github.com/souliane/teatree/issues/1861)).** `eval/corpus/` pairs captured sessions with independently-authored labels under an independent-oracle guard (a matcher label whose author is the rule author is refused). The `SessionAuditRecord` ledger (parallel to `EvalScenarioResult`, ids/slugs only) is populated by the conversation-audit pass and rendered by a confusion-matrix renderer; the corpus/audit curation CLI (`t3 eval corpus|audit|label`) is the thin surface over these engines.
- **Cost + benchmark.** A per-scenario cost ledger (`EvalScenarioResult.cost_usd`) powers two cost gates and `t3 eval benchmark`. `--gate-cost-regression` is *relative* drift vs a mutable DB baseline run; `--gate-cost-bounds` is the *absolute* ceiling ‚Äî each scenario's metered cost is checked against the CHECKED-IN `evals/cost_bounds.yaml` (calibrated `bound_usd` x `(1 + margin)`), a scenario over its ceiling or a configured-but-uncosted scenario fails loud (never skip-as-pass), and the engine is the pure `teatree.eval.cost_bounds`. Both gates read the durable ledger, so they run on the persisted weekly/manual metered path, off the per-PR path. `t3 eval benchmark` runs the suite per `model@effort` variant and reports pass-rate, cost-per-pass, and honest cache-aware token metrics (billed cost is the headline).

A skill's evals live in the central catalog `evals/scenarios/<skill>.yaml` (each spec carrying an explicit `agent_path: skills/<name>/SKILL.md`), never inline in the `skills/` tree (`tests/eval_replay/test_no_inline_skill_evals.py` enforces it); a skill with no eval declares `eval_exempt: <reason>`; `t3 eval coverage` is the warn-first per-skill coverage gate. Every run ends with a plain-language closing verdict, and a setup-skipped lane reads as `NOT RUN`, never as validated. [evals/README.md](evals/README.md) is the SOT for all eval architecture detail.
