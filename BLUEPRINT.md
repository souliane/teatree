# TeaTree Blueprint

The product spec. Code is an artifact; this file is the product.

If the entire `src/` and `tests/` tree were deleted, this document — plus the three architectural appendices linked below and the skills in `skills/` — should be enough to regenerate the project without ambiguity.

**Tone.** This blueprint is functional and architectural. It names classes and files where the structure carries the design, but it is deliberately language-agnostic — a competent engineer should be able to reimplement teatree in another language with this file alone. Prose-of-code (paragraph descriptions of method bodies, ticket-history rationale, line-by-line walkthroughs) does not belong here. The code itself is the source of truth for implementation; this file is the source of truth for architecture.

**Change policy.** Every code change to teatree must keep this file consistent with the architecture. Implementation details — new flags, new error messages, why a particular regex was tightened — belong in docstrings, commit messages, or the issue tracker, not here. Before modifying this file (or any of the three appendices) please ask the user for approval — this is the source of truth and the user validates every change.

**Status:** current architecture under [#541](https://github.com/souliane/teatree/issues/541). All phases (0–8) shipped.

- Statusline file is the only persistent UI surface (no HTML dashboard, no daemon, no ASGI server)
- Code-host + messaging Protocols unified; backends are selectable per overlay
- Fat loop + scanners + dispatcher; the headless executor runs in-process via `claude-agent-sdk`
- No-overlay-leak grep gate keeps the platform tenant-agnostic

The §17.1 Invariants list is parsed by `scripts/hooks/check_blueprint_invariant_numbering.py` (gapless 1..N — do not renumber, do not delete). Its CI cross-PR mode (`blueprint-cross-pr` job) is **fail-closed**: when BLUEPRINT.md is in the PR diff but the merge-base §17.1 snapshot cannot be read (unresolvable merge-base, or the file introduced since the branch-point), the gate reds rather than delegating to the approximate common-prefix fallback, which would silently approve the exact `[1..N]`-on-both-sides cross-PR collision the gate exists to catch. ## 1. What TeaTree Is

A personal code factory for multi-repo projects. It turns a ticket URL into a merged pull request by coordinating the full lifecycle — intake, coding, testing, review, shipping, delivery — across multiple repositories, worktrees, and agent sessions.

**Target:** service-oriented projects with databases and CI pipelines (any language). Not for docs-only repos or CLI tools.

**Operating mode.** TeaTree runs as a long-lived interactive Claude Code session orchestrated by a fat `/loop` (~10–15 min cadence). The loop fans out to in-session subagents per tick to sweep PRs, auto-review PRs assigned to the user, intake assigned issues, watch messaging mentions/DMs, and render a multi-line statusline that is the **only persistent UI surface**. There is no HTML dashboard. The loop runs in the same session the user types into, so debugging stays direct.

**Code-host neutrality.** Pull requests are the canonical concept. Both **GitHub** and **GitLab** are first-class in core; GitLab MRs map onto the PR abstraction at the Protocol layer.

**Messaging-backend pluggability.** Mentions, DMs, and outgoing posts go through a `MessagingBackend` Protocol declared per overlay. Slack (Socket Mode bot) is the first implementation. A `Noop` default lets overlays opt out.

**Core principle.** Infrastructure is deterministic code; development work is skill-guided. State management, port allocation, provisioning, task routing, code-host sync, and messaging integration are Python code with >90% branch coverage. The actual development — coding with TDD, debugging, reviewing, shipping — is driven by skills that encode methodology, guardrails, and domain knowledge.

**Core stays generic.** No customer-, tenant-, or product-specific names appear in the scanned roots (`src/teatree`, `docs`, `skills`, `agents`, `tests`, `scripts` + top-level `README.md`/`BLUEPRINT.md`/`AGENTS.md`); per-overlay specifics live in the overlay package and `~/.teatree.toml`. A gate (`scripts/hooks/check_no_overlay_leak.py`) enforces this — cited as `BLUEPRINT § 1` from `tests/test_no_overlay_leak_hook.py`. It runs two passes: the configured term list (whole-token via `teatree.hooks.term_match`) and an always-on opaque-ID pass (`teatree.hooks.opaque_id`) flagging a real-shaped Slack/forge id (`C0…`/`U0…`/`T0…`) — a leak class with no dictionary word — while an obviously-synthetic placeholder and any `leak-scan:allow` line are exempt. As a pre-commit hook it sees only staged files; the full-tree CI job `overlay-leak-tree` walks every root and passes `--require-terms`, making a missing `TEATREE_OVERLAY_LEAK_TERMS` secret a LOUD MISCONFIGURED failure (exit 2) rather than a silent no-op.

## 2. Architecture Principle: Code-First, Not Skills-First

Infrastructure and orchestration are code; development methodology is skill-guided prose. The split is load-bearing:

1. **Skills are prose, not code.** Prose produces different results depending on model, context, and what else is loaded. Anything that must be deterministic — state transitions, port allocation, provisioning, sync, the `/loop` tick — is code.
2. **Coordination needs transactional guarantees.** An FSM + ORM provide atomic transitions and row-locked workers. Coordination through JSON files cannot.
3. **Code is testable; prose is not.** Core logic must reach >90% branch coverage.
4. **One ABC with a handful of methods beats thirty thin extension points.** Overlay customization goes through `OverlayBase` — typed methods with defaults, no priority system, no plugin registries.
5. **Terminal today, SDK/API-portable by design.** Teatree is driven through Claude Code's interactive terminal app today, but every architectural decision is made so the same flow can be driven through the Anthropic Agent SDK / API. Don't adopt a mechanism that only works in the terminal CLI when an SDK/API-portable equivalent exists. The terminal-driven flow keeps working; SDK/API-portability is the binding constraint when the two pull apart. This is why the headless executor (`agents/headless.py`) is kept deliberately slim — it is the swap point for an SDK runtime — and why dispatch, state, and skill-loading route through code rather than terminal-only affordances.

## 3. Package Structure

Package name: `teatree` (double-e). Repo/CLI: `teatree` / `t3`. Python: 3.13+. License: MIT. Build: `uv`. Entry point: `t3 = t3_bootstrap:main` — a top-level shim that bootstraps Django settings before importing `teatree.cli`.

## 4. Domain Models

Five core lifecycle models in `teatree.core.models/`, all FSM-driven (`django-fsm`): **Ticket**, **Worktree**, **Session**, **Task**, **TaskAttempt**. Around them, supporting rows under the same package: `BotPing`, `ConsolidatedMemory`/`DreamQaProbe`/`DreamRunMarker` (memory-consolidation, below), `DailyDigestThread`/`DailyDigestMessage`, `DbApproval`, `DeferredQuestion`, `EvalRunRecord`/`EvalScenarioResult` (eval-run ledger), `SessionAuditRecord` (eval audit layer), `HonestyEscalation`, `IncomingEvent`, `IntentClassification`, `LivePostApproval`, `LocalStackQueueItem`, `LocalStackReaperMarker`, `Loop` (DB-configured autonomous loop — name/prompt/delay/enabled, §5.6), `LoopLease`, `LoopState` (durable per-loop control plane, §5.6), `MergeClear`, `OnBehalfApproval`, `OutboundClaim` (drift verifier), `PendingChatInjection`, `PullRequest`, `RedCardSignal`, `ReplyDispatch`, `ReviewAssignment`, `ReviewLoop`/`ReviewLoopRound` (the bounded author↔reviewer ping-pong FSM + its round-leg slot, §5.6), `ReviewRequestPost`, `ReviewVerdict`, `ScannedBroadcast`, `SelfImproveFiring`, `SessionHandover`, `TicketTransition`, `WorktreeEnvOverride`. Every row name above is cited by name from §5.6, §17.1, §17.4, or §4 prose.

**Memory-consolidation data layer ([#1933](https://github.com/souliane/teatree/issues/1933)).** Three durable rows back the idle-time "dreaming" pass that clusters session feedback and distils each cluster into one imperative rule. `ConsolidatedMemory` is the consolidation ledger (the DB row is canonical state + audit trail; a rendered topic file is the output), idempotency-anchored on a sha256 `cluster_key` and owning its status ladder `CANDIDATE → VERIFIED → PROMOTED` (plus `SUPERSEDED`/`EXPIRED`); it refuses an uncited rule, refuses to silently expire a BINDING row, and gates index-line pruning on a terminal status with a recorded durable home. `DreamQaProbe` persists the retention/interference/monotonicity QA corpus; `DreamRunMarker` is the staleness marker keyed on the success timestamp (a run that keeps failing stays stale). `t3 dream run` (manual) and `t3 dream tick` (cadence-gated cron, ~04:00) take a `dream-tick` `LoopLease` and run the distillation engine (`teatree.loops.dream.engine.run_consolidation`), stamping `DreamRunMarker` only on a clean pass; `--dry-run` writes nothing. Engine v1 covers phases 1–3 (rank a bounded extract → an injected `Distiller` clusters by root cause → upsert by `cluster_key`, rejecting any uncited/hallucinated cluster and never overwriting a BINDING rule); the injected-distiller seam keeps the pass testable, and an SDK failure propagates rather than laundering into a fake success. **Phases 4–6 are LIVE** ([#1933](https://github.com/souliane/teatree/issues/1933) § 6): on a clean pass the cron runs three memory-file phases over the discovered `~/.claude` memory dirs, each fault-isolated in its own try/except so one phase (or one dir) failing never crashes the tick or stops the others, and each LIVE by default behind its own two-layer kill-switch (`[loops.dream] <phase>` toml / `T3_DREAM_<PHASE>` env, first match wins). **Phase 4** (`teatree.loops.dream.cross_link`) appends symmetric `[[name]]` wiki-links between related memories — related when their significant-token Jaccard clears a deterministic floor (no model) — and is idempotent (a re-run on an unchanged set adds nothing). **Phase 5** (`teatree.loops.dream.reindex`) regenerates `MEMORY.md` as one ≤200-char line per memory (deduped, filename-stable, content stays in the topic file), byte-identical on a no-change re-run. **Phase 6** (`teatree.loops.dream.decay`) ages a memory out by ARCHIVING it (moved to `<memory_dir>/archive/` with a provenance header, never deleted) — gated by a NON-VACUOUS retention guard that retains a memory that is EITHER fresh (mtime within the window) OR still referenced (another live memory or the index `[[name]]`-links it); only a memory that is BOTH old AND unreferenced is archived, and a guard-disabled probe proves the guard has teeth. All three accept an explicit `memory_dir` and never touch the real `~/.claude` under test. **Phase 3b** ([#2346](https://github.com/souliane/teatree/issues/2346)) closes the drift → live-eval loop: `teatree.loops.dream.eval_proposer` maps each grounded cluster to an eval CANDIDATE in a JSONL review queue, then `teatree.loops.dream.promote` PROMOTES it to a real `under_load` scenario (`evals/scenarios/promoted_drift.yaml` + `_pass`/`_fail` replay fixtures) — the artifacts the deterministic replay gate and the weekly metered lane run unchanged. The nightly `tick` requests proposals **LIVE by default** (the `[loops.dream] propose_evals` / `T3_DREAM_PROPOSE_EVALS` kill-switch flips it off). Promotion is auto but gated by a **non-bypassable anti-vacuity guard** (`promote.guard_can_fail`): it runs the real grader (`teatree.eval.report.evaluate`) against a synthesised drift-recommitting transcript and promotes only when that verdict is FAIL (the matchers have teeth) AND a compliant transcript grades PASS (not a tautology); a candidate whose grader cannot fail is rejected — no file written. The guard is deterministic; live metered grading of a promoted scenario lands with the weekly Agent-SDK lane. The `dream` mini-loop is registered `off_live_tick` (driven only by its own cron); `t3 doctor` surfaces the 48h staleness alarm.

`Ticket` also carries a `Role` enum (`AUTHOR` / `REVIEWER`) orthogonal to its state. The reviewer role drives the §5.6 `reviewer_prs` scanner (`Ticket(role="reviewer", issue_url=<pr_url>)`) and short-circuits to `delivered` once the review work completes; the author role is the default lifecycle.

**State machines** (one row per `Ticket` is the unit of work):

| Model | States |
|---|---|
| `Ticket` | `not_started → scoped → started → planned → coded → tested → reviewed → shipped → in_review → merged → retrospected → delivered` (plus terminal `ignored`) |
| `Worktree` | `created → provisioned → services_up → ready → torn_down` (plus the reversible `services_up`/`ready → provisioned` `stop_services` demotion, [#2190](https://github.com/souliane/teatree/issues/2190): docker down only, DB + worktree preserved) |
| `Session` | per-phase quality-gate tracker keyed `(ticket, phase, agent_id)` |
| `Task` | `pending → claimed → in_progress → completed / failed` |
| `TaskAttempt` | execution history rows (immutable) |

**§4 invariant — worker enqueue pattern (load-bearing).** Transitions that own long I/O follow one rule:

- Transition body stays pure: state change + metadata only, then `transaction.on_commit(lambda: execute_X.enqueue(self.pk))`. The state change and the queued work land atomically.
- Workers take a row lock (`select_for_update()`), re-check the source state, run the runner, and on success call the next transition.
- At-least-once delivery is safe because the state guard makes redelivery a no-op.
- `post_transition` signals are reserved for lossy cross-cutting side effects (audit log, Slack reactions) — never for the main work of the transition.

Auto-scheduling chains the phases: `start → provision worker → schedule_planning`, `plan → schedule_coding` (gated on `PlanArtifact` DB record — no artifact raises `NoPlanArtifactError`; the only path from STARTED to CODED passes through PLANNED), `code → schedule_testing`, `test → schedule_review`, `review → schedule_shipping` (gated on `has_shippable_diff()`). The runner classes live in `core/runners/`; the workers in `core/tasks.py` and `core/worktree_tasks.py`. The PLANNED `plan_text` also carries an `E2E test plan / Acceptance scenarios` section for UI-visible tickets (prose-gated on `frontend_repos`, mirroring `is_ui_visible`) so behaviour-level TDD is planned up front (#2240).

**Who drains the queue.** No always-on `db_worker` is assumed; a won-owner tick drains a bounded batch of `DBTaskResult` jobs in-process (`teatree.loop.queue_drain`), standing down while a real `db_worker` holds the `teatree-worker` singleton, and first expires READY jobs older than `T3_QUEUE_STALE_HOURS` (default 24h) to the reversible `FAILED` state so a fresh drainer never fires stale heavy jobs.

**Clean-worktree preflight (#884).** `code()`, `test()`, `review()`, `ship()` refuse the transition if any worktree has tracked uncommitted changes. The refusal raises a `DirtyWorktreeError` (an `InvalidTransitionError` subclass, **not** `TransitionNotAllowed`) inside the caller's outer atomic block, so the FSM state change is rolled back and the lease reaper returns the CLAIMED task to PENDING on the next tick. No auto-stash — worktrees share one `.git`, so a stash is repo-global and could clobber an unrelated branch's work.

**DoD local-E2E preflight ([#88](https://github.com/souliane/teatree/issues/88)).** `ship()` runs a second preflight (`core/gates/dod_gate.check_local_e2e_dod`) after the clean-worktree one: a UI-visible ticket (scoped `repos` intersect the overlay's `frontend_repos`) needs a green local-stack E2E artifact (`Ticket.extra['e2e_recipe'].last_run` with `result == "green"` and `env == "local"`) before leaving the implementation lifecycle. Refusal raises `DodLocalE2EError`; hatch `dod_e2e_override` with a non-empty reason; an overlay without `frontend_repos` is silent. `is_ui_visible` fails **closed** for an unregistered overlay ([#1426](https://github.com/souliane/teatree/issues/1426)), while a known **path-only** TOML overlay is answered from its `[overlays.<name>]` table ([#733](https://github.com/souliane/teatree/issues/733)).

**Mandatory-E2E gate for customer-display-impacting changes ([#1967](https://github.com/souliane/teatree/issues/1967)).** `core/gates/e2e_mandatory_gate` runs at `pr create` and §17.4 `ticket clear`: a change the fail-closed classifier `OverlayBase.classify_customer_display_impact` marks impacting needs a satisfier — a green AND **posted** SHA-bound `E2eMandatoryRun`, a single-use **user** `E2EBypassApproval` (maker/agent/loop refused), or the `e2e_mandatory_gate_enabled` kill-switch. Deny names both remedies. Additive to #88 (which keys on frontend repos; this keys on diff content).

**Pre-merge migration-fork probe ([#995](https://github.com/souliane/teatree/issues/995)).** Two branches each adding a migration off the same parent both pass per-branch CI (each graph is linear), then the post-merge `migrate` fails with `Conflicting migrations detected`. `core/migration_leaf_probe.sha_forks_migration_graph` predicts the merged tree via `git merge-tree --write-tree` (no worktree mutation), parses every migration's `dependencies`, and refuses at §17.4 `ticket clear` when any app's graph would have more than one leaf, with a `makemigrations --merge` hint. Fails open on an uncomputable merge-tree (the merge-time CI re-check backstops). **Commit-time layer ([#2277](https://github.com/souliane/teatree/issues/2277)):** `django-linear-migrations` records a per-app `max_migration.txt` sentinel; a fork rewrites it with a conflict marker surfaced by `manage.py check --tag models` and a pre-commit hook (`scripts/hooks/check_linear_migrations.py`), so the conflict shows at commit time.

**Fix-ticket FixRecord DoD gate ([#1661](https://github.com/souliane/teatree/issues/1661)).** `Ticket.kind` is `feature` (default) or `fix`. For `kind=fix`, `mark_delivered` (RETROSPECTED → DELIVERED) runs `check_fix_record_dod`: refused (`FixRecordDodError`) unless `Ticket.extra['fix_record']` carries `root_cause`, `evidence`, `regression_test`, `observed_red`, and `recurrence_fingerprint` — done = root cause + verified, not a merged manifestation patch. Hatch `fix_record_override` with a reason; `feature` tickets pass.

**Per-ticket spec-coverage DoD gate ([#2232](https://github.com/souliane/teatree/issues/2232)).** The same `mark_delivered` chokepoint runs `check_spec_coverage` after the fix-DoD check. Opt-in `require_spec_coverage` (default NO-OP); when on, every acceptance criterion in `Ticket.extra['spec_coverage']` must name a backing test, else `SpecCoverageDodError` naming the uncovered ACs. Hatch `spec_coverage_override`. Automatic AC extraction from the spec is a follow-up.

**Reviewing-phase review-skill evidence gate ([#1539](https://github.com/souliane/teatree/issues/1539)).** When `review_skill` is set, recording the `reviewing` attestation is refused (`ReviewSkillEvidenceError`) unless `Ticket.extra['review_skill_run']` attests the skill ran (`lifecycle record-review-skill-run`). Empty (default) is a NO-OP. Distinct from the `architectural_review_skill` cadence scanner.

**Review deep-retrieval constraint — path-independent.** Reviewing carries the same responsibility as implementing: a verdict from the diff alone is not a review. `Ticket.review_context_satisfied()` is a `django_fsm` condition on `review()` and `mark_reviewed_externally()`, so when `require_review_context` (default off) is on, every verdict path is refused until `review_context_gate.is_complete` accepts `Ticket.extra['review_context']` (the work item fetched from source, links followed, referenced docs analyzed against the diff). Stamped via `lifecycle record-review-context`; `mark_review_no_action` (bot/no-diff) is exempt.

**Anti-vacuity attestation gate ([#1829](https://github.com/souliane/teatree/issues/1829)).** `core/gates/anti_vacuity_gate` extends the §17.4 CLEAR machinery so a maker cannot request-review/merge an MR whose new regression test is *vacuous* (green even with the bug present). Opt-in `require_anti_vacuity_attestation` (default NO-OP); on, **merge** and **request review** refuse unless `Ticket.extra['anti_vacuity_attestation']` maps the diff to the acceptance criteria and proves each new test anti-vacuous (revert fix → RED) or claims `no_new_tests`. SHA-bound like `MergeClear.reviewed_sha`; stamped via `lifecycle record-anti-vacuity`. `autonomy = full` does NOT collapse it.

**Rubric→verifier done-gate ([#2241](https://github.com/souliane/teatree/issues/2241)).** `core/gates/rubric_gate` adds a sibling §17.4.3 merge precondition: a ticket's `Rubric` must be graded PASS by an INDEPENDENT verifier (≠ maker). Opt-in `require_rubric_verification` (default off), **fail-closed**. Full spec: [docs/blueprint/rubric-done-gate.md](https://github.com/souliane/teatree/blob/main/docs/blueprint/rubric-done-gate.md).

**Sync writers honor the same gate ([#1426](https://github.com/souliane/teatree/issues/1426)).** The DoD decision is factored into `dod_gate.sync_gate_allows` so every sync writer — not just `ship()` — respects it, closing the bypass where sync wrote a post-ship state directly. A **post-ship** state is at or past `SHIPPED`. A *workflow* state inferred from a live open PR is *capped* — a gate-refused state demotes to `STARTED`; a *terminal* state reflecting an external fact (merged PR → `MERGED`, board "Done" → `DELIVERED`) is **never demoted** but records a durable `dod_e2e_violation` audit marker and logs loudly, so the gap is auditable not silent.

**Concurrent-stack cap (#1397).** `max_concurrent_local_stacks` caps the distinct tickets whose worktrees can be in `services_up`/`ready` at once per overlay. The gate (`local_stack_gate.check_local_stack_limit`) runs before `Worktree.start_services()` and raises `LocalStackLimitExceededError` naming the blockers. Default `0` is unbounded; sibling worktrees of one ticket count as one stack; each blocker is reconciled against docker first (a zero-container row is a phantom and excluded; a mid-restart row stays counted, fail-safe).

**Provisioning time-box + loud alert ([#2220](https://github.com/souliane/teatree/issues/2220)).** A long-blocking provisioning subprocess (DSLR restore, `migrate`, `--create-db`) must fail loud, never grind silently. `provision_timebox.run_timeboxed_step` bounds each step by `provision_step_timeout_seconds` (default 1800); on the ceiling it aborts and DMs the user, while a heartbeat distinguishes slow-but-moving from a hang. `detect_migration_conflict` matches a running `migrate`'s conflict output and surfaces the `makemigrations --merge` remedy. Best-effort.

**Wrong-repo worktree guard ([#2276](https://github.com/souliane/teatree/issues/2276)).** `find_clone_path` resolves a ticket's repo by basename, so a sibling clone of the same name (different `origin`) could be cut silently into the worktree. `WorktreeProvisioner._create` calls `git_guard.guard_repo_remote_slug` before `git worktree add` whenever `repo_name` is an `owner/repo` slug: a mismatch between the clone's `origin` slug and the expected slug raises (a missing `origin` fails closed). A bare basename carries no canonical slug, so the legitimate `--repos <basename>` flow is untouched.

**Idle-stack reaper + acquisition queue ([#2190](https://github.com/souliane/teatree/issues/2190), [#44](https://github.com/souliane/teatree/issues/44)).** Four composed parts replace the hard `SystemExit` at the cap; all are global, mechanical-only, and individually kill-switched.

- **Reaper** (`IdleStackReaperScanner` → `reap_idle_stack`): demotes an IDLE running stack to `provisioned` via `stop_services` (reversible — DB + worktree preserved). Idle (`core.gates.idle_stack.preserve_reason`) = no live `Session`/`Task`, `last_used_at` past `idle_stack_idle_minutes`, not the CWD, no live external-delivery lease, no recent E2E run, not `reaper_pinned`; fail-safe ⇒ KEEP. `classify_running_worktrees` logs preserved-vs-reaped so a reap is never silent (#2227).
- **Queue**: at the cap `acquire_or_enqueue` reaps → retries → enqueues a `LocalStackQueueItem`; `LocalStackQueueDrainerScanner` re-checks the cap each tick with Fibonacci-minute backoff and re-fires `start` once a slot frees — never tearing down another ticket's stack.
- **Stale-stack reaper** ([#2207](https://github.com/souliane/teatree/issues/2207)): unowned compose projects (no `Worktree` row) older than `stale_stack_min_age_minutes` (default off) are downed via `workspace reap-stale`; unknown age ⇒ KEEP.
- **Safe disk reclaim** ([#2246](https://github.com/souliane/teatree/issues/2246)): `workspace reclaim-disk` runs only three zero-data-loss prunes (build cache + dangling-only images + unreferenced-only volumes) via a fixed argv, so `-a`/`system prune` can never enter the set — the sanctioned path so the agent stops hand-rolling raw `docker image prune -af`.

## 5. Agent Execution

The agent layer is `teatree.agents` (headless executor + prompt + skill bundle + structured result schema) and `teatree.loop` (the /loop topology). The orchestrator-as-keystone contract is §17.8 — every implementation, review, test, debug, and ship action is dispatched to a sub-agent; the orchestrator's job is synthesis and dispatch, not execution.

**Headless executor (`agents/headless.py`).** Drives the agent in-process via `claude-agent-sdk` (`ClaudeSDKClient`) — a REAL-environment run, with the resolved phase model, the worktree `cwd`, and the parent session via the SDK `resume=` option. The typed `ResultMessage` maps to a `TaskAttempt` through the shared `attempt_recorder` chokepoint (#2204; the prior `claude -p` subprocess is gone). A heartbeat-driven `LoopWatchdog` interrupts the SDK client on a turn/cost/runtime breach so an in-flight agent flushes its final status ([#997](https://github.com/souliane/teatree/issues/997)); a per-ticket `TicketBudget` caps cost; a `ResultMessage(is_error=True)` naming a usage/weekly limit lands FAILED, never a silent success.

**Interactive-by-default phase dispatch (post-2026-06-15 billing).** A detached `claude -p` is metered now, so loop-dispatched phase work runs as in-session `Agent` sub-agents. The `Task.save` invariant routes a task whose `(role, phase)` has a registered agent (`Task.loop_dispatched`) to `INTERACTIVE`; free-form phases stay HEADLESS. The `/loop` slot claims each (`claim-next --json`, carrying loop-resolved `model`+`skill_bundle`), spawns its `Agent`, then records via `tasks record-attempt` — the shared `agents/attempt_recorder` applies the same schema+evidence gate as `claude -p`. Fail-closed: `execute_headless_task` refuses a loop phase unless `LOOP_ALLOW_HEADLESS_DISPATCH` (default off). Evals split the same way: `t3 eval run` defaults to `--backend subscription` (grade in-session transcripts, no API spend); CI passes `--backend sdk` for the budgeted in-process Agent-SDK path, and `--trials`/`--models` force the metered runner. **Metered-lane auth + isolation.** The `sdk` lane authenticates from `CLAUDE_CODE_OAUTH_TOKEN` and runs **in a container, never on the host** (`--docker` → `dev/Dockerfile.test`, which ships the `claude` CLI; the credential is forwarded via docker `-e` pass-through, never argv/logs). The runner and judge run **virgin** (`isolated_claude_env` + clean-room `ClaudeAgentOptions`: `setting_sources=[]`, empty `settings`, a neutral `HOME`/cwd) so the developer's `CLAUDE.md`, auto-memory, and project config never bias a result. **All-skipped guard ([#1811](https://github.com/souliane/teatree/issues/1811)).** A run that never happened (`claude` absent/unauthenticated) would SKIP as passed; `--require-executed` exits non-zero when `collected > 0` but `executed == 0`. The standalone metered workflow (`.github/workflows/eval.yml`, weekly cron + manual) passes `--require-executed` unconditionally (never key-gated) and is no-PR-guarded so a window with nothing merged skips cleanly.

**Structured result schema (`agents/result_schema.py`).** Agents return JSON: `summary`, `files_modified`, `tests_run`, `tests_passed`, `tests_failed`, `decisions`, `needs_user_input`, `user_input_reason`, `next_steps`, `commands_executed`. `additionalProperties: false`. Validated without the `jsonschema` library to keep the dep tree small.

**Skill bundle (`agents/skill_bundle.py`) + delegation map (`skill_support/map.py`).** A phase → companion-skills map (e.g. `coding → test-driven-development`), topo-sorted `requires:` resolution, per-overlay `companion_skills`. Reviewer-dispatch review skills come from `OverlayConfig.get_review_companion_skills()` (default `code-review`); on the `reviewing` phase `build_system_context` embeds them in full, and an orchestrator-built reviewer dispatch goes through the single shared `build_reviewer_dispatch_prompt()`, which prepends a REQUIRED Skill-tool load block (`t3:review` + overlay review skills) so conventions reach the reviewer structurally, not via orchestrator memory. The **coding phase** force-loads a symmetric contract: a REQUIRED load block (`/t3:architecture-design`+`/t3:code`), `architecture-design` embedded in full, and behavior-preservation + no-AI-signature clauses, so the discipline reaches a skill-stripped builder. It also injects an explicit stack+overlay load block computed from the resolved bundle (framework skill `/ac-django`/`/ac-python`/`fastapi` plus the active overlay skill), so a code-touching dispatch never relies on auto-detect; an unresolved stack falls back to a "load your stack's coding skill yourself" instruction ([#1368](https://github.com/souliane/teatree/issues/1368)).

**Model tiering.** `agents/model_tiering.resolve_phase_model(phase)` pins the genuine-design phase (`planning` → opus) and downgrades mechanical phases (`reviewing`/`requesting_review`/`testing`/`shipping` → sonnet, `retrospecting` → haiku) by default; reasoning phases (`coding`, `debugging`) inherit the user's default. Per-phase overrides via `[agent] phase_models` in `~/.teatree.toml`.

**Model capability order.** `core/cost.tier_rank(model)` ranks a model id / tier name along a *capability* axis `haiku < sonnet < opus < fable`, distinct from `core/cost` pricing (the new `fable` tier prices at $10/$50 per MTok, above opus). An unknown full id ranks *above* every known tier (most-capable, so a lower floor never silently downgrades an unknown spawn target), the inverse of the conservative pricing fallback. This order is the comparator for the most-capable-wins floor merge below.

**Per-skill model floor + spawn-model merge.** `model_tiering.resolve_spawn_model(phase, *, skills)` is what both spawn paths call (headless `claude -p --model` and interactive Agent-tool dispatch). It starts from the per-phase tier and raises capability for each loaded companion skill carrying a `[agent.skill_models]` MODEL floor (most-capable-wins via `tier_rank`, order-independent). The floor is MODEL only — no per-skill effort axis — and returns `None` (inherit) when nothing applies, so absent config is byte-identical to the prior phase-model behaviour.

**Session-level pins (interactive main agent).** Effort is settable **only session-wide**: `[agent] session_effort` (`low|medium|high|xhigh|max`) and `[agent] session_model` are injected as `--effort`/`--model` into the interactive `claude` spawn argv, so the user never runs `/model`. Effort is never per-sub-agent (the Agent tool has no effort param) and never threaded through `claude -p`. `t3 doctor` validates the values (a bad effort FAILs, an unknown model WARNs) and `_check_stale_path_t3` ([#2281](https://github.com/souliane/teatree/issues/2281)) FAILs when a `t3` binary precedes `$UV_TOOL_BIN_DIR` on `PATH`, masking `uv tool install` updates.

**Selective per-phase fan-out ([#2229](https://github.com/souliane/teatree/issues/2229)).** A declarative, default-OFF way for an opted-in phase to fan work out (adversarial-verify on review, judge-panel on planning) without the blanket session-wide `ultracode`. The `core.phases.FANOUT_BY_PHASE` registry maps `(role, phase) → PhaseFanout`, parallel to `SUBAGENT_BY_PHASE`; the opt-in is `[agent.phase_fanout]` keyed `"role:phase"` (`true` = default width, int overrides). `resolve_fanout_directive` renders `""` with no entry/opt-in (the byte-identical spine) else a **plain-text** directive (§2 SDK/API-portable, [#2236](https://github.com/souliane/teatree/issues/2236)) threaded as the `fanout_directive` dispatch field. Ships **route (b)** only (rigor inside the one in-session sub-agent); real top-level parallel fan-out is deferred.

**Situational honesty-critical escalation ([#2263](https://github.com/souliane/teatree/issues/2263)).** Routes a *verification* spawn to the most-honest model (`[agent] honesty_model`, default Fable) when the agent's honesty is in question — situational, NOT a phase floor. Four triggers write a session/task-scoped `HonestyEscalation` row (mostly agent-fired via the `skills/rules` directive `t3 <overlay> honesty escalate`; `shipped_incomplete` also has a deterministic backstop in the rubric done-gate). `resolve_spawn_model` raises the winner to `honesty_model` when a verification phase has an active escalation — after the skill-floor loop, before `_downgrade_fable`, so the `fable_enabled` kill-switch still applies (load-bearing ordering). `is_active` auto-clears at read-time (`cleared_at IS NULL AND expires_at > now()`, 6h TTL backstop); the primary clear is a fully-passed `rubric-grade`.

### 5.6 Loop Topology

**Track A vs Track B ([#1838](https://github.com/souliane/teatree/issues/1838)) — read this first.** Two orthogonal axes restructure the loop, and "panes" belongs to only one of them:

- **Track A — restructure the loop on the lead session, NO panes.** Arm the dormant `orchestrate_phase` (`[teatree] orchestrate_claim_enabled`, [#2315](https://github.com/souliane/teatree/issues/2315)), per-loop ownership ([#2318](https://github.com/souliane/teatree/issues/2318)), and split the one fat `/loop` into N dedicated `/loop` slots (`[teatree] dedicated_loops`, [#2319](https://github.com/souliane/teatree/issues/2319)). The fat single-`/loop` shape below is the **default of a default-off toggle** (`dedicated_loops = false`), not the only structure — flipping the toggle restructures the lead's own loop, on the lead session, with no separate sessions involved.
- **Track B — pane-backed teammates.** A pane is a separate long-lived Claude session executing claimed `team:<role>` work. CORE-MAKER + OVERLAY-MAKER are built; REVIEWER panes are prohibited in code (a code-level raise pinned by a fitness test). Track B is the only axis that introduces panes.

`[teams] enabled = false` (default) is the master agent-teams off switch ⇒ the loop runs the classic in-session `Agent(run_in_background)` sub-agent fan-out, no panes. Flip it via `t3 teams on` / `t3 teams off` (`t3 teams status` to read). The two axes compose independently: Track A's `dedicated_loops` reshapes the lead loop whether or not Track B's panes are enabled.

TeaTree drives the day from a single long-lived Claude Code session running a fat `/loop` (the `dedicated_loops = false` default of Track A above), firing on a fixed cadence (default 12 min). The tick body `teatree.loop.tick.run_tick` is code — tested, typed, version-controlled — composing named single-responsibility phases ([#1796](https://github.com/souliane/teatree/issues/1796)): `scan_phase` (parallel world-scan), `sweep_phase` (maintenance scanners split out of the scan), `act_phase` (dispatch + mechanical + persist), and `orchestrate_phase` (the speed-driven autonomous fan-out — a no-op at default `medium`; `slow` admits one, `full`/`boost` clamp to the per-overlay `max_concurrent_auto_starts` budget). `orchestrate_phase` is wired as a **read-only budget PLANNER** — it computes the clamped cap but never claims a `Task` (the single claim point stays the live `claim_next` CAS, so the tick can never orphan a claim). When the opt-in, default-OFF `orchestrate_claim_enabled` toggle is ON and the speed clamps, the tick persists the cap as an admit-budget ceiling to the `tick-meta.json` sidecar (`teatree.loop.admit_budget`); the live claimer reads it and refuses the marginal claim once in-flight CLAIMED WIP hits the ceiling, so claimed ≡ spawned. At `medium` or with the toggle OFF no budget key is written (absence = unclamped = byte-identical throughput), and the reader fails open on an absent/stale/corrupt budget so a dead loop never throttles live dispatch ([#1838](https://github.com/souliane/teatree/issues/1838), [#1917](https://github.com/souliane/teatree/issues/1917)).

**Agent-teams Track-B — the inert maker-only pane layer ([#1838](https://github.com/souliane/teatree/issues/1838)).** A maker pane is a long-lived `team:<role>` claim of the existing `Task` + lease (no new model). `teatree.teams` houses the lifecycle FSM (`spawn → active → idle → stopped`, state derived from the live `Task` claim + lease + `Session`), the idle-pane reaper (demote a pane with no live `Session` past `[teams] idle_minutes`, fail-safe KEEP), and the loop-owner collision guardrails (a pane claims ONLY `team:<role>`, never `loop-owner` or a `loop:` slot — proven-disjoint key spaces). Budget knobs `[teams] max_panes` (1) and `idle_minutes` (30) on `UserSettings` are fail-safe to their defaults at every tier, so the safety bound cannot be configured away; a pane heartbeats via `Task.renew_lease`, so the existing orphan-reclaim recovers a dead pane. **Default-OFF** (`teams_enabled` false): the maker path and reaper are no-ops. The only live-path importer of `teatree.teams` is the sanctioned, `None`-when-off pane-reaper scanner, pinned by the [#2320](https://github.com/souliane/teatree/issues/2320) AST inertness scan. On `SessionStart(source=compact)` the lead synchronously orphans the stale `loop-owner` lease so its next tick re-anchors uncontested and no pane can win the compaction-window CAS race. REVIEWER panes are never spawnable (a code-level raise pinned by a fitness test); reviewing stays on the fresh-spawn-per-head_sha AutoReviewDispatch path. **Pane DISPLAY layer (WI-5, presentation-only).** `pane_display` optionally renders a pane's in-process SDK session in a visible terminal via plain `tmux split-window -- claude --resume <session_id> …` (the resumed id is the SAME SDK session, the single SOT); native iTerm2 splitting is just a property of how the user attached (`tmux -CC`). It never replaces the SDK run — no tmux/TTY/any failure falls back to the in-process path — and the in-pane `claude` is plain interactive (never `claude -p`), so billing stays on the interactive lane. Teardown is DB-driven: demote-to-STOPPED triggers `reconcile_orphan_panes` to `kill-pane` every orphaned `team:*` pane. Mode `[teams] display = auto|tmux|none` (default `none`, fail-safe); `none` + headless/CI ⇒ byte-identical to today.

**#786 epic — the immortal-singleton roster model is fully retired (WS1–WS5 + #54, all merged).** The original model — a coordinator spawning a fixed roster of long-lived loop sub-agents it had to keep alive and re-spawn on death/compaction — was the root cause of the recurring "loop died on compaction / re-spawned" toil and the duplicate-on-restart hazard. It is **fully retired**: no roster, no `spawn_brief`, no takeover-respawn, no resume-by-agentId. The replacement satisfies three acceptance-contract invariants, each delivered by a specific workstream and detailed in the appendix:

- **Invariant 1 — 0 sessions ⇒ nothing runs.** The loop is session-bound; zero open sessions ⇒ dormant, by design (WS3). The optional macOS LaunchAgent (`t3 loop install-watchdog`, [#1139](https://github.com/souliane/teatree/issues/1139)) is a session-watchdog, not an OS daemon — it re-runs `t3 loop spawn-headless` on Claude Code exit and after `/login` so a session is normally available. Two in-session siblings ride `SessionStart` / `t3 doctor check` / the recovery path: `account_switch` ([#1916](https://github.com/souliane/teatree/issues/1916)) compares `oauthAccount.accountUuid` and, on a `/login`, invalidates the backend token cache so DMs do not route to the old workspace; `mcp_connectivity` ([#2282](https://github.com/souliane/teatree/issues/2282)) live-probes every enabled MCP server via `claude mcp list` and validates each against its overlay-declared provider (`get_mcp_provider_expectations()`), making a disconnected/mismatched MCP a loud finding.
- **Invariant 2 — ≥1 session ⇒ exactly one machine-wide tick.** Driven by the recurring `t3 loop tick` cron; the executor mutex is the WS2 `LoopLease` DB row (backend-agnostic conditional-UPDATE CAS, expiry-reapable), and the WS3 single Django-free `_OWNER_LOOP` tick-owner record names which session ticks. Atomic per-unit claim is WS1 `t3 loop claim-next` (claim == spawn boundary). A second concurrent tick loses the CAS and SKIPs. **Durable per-loop control plane ([#1913](https://github.com/souliane/teatree/issues/1913)).** `LoopState` is the canonical control tier above the env/toml `[loops]` config (mirrors `ConfigSetting`): one row per loop name carries `enabled`/`paused`/`disabled`, an absent row resolves to `enabled` (empty table is a provable no-op), and a `paused`/`disabled` row beats `always_on`. `t3 loop {pause,resume,disable,enable,loop-state} <name>` are atomic, idempotent single-row upserts via the `loop_state` management command (`resume`/`enable` are the one "make it run again" verb clearing either hold). The hold is consulted first by the tick gate (`LoopsConfig.is_enabled`), the review-claim chokepoint (`review_claim_signals.review_loop_enabled`), AND the in-session Stop self-pump on the `dispatch` loop — the restart-surviving "pause everything" the in-process `T3_LOOPS_DISABLED=all` kill-switch could not durably do (2026-06-03 incident). The shared read `loop.loop_state_db.loop_held_in_db` is a `domain` leaf and every DB read fails open/safe so a hiccup never silently disables a loop or crashes the Stop hook. **Toward autonomous DB-configured loops ([#1796](https://github.com/souliane/teatree/issues/1796)).** A `Loop` row (`teatree_loop`: `name`/`prompt`/`delay_seconds`/`enabled`/`last_run_at`) is the durable definition of one autonomous loop — the DB-configured replacement for the fat tick, runnable on its own `delay` (a never-run loop is due immediately). The `prompt` is the instruction to run the loop's work each tick (e.g. "run a sub-agent to call the work scan"): the domain scanners under `teatree.loops` stay as the Python code the loop invokes — they are scan units, not separate loops. A data migration seeds one autonomous row per loop, each with its own cadence — a fixed `delay_seconds`, or a `daily_at` wall-clock time for once-per-day loops (`news` at 08:00, `dream` at night). Loops are not only the registry scanners — the reactive Slack-answer loop is one too. Rows are managed from the Django admin and rendered read-only via `t3 loops list` (the `loops_list` management command). This first stage is additive — the legacy singular `t3 loop` surface and the `LoopState`/`MiniLoopMarker`/`[loops]`-toml control tiers above remain until later #1796 stages cut the row over to the single source of truth. The **master runner** drives this: `t3 loops tick` runs ONE pass — every enabled, due `Loop` row fans out through the shared `run_tick` pipeline, gated by `build_loop_table_jobs` on the row's `is_due` (interval or `daily_at`) and bumping `last_run_at` — and `t3 loops run` repeats it on an interval. The master claims the singleton `t3-master` lease (the renamed `loop-owner`): the owning session re-claims each tick, a non-owner SKIPs.
- **Invariant 3 — exactly one TODO-consolidation loop per agent identity, across all sessions.** The WS4 per-agent consolidation self-pump, keyed by `agent_id` in a separate consolidation-registry.

**Session-start auto-load is opt-in ([#256](https://github.com/souliane/teatree/issues/256)).** The `SessionStart`/first-prompt loop machinery — the tick-owner claim (`handle_session_start_bootstrap`), the `CronCreate` registration nudge (`handle_enforce_loop_on_prompt` / `_loop_registration_exempt`), and the loop statusline (`hooks/scripts/statusline.sh`) — fires only when the session both opted into teatree (the `<session>.teatree-active` marker) AND the operator enabled auto-load. The single gate is `hook_router._loop_auto_load_active` = `_teatree_active AND _loops_auto_load_enabled`. `_loops_auto_load_enabled` defaults **OFF** (env `T3_LOOPS_AUTO_LOAD` first, then `[loops] auto_load` in `~/.teatree.toml`; fails CLOSED on absence — the inverse of the `[loops] enabled` fail-OPEN kill-switch). So a colleague cloning the repo, even one who loads a teatree skill for dogfooding, is never nagged to register a cron or shown the loop statusline; the loop owner opts in once and keeps the prior behaviour intact.

**Subsumed issues (WS5 — documented, not closed here).** [#789](https://github.com/souliane/teatree/issues/789) (a non-owner session still arming the tick cron) is **subsumed**: under the WS1 claim/lease a non-owner tick finds nothing to claim, so the concern dissolves — #789 was closed-as-completed when WS3 landed and is **not** reopened. Board task #50 (the per-agent TODO-consolidation loop) is **subsumed by invariant 3 / WS4**; #50 is a project-board card, **not** a repository issue, so it is documented as subsumed here and tracked on the board — no repo issue to close. WS5 itself carries no GitHub closing keyword on the #786 umbrella; only an explicitly-authorized epic-completion step does.

**Deep mechanics live in [docs/blueprint/loop-topology.md](docs/blueprint/loop-topology.md).** The DB-lease singleton, the loop-owner claim and `SessionStart` tick-owner record, the per-agent self-pump, the Stop-gate family, post-compaction snapshot recovery, the three-stage tick (scan → dispatch → render), the full scanner set, multi-overlay/host/identity scanning, the auto-start/dispositions/completion phases, and §5.6.1 Statusline / §5.6.2 Mode + training-wheel / §5.6.3 Availability all live there. CORE always-on or cross-referenced scanners worth naming here: `architectural_review` (cadence-and-merge-count, `architectural_review_disabled`); `scanning_news` ([#1191](https://github.com/souliane/teatree/issues/1191), with the `ask_before_creating_news_tickets` gate recording a `PendingArticleSuggestion` rather than auto-filing); `dogfood_smoke`/`eval_local` ([#1308](https://github.com/souliane/teatree/issues/1308)); `review_request_merge_react` (reacts `:merge:` once an MR merges); the closure-reverify Stop WARN (non-blocking so it cannot deadlock the loop); the `SessionHandover` hand-off (`t3 <overlay> handover`, injected on `SessionStart`); the open-PR statusline anchor ([#271](https://github.com/souliane/teatree/issues/271)) read from a per-tick `open-prs.json` cache so the display costs no API call; and the typed `jobs_for_domain` seam ([#1482](https://github.com/souliane/teatree/issues/1482)) partitioning the per-overlay scanner fan-out.

The [#1554](https://github.com/souliane/teatree/issues/1554) `issue_implementer` mini-loop closes the auto-implement-intake gate: each tick its per-overlay scanner claims open issues carrying `issue_implementer_label` via the TOCTOU-safe `ImplementedIssueMarker.claim`. It is **default-OFF behind a triple gate** — the master `issue_implementer_enabled` flag, an `in_flight_count < issue_implementer_max_concurrent` budget (default 1), and per-issue `claim()` idempotency; an empty label is a safe no-op that logs a WARNING. Each claim emits `issue_implementer.claimed` routing to `t3:orchestrator` as a **maker-side kickoff** — it starts the normal pipeline, issues no `MergeClear`, gains no merge authority (the §17.4 maker≠checker boundary holds). The scanner skips any issue carrying `needs-triage` (a maintainer hold); since the factory files issues *as* the maintainer, agents self-apply `needs-triage` to anything they file that is not a direct user order.

### 5.7 Self-Improving Monitor

A detector swarm that rides the same tick the regular `/loop` runs. It watches for smells the rest of the loop cannot self-report — dispatcher silently skipping a phase, a `MergeClear` issued but never reconciled, a statusline entry whose evidence has gone stale — and converts each into a `SelfImproveFiring` row plus a graduated action (`log → statusline → slack → ticket → auto_fix`, monotonic ladder). It is the legibility substrate §§17.4–17.8 relies on. Auto-fix is whitelisted: today only `StaleStatuslineEntryDetector` carries `auto_fix = True`. The shipped detector set (`detectors/registry.py`) is `DispatchGapDetector`, `ForgottenMergeDetector`, `StaleStatuslineEntryDetector`; additional `auto_fix` slots land with their own structural whitelist test.

Never tears down another ticket's stack; per-item backoff IS the cadence (no marker); `local_stack_queue_disabled` kill-switch; mechanical-only. |
| `TodoSweepScanner` ([#129](https://github.com/souliane/teatree/issues/129)) | stale TODOs | Verifies each open `Task`'s artifact via `is_issue_done`; terminal → `todo.completion_detected` re-checks live (fail-CLOSED) before `Task.complete`; unverifiable → `todo.orphaned` (fail-OPEN). Per-item, idempotent via `Task.last_sweep_check_ts`; `todo_sweep_disabled`. |
| `IssueDispositionScanner` ([#2122](https://github.com/souliane/teatree/issues/2122)) | `needs-triage` dead noise | Lists `needs-triage` open issues and auto-closes ONLY high-confidence DEAD ones via three deterministic buckets — `already_shipped` (a delivered ticket exists, falsified by a live in-flight ticket), `exact_duplicate` (title fingerprint matches another open issue, falsified by a unique fingerprint), `obsolete` (every body-referenced path is gone, falsified by one still-existing path). Conservative bar: ANY uncertainty emits nothing. Emits `issue_disposition.close_candidate` → the idempotent `close_dead_issue` mechanical handler closes with an audit-trail comment; bounded `auto_disposition_max_closes_per_tick`. **Default-OFF** behind `auto_disposition_enabled` (per-overlay overridable). It can CLOSE noise but is physically unable to enqueue work — no `Task`, no claim, never an agent route — so the burn-down doctrine (done-signal = open-bug count, never an empty queue) is structurally preserved. |

`CodexReviewScanner`'s and `PrSweepScanner`'s auto-dispatch is gated on the fleet doctrine (`mode = "auto"` + `require_human_approval_to_merge = false`); every other overlay stays manual. All scanner knobs are per-overlay overridable.

The monitor never auto-merges substrate, never auto-edits memory / skills / `BLUEPRINT.md`, and never bypasses the §17.4 `MergeClear` reviewer-attestation **except** on a solo overlay the user declared end-to-end-trusted (`mode = "auto"` + `require_human_approval_to_merge = false`) — where the per-diff CLEAR cannot be issued because maker and reviewer are the same human identity, so an unrelaxed gate would silently no-op every green PR ([#1309](https://github.com/souliane/teatree/issues/1309)). The carve-out is *minimal*: every precondition gate (draft, changes-requested, CI, uv-audit escalation) stays in force, the per-diff CLEAR is replaced by the SHA-bound `merge_pr_squash_bound` fallback (#1985, re-checking live SHA-bind/draft/CI), and the **cold-review floor still holds** — the bypass requires a recorded INDEPENDENT `merge_safe` `ReviewVerdict` at the live head (`reviewer != maker`) ([#68](https://github.com/souliane/teatree/issues/68)). Non-opted-in overlays keep the CLEAR requirement.

**Auto-review dispatch closes the no-review loop ([#68](https://github.com/souliane/teatree/issues/68)).** On `flag_no_review` for a green+clean own PR the scanner enqueues ONE claimable `Task(phase=reviewing)` (gated on `auto_review_dispatch = solo_overlay and not require_human_approval_to_merge`); the reviewer's recorded `merge_safe` verdict is the artifact the sweep merges on. The merge is **event-driven, not cadence-bound** ([#2026](https://github.com/souliane/teatree/issues/2026)): `review record` calls `trigger_sweep_for_verdict` the moment a `merge_safe` verdict lands, rebuilding the overlay sweep scanner and running its single-PR `evaluate_one` — the same decision ladder `scan` runs, so the two paths cannot drift; the periodic sweep stays the backstop. Dedup/contract mechanics live in `AutoReviewDispatch`'s docstring.

**Review loop FSM — the bounded author↔reviewer ping-pong ([#2298](https://github.com/souliane/teatree/issues/2298)).** `ReviewLoop` is an independent FSM (`AUTHORING → REVIEWING → {PASSED, EXHAUSTED}`, NOT a `Ticket.State` transition) that encodes iterate-then-terminate review: each verdict-guarded transition reads the *recorded verdict*, never the caller, so a HOLD re-arms a fresh author leg carrying the reviewer punch-list (bounded by `max_rounds`; `exhaust` surfaces `needs_user_input`). The `review record` chokepoint drives it (`_advance_review_loop`), closing the pre-#2298 gap where a HOLD was inert. Two variants: the **SELF** variant records an internal verdict and posts nothing (no `ReviewVerdict`, no egress); the **EXTERNAL** variant binds a real `ReviewVerdict` through the cold-review path and `pass_` sets the `passed` proceed flag (it does NOT itself merge — that stays the sweep's job), and runs the `e2e`/`e2e_reviewing` phases (author `t3:e2e`, reviewer `t3:e2e-review`). Leg dispatch is idempotent per `(review_loop, round, leg)` via the `ReviewLoopRound` unique constraint.

**Delivery-ownership predicate stops double-scheduling externally-delivered units ([#2104](https://github.com/souliane/teatree/issues/2104), [#2217](https://github.com/souliane/teatree/issues/2217)).** A unit delivered out-of-band by a hand-dispatched delivery agent (per `/teatree-batch`) is implemented directly with no planning phase and no loop-armed reviewer, so the loop must not re-derive that work each tick. `external_delivery.under_external_delivery(ticket)` is true while a TTL'd `external_delivery` lease on `Ticket.extra` is live; it is claimed at the `workspace ticket` entry the delivery agent runs (never by the loop's own FSM) and refreshed on each external-owner seam (`ticket plan`/`transition`), so a hand delivery cannot lapse mid-flight while a crashed owner still self-reaps on the 1h TTL. The lease is consulted at **three** chokepoints: `execute_provision` skips `schedule_planning`; the `pr_sweep` review-arm no-ops (resolving the PR back to its author/delivery ticket via `resolve_author_ticket`, never by `issue_url == pr_url`); and the global dispatch chokepoint `orchestrate._dispatchable_filter` excludes live-lease tickets at the DB layer via `not_under_external_delivery_q` (held in lockstep with the Python predicate by a parity test). Without the third, hand-advancing STARTED → PLANNED made any non-planning phase dispatchable and a second coder was claimed (the #2217 incident).

**Filesystem-evidence double-dispatch guard ([#2217](https://github.com/souliane/teatree/issues/2217)).** The lease above only protects a unit the DB knows about; the #2217 incident produced a worktree on disk with **no `Ticket` row**. `worktree_collision.find_foreign_issue_worktrees` keys on the issue number instead — the dir for issue `N` is deterministically `<N>-<slug>`, so globbing `<N>-*` finds any existing worktree regardless of DB state. The single chokepoint is `workspace ticket`: inside the same `atomic()` that creates the ticket, a foreign `<N>-*` dir raises `ForeignIssueWorktreeRefusedError`, rolling the ticket back so a refusal leaves zero DB trace; the CLI prints the colliding path with a `--take-over` opt-out (the ticket's own dir is excluded, re-provision is idempotent). It is the filesystem-keyed complement to the DB lease.

**Lightweight trivial-work plan-skip carve-out.** Some AUTHOR tickets are trivial mechanical edits (a typo, a one-line constant) where planning is pure overhead. `trivial_plan_skip.mark_trivial_plan_skip(ticket, reason=…)` stamps a durable marker (no TTL) carrying a MANDATORY reason (a malformed/empty marker is treated as absent, fail-safe to "planning required"). Consumed at the same two seams — `execute_provision` skips `schedule_planning`, and `check_plan_artifact` accepts the marker so the ticket advances STARTED → PLANNED with no `PlanArtifact`. Recorded via `t3 <overlay> ticket skip-planning <id> --reason`. The lightweight sibling of the heavyweight `plan-bypass` (`--human-authorize`, which fabricates a `PlanArtifact`); both stay intact and scoped to a ticket that explicitly carries the marker.

### 5.8 Reactive Slack-Answer Loop

A tight-cadence (default 20s), token-cheap third `/loop` slot that answers user DMs out-of-band so a quick ack / status question gets a reply in seconds, not at the next fat tick. Coalesces consecutive same-user messages into one logical turn, classifies (pure Python) into `ACK_ONLY` / `SIMPLE` / `NEEDS_WORK`, and either reacts, posts a threaded reply, or delegates to the `t3:answerer` sub-agent.

## 6. Overlay System

An overlay is a downstream Django project that customizes teatree for a specific project/organization.

**§6.0 Overlay Thinness Principle (Non-Negotiable).** Generic workflow logic belongs in core, not in overlays. Before adding logic to an overlay, ask: "Would a different project using the same framework need the same logic?" If yes, it belongs in core — parameterized and configurable. Overlays provide only: (1) configuration values, (2) project-specific glue, (3) truly unique workflows. Everything else — DB provisioning strategies, migration runners, symlink management, service orchestration — is a configurable engine in core; the overlay configures it, never reimplements it. An overlay method exceeding ~30 lines of non-configuration code likely contains generic logic to extract.

**OverlayBase ABC (`teatree.core.overlay`).** Composition over inheritance: `overlay.config` is an `OverlayConfig` dataclass; `overlay.metadata` an `OverlayMetadata`. All methods take a `worktree` for context.

| Required | Returns | Purpose |
|---|---|---|
| `get_repos()` | `list[str]` | Repositories to provision |
| `get_provision_steps(worktree)` | `list[ProvisionStep]` | Ordered setup steps |

Optional hooks cover env, services, Docker base images, DB import strategy, post-DB steps, symlinks, PR validation, sync targets, skill metadata, CI config, E2E config, variant detection, tool subcommands, visual QA targets, E2E env extras + preflight, the per-worktree task commands (`get_test_command` → `run tests`, `get_lint_command` → `run lint`), the auto-merge guard (`can_auto_merge` → `MergeGuard`), and the §17.4 cross-repo merge probe's working-repo declaration (`get_merge_candidate_repo_slugs()` → `list[str]` of `owner/repo` slugs for repos the overlay operates merges on but does not package, so a CLEAR whose PR lives in one of them can be bound — [#2323](https://github.com/souliane/teatree/issues/2323)). All default to empty / permissive so existing overlays keep working — core enforcement only activates for overlays that opt in.

**Repo SCOPE axis (`owned_repos`).** `OverlayConfig.owned_repos` is a forge-host-keyed dict of the namespace patterns the overlay legitimately works on — the **scope** axis, orthogonal to **visibility** (`private_repos`/leak-prevention, fails open) and **collaboration** (`author_is_self`). Classification (`teatree.core.repo_scope`) gates host equality before the namespace half, so a `gitlab.com` repo can never match a `github.com` scope. The unknown-repo gate (`owned_repo_guard`) holds a push or keystone merge to a repo no opted-in overlay owns — fails **CLOSED** on a clean unknown verdict (opposite the visibility gate) and **OPEN** on a resolver exception (never-lockout). It ships **INERT** (`require_owned_repo_approval` default `false`); enabling it requires first declaring the FULL owned host/namespace list in private `~/.teatree.toml`, since a partial list would hold the operator's own private merges as unknown. A path-only TOML overlay cannot opt itself in (its repos are declared under an instantiable overlay). The PreToolUse hook bootstraps Django before resolving overlays so the gate actually fires in production. Owned does NOT imply auto-merge: a shared product repo is in scope yet still needs colleague review.

**Overlay API version pin.** `teatree.__overlay_api_version__` is bumped on any breaking change to the overlay-facing API. Overlays assert this at import to fail loudly when teatree diverges from what they were built against.

**Docker base-image sharing (§6.2a).** Teatree builds each `BaseImageConfig` exactly once per `(image_name, lockfile-hash)` and shares the image across every worktree that needs it. Code isolation is volume-mount level; the image itself is shared.

**`t3 startoverlay <name> <dest>`** scaffolds a lightweight overlay package: `src/<name>/{__init__,overlay,apps}.py`, `skills/overlay/SKILL.md`, `pyproject.toml`. No `manage.py`/`settings.py`/`urls.py` — teatree is the Django project.

**Discovery:** `~/.teatree.toml [overlays.<name>]` wins over `teatree.overlays` entry points on name conflicts. `discover_active_overlay()` returns the unique overlay if one exists, else the one whose `manage.py` is in cwd ancestors.

## 7. Backend Protocols and ABCs

Every external API concern is a `@runtime_checkable Protocol` in `teatree.core.backend_protocols` (the domain owns the abstractions it consumes; `backends` implements them). PR is the canonical term in core; GitLab MRs are translated at the API edge.

| Protocol | Implementations |
|---|---|
| `CodeHostBackend` — PR/issue/comment (incl. `list`/`update_issue_comment`)/upload/review-state + the §17.4.3 merge-RPC surface (`fetch_live_head_sha`, `fetch_pr_merge_state`, `fetch_pr_is_draft`, `fetch_required_checks_rollup`, `merge_pr_squash_bound` — raw forge I/O resolved via `core.backend_registry`; `merge.execution`/`merge.ci_rollup` keep the verdict/error classification) | `GitHubCodeHost`, `GitLabCodeHost` |
| `CIService` — pipeline cancel/trigger/quality-check | `GitLabCIService` |
| `MessagingBackend` — mentions/DMs/post/reply/react | `SlackBotBackend`, `NoopMessagingBackend` |

Request parameters are grouped into frozen `slots=True` dataclasses (`PullRequestSpec`, `MessageSpec`). `repo + pr_iid` is the natural unit on both code hosts — protocol methods never accept free-form PR URLs.

**Selection.** Per-overlay configuration (`~/.teatree.toml`) declares `code_host = "github" | "gitlab"` and `messaging_backend = "slack" | "noop"`. The loader (`backends/loader.py`) resolves the overlay's selected backend with no platform branches in caller code, cached `lru_cache(maxsize=1)` per overlay identity.

**Inbound events.** `t3 slack listen` runs a Socket Mode receiver that writes events to append-only JSONL queues (`slack-events.jsonl`, `slack-reactions.jsonl`) so scanners can drain atomically without racing.

**Reaction surface (#1281).** FSM `reactions.add` failures (`missing_scope`, `not_in_channel`, `mcp_externally_shared_channel_restricted`, …) raise `SlackReactionError` from `backends/slack/react_errors.py` — never silently return `False` — so callers cannot fall back to a `chat.postMessage(text=":emoji:")` thread reply. `SlackBotBackend.post_message` / `post_reply` reject bodies matching `^:[a-z0-9_+\-]+:$` with `SingleEmojiBodyRefusedError`, foreclosing the failure-mode shape at the backend boundary. FSM-side wrappers (`add_reactions_for_transition`, `add_approval_reaction`) catch the raise locally so Slack auth gaps cannot roll back FSM transitions.

**Destination-routed post/react ([#1750](https://github.com/souliane/teatree/issues/1750)).** Token picked by *destination* via `SlackBotBackend._route_token`: user's own DM → bot; colleague/channel → personal `xoxp`. Every colleague-surface post/react under the user's identity routes through the one gate+route+audit chokepoint `core/on_behalf_egress.OnBehalfSlackEgress` (self-DM acks ungated); the call-site authorization is pinned by the `on-behalf-routed-egress` / `on-behalf-colleague-primitives` entries in the chokepoint registry (`quality/chokepoints.yaml`, enforced by `scripts/hooks/check_chokepoints.py`); see [`skills/rules/SKILL.md`](skills/rules/SKILL.md).

**Deterministic reference linkifier.** The clickable-references rule (a bare `#N` / `!N` on a user-facing surface must become a clickable link) is enforced *in code* by `core/reference_linkifier.py`, not by asking the model. `ReferenceResolver` resolves DB-first (`PullRequest`, `Ticket.issue_url`) then constructs from the active repo context; `linkify` (idempotent; skips linked refs, inline code, fenced blocks) runs at the Slack chokepoint `SlackReplier._deliver`. Forge posts are NOT linkified (forge auto-links bare ids). It is the sole mechanism for the rule — the former bare-reference *blocking* gate was removed (it over-blocked and asked the model to rewrite refs non-deterministically); the linkifier rewrites in code with no block and no model round-trip.

**Sync ABC (`core/sync.py`).** `SyncBackend` is an ABC with `is_configured(overlay)` and `sync(overlay) → SyncResult`. Implementations: `GitHubSyncBackend`, `GitLabSyncBackend`. Both consume `CodeHostBackend` — platform-specific code lives only in the Protocol implementation, not in sync logic.

## 8. Command Tiers

| Tier | Tool | Needs Django? | Examples |
|---|---|---|---|
| Runtime | django-typer management commands | Yes | `worktree provision`, `tasks work-next-sdk`, `followup sync` |
| Bootstrap | Typer CLI (`t3`) | No | `t3 startoverlay`, `t3 info`, `t3 ci cancel` |
| Overlay | Typer CLI delegating to `manage.py` via subprocess | Indirectly | `t3 <overlay> start-ticket`, `t3 <overlay> worktree start` |

Internal utilities (`utils/`) are Python modules, not a CLI tier.

**Runtime commands** (`core/management/commands/`): `lifecycle`, `tasks`, `followup`, `workspace`, `worktree`, `db`, `env`, `run`, `pr`, `ticket`, `tool`, `e2e`, `overlay`, `standup`, `mr_reminder`, `checking`, `availability`, `retro`, `loop_tick`, `generate_*_docs`. Each is a django-typer command group with subcommands. `db query` and `db shell` enforce read-only at two layers (leading-keyword filter + transaction `READ ONLY` / `query_only=ON`).

**Retro enforcement tooling** (`t3 <overlay> retro review-findings`, [#1573](https://github.com/souliane/teatree/issues/1573)): the scaffold behind invariant 6 / §17.6. Fingerprints a PR's review comments, records the **supplied** A/B/C verdict (never auto-guessed), and files one deduped enforcement issue per class-C finding. The untrusted comment text is a leak vector, so its bare refs are neutralized and the rendered body banned-term scanned before filing — the `gh api` stdin path bypasses the PreToolUse publish gate, so this is the only guard.

**Gate-failure feedback loop** (`t3 <overlay> retro gate-failures`, [#2024](https://github.com/souliane/teatree/issues/2024)): the review-findings sibling whose input is a **gate firing on the agent's own output**. A teatree gate BLOCK lands in the session transcript as a `hook_blocking_error` attachment whose text leads with a `TEATREE GATE — <phrase>` marker. `eval.gate_failures.extract_gate_failures` reads the single transcript chokepoint, keys on the attachment type + marker (never `exitCode`), and reduces the gate identity to a bounded slug; `classify_gate_failure` routes each as `environmental` (a `hook_non_blocking_error` — missing plugin dir, hook-runner traceback) or `preventable` (anything agent-output-shaped, and any unknown BLOCK — fail toward an eval), and `--escalate` files one deduped enforcement issue per *recurring preventable* failure. Privacy by construction: `GateFailure` stores only the slug + session id, never the message/`stderr`/`command`. It lives in `teatree.eval` (a `core → eval` import would be a backwards tach edge; `retro` calls in on a forward edge). Auto-invoking it from the loop tick is orchestrator-only ([#837](https://github.com/souliane/teatree/issues/837)), a separate change.

**Checking report** (`t3 <overlay> checking show`, #1529): a terse, read-only "what did I miss" catch-up for when the user checks in mid-loop. By default aggregates ALL configured overlays into one `AllOverlaysReport`; `--this-overlay` restores single-overlay scope. Each overlay has its own `checking_checkpoint_<overlay>.json` marker (atomic `tmp.replace` write, tolerant read); each marker advances independently AFTER gathering so a second run sees an empty window. Three groups: Merged / In-flight / Needs you, every reference clickable, capped at 5. `DeferredQuestion` is queried ONCE for the whole report; overlay-scoped items carry an `[overlay]` inline tag. A window start at/after `now` falls back to the 24h lookback; advance is monotonic. `--since` and `--no-advance` inspect without advancing. The needs-you group is overlay-extensible via `OverlayBase.get_checking_sources()` (default `[]`); core makes no live forge calls.

**Global CLI** (`cli/`): `t3 startoverlay`, `t3 agent`, `t3 info {,artifacts}`, `t3 sessions`, `t3 cost`, `t3 docs`, `t3 ui`, `t3 admin`, `t3 ci ...`, `t3 review ...`, `t3 review-request ...`, `t3 tool ...`, `t3 config ...`, `t3 doctor ...`, `t3 update`, `t3 setup ...`, `t3 assess`, `t3 infra ...`, `t3 loop {start,stop,status,tick,slack-answer,claim-next,pause,resume,disable,enable,loop-state}`, `t3 recover`, `t3 overlay {install,uninstall,status,contract-check}`. `t3 ui` is a trogon-backed terminal browser of the command tree (optional `ui` dependency group). `t3 admin` runs the Django admin for the teatree project on a local runserver — it migrates, ensures a superuser (creating one non-interactively from `T3_ADMIN_USER`/`T3_ADMIN_PASSWORD` when absent), and opens `/admin/`. `t3 cost` reports cycle-to-date SDK-equivalent spend vs the monthly Agent-SDK credit from each `TaskAttempt`'s captured cost. `t3 info artifacts <ticket>` is the read-only "find our eggs" report — where a ticket's worktrees/stacks live, its `PlanArtifact` rows, each `Task.result_artifact_path`, and its `E2eMandatoryRun` evidence (aggregated by `Ticket.artifacts()`, no new storage). `t3 overlay install <name>` editable-installs a sibling overlay checkout into a teatree feature worktree — refuses to run in the main clone.

**Attachment ingestion** (`t3 tool to-markdown <file>`, #1479): converts binary spec attachments (PDF, XLSX, DOCX, PPTX) to Markdown so the agent can read them as structured text. Wraps `markitdown` (`teatree.backends.markdown_conversion.MarkdownConverter`) behind the **optional** `markdown` extra (`markitdown[pdf,docx,xlsx,pptx]` — never `[all]`); absent the extra the command exits non-zero with an install hint rather than crashing. Plugins are disabled and no LLM client is wired — converted output is treated as untrusted data and emitted verbatim.

**Child work items** (`t3 <overlay> ticket create-sub --parent <url> --title <…> [--type Task|Incident|Issue]`): the sibling of `ticket comment`, resolving the code host per-URL across overlays and delegating to `CodeHostBackend.create_sub_issue`. On GitLab this folds the three error-prone hops into one — REST create, `workItemConvert` to the child type (an Issue→Issue parent link is forbidden, so the default `Task` is the natural sub-item), then `workItemUpdate hierarchyWidget.parentId` to nest it — returning the child IID + URL for chaining. GitHub child work items use a different API and are unsupported.

**Overlay contract check** (`t3 overlay contract-check --compose <paths>`) reads every `${VAR}` reference in compose files and fails if any is neither defaulted nor declared by core (`_declared_core_keys()`) or the active overlay (`OverlayBase.declared_env_keys()`).

**Teatree source resolution in overlays.** `[tool.uv.sources] teatree = { path = "../../souliane/teatree", editable = true }` is the committed default — no SHA pinning, no mode switching. CI clones teatree at the same relative path before `uv sync`. Local dev uses whatever is checked out locally.

## 9. Code Host Sync

`teatree.core.sync.sync_followup() → SyncResult` is platform-agnostic. Per-overlay it resolves the overlay's `CodeHostBackend`, fetches open PRs authored by the current user (incremental via cached `updated_after`), upserts tickets by `issue_url` (or PR URL if no issue linked), enriches non-draft PRs with pipeline + approvals + review threads, infers ticket state from PR data (`infer_state_from_prs()` advances forward only, never regresses — and a post-ship inferred state is routed through the shared DoD decision per § "Sync writers honor the same gate"), and detects merged PRs.

| PR data | Inferred state |
|---|---|
| Draft | `started` |
| Non-draft | `shipped` |
| Non-draft + review-requested or approvals > 0 | `in_review` |

Review threads are classified `waiting_reviewer` / `needs_reply` / `addressed`. Draft notes (GitLab) / pending reviews (GitHub) surface as a statusline `review_draft` prompt to publish.

Posting discipline (#1207): `t3 review post-comment` defaults to creating a DRAFT and DMs the user the link; the colleague-visible `--live` path is gated on a single-use, MR-URL-scoped `LivePostApproval` minted by `t3 review approve-live-post <mr-url> --slack-ts <ts>` after the Slack DM at that timestamp is verified (from the user, recent within 15 min, contains an explicit approval phrase). The historical immediate-post default is retired; CLI enforces draft-by-default rather than relying on prose discipline.

One-step authorization (#126): `t3 review authorize <repo>!<mr> --approver <id>` collapses the former two-command on-behalf dance — it records the durable `OnBehalfApproval` AND mints the matching single-use `LivePostApproval` in one call. The `post-comment --live` path consults one consolidated read-only decision (`review_authorize.resolve_live_authorization`); the two original commands remain for the Slack-ts verification path.

Published-note removal: `t3 review delete-discussion <repo> <mr> <note_id>` removes a published MR note; `t3 review delete-issue-note <repo> <issue_iid> <note_id>` is the issue/work-item twin (`DELETE /issues/{iid}/notes/{id}`). Both are colleague-visible on-behalf mutations routed through the recorded-approval gate (#960) — the issue path scopes the approval to `<repo>#<issue>` (distinct from the MR `<repo>!<mr>` scope, via `issue_gate_target`) and verifies the delete took (#2081, `verify_issue_note_deleted`). These CLIs are the only sanctioned path: the `block-raw-review-post` hook (#1164) hard-denies a raw `glab api`/`gh api` DELETE to `…/issues/<iid>/notes/<id>`.

Verified-delivery notify wrapper ([#1181](https://github.com/souliane/teatree/issues/1181)): `teatree.messaging.notify_with_fallback` tries `notify_user` and, on a transport `FAILED`, falls back to a round-trip-verified backend send (`BotPing.transport` records the path). Loop/CLI sites route through the wrapper; `teatree.core` callers stay on `notify_user` (no `core → messaging` edge). A `NOOP`/`FAILED` INFO `BotPing` is durably parked, and the always-on `undelivered_notify` scanner re-attempts parked rows each tick under their `idempotency_key`; the drain is bounded ([#2064](https://github.com/souliane/teatree/issues/2064)) — a row past `MAX_REDELIVERY_ATTEMPTS` or 72h is `EXPIRED`, so the backlog never grinds and stale DMs never surface late.

Review-shape audit (#1206): `t3 review run <MR_URL>` is the read-only entry point reviewer sub-agents call before scanning a diff. It fetches MR metadata, classifies complexity, counts existing-review state (open discussions + draft notes + approvals), and emits a structured JSON summary so every reviewer starts from the same shape rather than improvising. The command never publishes — it stays outside the on-behalf surface. A merged/closed MR (live `state` field from `/changes`) short-circuits to a `skipped_merged` / `skipped_closed` verdict (#2081) — a note can never land on one, so a mid-flight close aborts the review. GitHub PR URLs return `unsupported_forge` (exit 2) deterministically until a parallel GitHub backend lands.

Structured-evidence gate (#1280): `t3 review post-comment` and `post-draft-note` refuse a finding whose body matches an "X is missing/wrong/broken/stale" pattern unless an accompanying `FindingEvidence` record (`--evidence-json '{...}'`) carries verified receipts — it passes only when `confidence='verified'` AND at least one verified-path field is non-empty (full schema in code). Implemented in `teatree.cli.review.evidence_gate`; runs alongside the on-behalf (#960), colleague-MR shape (#1114), and TODO-anchor (#1186) sibling gates inside `ReviewService._run_pre_publish_gates`.

Close-trailer scanner (#1398): `[teatree.publish_gates] ban_close_trailers_on_namespaces` lists fnmatch patterns over `namespace/repo`. When the target PR/MR's repo matches and the body carries a `Closes|Fixes|Resolves` trailer (`part of` and full-URL variants too), `ShipExecutor._build_pr_spec` silently strips those lines before opening the PR (`teatree.core.close_trailer_scanner`). Distinct from the overlay-scoped `forbid_close_keywords` gate (#1012), which refuses the publish; this scanner cleans the body and proceeds.

Open-questions warn (#1933): any open question (solved or not) and any non-explicit assumption must be listed in BOTH the commit body AND the PR description under an `Open questions & assumptions` section (per-item status `decided-by-user` / `assumed` / `open`). `teatree.core.gates.open_questions_gate` runs at both PR-creation chokepoints (`ShipExecutor._build_pr_spec`, orphan-branch `create_or_defer_pr`): a body lacking the heading warns and proceeds. Warn-only per the "a gate without a reliable heuristic warns" rule — the heading wording is not separable enough to block on. Canonical doctrine in `skills/ship/SKILL.md` § 5.

Egress-leak gate family — one doctrine, several entry points, each named here with its load-bearing invariant; detector lists, exit codes, thresholds, and exempt sets live in the code:

- **Public-repo diff privacy-scan** (#685, #730): pre-push hook `refuse-public-push-with-leak.sh` runs `t3 tool privacy-scan` over the pushed diff + commit messages when `origin` is PUBLIC, blocking emails / home paths / private IPs / keys / internal hostnames / banned terms. It fails CLOSED on a genuine finding (a dedicated exit code) but fails OPEN on any other non-zero so a scanner crash cannot wedge every push (#126).
- **Foreign-open-MR push guard** (#2211): pre-push hook `refuse-push-to-foreign-mr.sh` — a collaboration-safety sibling (not a leak gate). For each pushed ref it resolves the branch's backing OPEN PR via `gh pr list --head <branch> --state open` and BLOCKS, naming the author + PR number, when the PR's author login != our `gh api user` login — a teammate's open MR our changes would silently modify (our work belongs on our own branch; a worktree opened to INSPECT a colleague's MR is read-only). Our own open MR, a branch with no open MR, and a foreign CLOSED/merged MR (which `--state open` excludes) all pass; an explicit `[push-to-foreign-mr-ok: <reason>]` commit-message token allows a genuine co-authoring push. Fails OPEN on a `gh` failure / gh-unavailable / unresolved login so a transient forge-API error never bricks a legitimate push.
- **Banned-terms posting gate** (#1415, `teatree.hooks.banned_terms_scanner`): the `PreToolUse` non-commit sibling. It and the #1213 quote-scanner share `extract_bash_payload` and a per-segment override (#2031/#2034): the `--allow-banned-term`/`--quote-ok` flag token OR a leading `ALLOW_BANNED_TERM=1`/`QUOTE_OK=1` env-prefix clears only on the publish segment, so a chained-segment decoy cannot vouch. The deny message names the **env-prefix** form (`ALLOW_BANNED_TERM=1 <command>` / `QUOTE_OK=1 <command>`), not the `--…` flag: the flag is consumed by the gate's parser, never the posting command, so a `t3 review post-comment` subcommand would reject it as an unknown option — the env prefix is the spelling that actually works on every command. A `> path <<EOF` heredoc is scanned only when its path is posted via a body-file flag; a stdin `--body-file -` heredoc always is. An inline `--body`/`-m`/positional-NOTE body whose `$(...)` command-substitution sits inside a SINGLE-quoted span is inert literal text bash passes verbatim, so `resolve_inline_body_value` reads the value token's source span (`raw`) and SCANS such a body rather than fail-closing on "publish body could not be read" — a body bash WOULD expand (unquoted / double-quoted `$(...)`) still fails closed since its real content is unknown before the command runs (#1415); the raw-aware `command_has_opaque_forge_transport` mirrors this so a non-forge segment that merely mentions a single-quoted `$(...)` is not treated as a hidden forge transport. The gate delegates matching to `check-banned-terms.sh`, whose exit-code contract is 0=clean / 1=banned-term-found / 2=scanner-could-not-run; when the shell fallback resolves to an old system `python3` (below the >= 3.13 floor) the matcher import crashes, so the fallback probe-imports the module first and exits 2 (fail CLOSED, loud) rather than letting the crash exit 1 (which collides with "found") become ALLOW. `scan_text` mirrors this: a crash, timeout, unexpected exit, or exit-1-with-no-report returns `SCANNER_UNAVAILABLE_MARKER` and BLOCKS — a security gate that fails open on a crash is the bug class (#1954).
- **Pre-dispatch quote-scanner** (#1401, `handle_dispatch_prompt_quote_scanner`): the dispatch-boundary companion to #1213 — denies only a HIGH user-voice/PII match in an `Agent`/`Task` prompt, with a `[quote-ok: <reason>]` opt-out, so a verbatim quote cannot leak downstream.
- **Diff comment detectors** (added-lines-only): `code_comment_self_reference` (#1465) is a **blocking** `privacy_scan.py` diff detector (**fail-open per detector**, #1536) for bookkeeping self-references. `code_comment_density` (#1538) is the commit-side half of the near-zero-comments rule (#1532); it is **advisory** — the standalone check (#1369, `t3 tool comment-density`), the pre-push hook, and the `comment-density-warning` CI job warn and **exit 0**, and it is NOT a `privacy_scan.py` blocking detector (#1844 — no content-blind "overly long prose" heuristic that would flag legit long comments). A golden corpus (`tests/test_comment_density_gate.py`) + an eval pin must-DENY symmetric with must-ALLOW. The push-stage gates (`doc-update-gate`, `ensure-pr`, the leak gate) are `stages: [push]` — skipped by a bare `prek run --all-files` but re-run by CI — so `t3 tool verify-gates` runs both stages and returns the combined exit code, so local green == CI green.
- **Full-tree banned-brand backstop** (#1570/#1591, `core.banned_terms_tree`, CLI `t3 banned-terms scan-tree`, CI job `banned-terms-tree`): catches what the diff/payload gates structurally cannot — a brand ALREADY committed, invisible to any post-landing diff — by scanning every tracked file's **committed blob** (`git show HEAD:<path>`, working-tree fallback only for an uncommitted file) with the shared whole-token matcher (`teatree.hooks.term_match`, pinned by the parity test). The brand list (`[teatree].banned_brands`) is a curated high-confidence subset distinct from the egress `banned_terms`; unpopulated, the scan is INERT and says so loudly rather than returning a silent clean 0. The CI step passes `--require-brands`, upgrading that inert state to a hard MISCONFIGURED failure (exit 2, distinct from exit 1 = findings) so a missing secret reds the job loudly instead of a fake-green no-op. Local dev omits the flag.

## 11. Skills & Plugin Architecture

Skills live in `skills/*/` — one `SKILL.md` + optional `references/` per skill. When installed as a plugin, skills are namespaced under `t3:` (e.g. `/t3:code`). The lifecycle skill set is `code`, `debug`, `test`, `review`, `review-request`, `ship`, `ticket`, `workspace`, `followup`, `handover`, `next`, `retro`, `contribute`, `setup`, `platforms`, `rules`. Read-only auxiliary skills sit alongside it: `checking` (what-did-I-miss) and `todos` (the session's task list, via `tasks list --session`).

Skills declare dependencies via YAML frontmatter `requires:` (transitive, topo-sorted) and optional `companions:` (best-effort, warn on miss). Third-party skill frameworks (e.g. superpowers) are absorbed into the `rules` skill rather than delegated, to avoid context duplication.

**Sub-agents (`agents/`).** Phase agents wrap skill bundles (`orchestrator`, `coder`, `tester`, `e2e`, `e2e-review`, `reviewer`, `shipper`, `debugger`, `followup`, `planner`, `answerer`, `scanning-news`) — each a YAML+description wrapper referencing skills via `skills:` frontmatter, no content duplication. `e2e-review` ([#2298](https://github.com/souliane/teatree/issues/2298)) is the reviewer leg of the EXTERNAL review loop (dispatched for the `e2e_reviewing` phase), making the skill↔agent pairing 1:1 for `e2e`/`e2e-review`. They are invoked by lifecycle skills, by the headless executor (§5.2) on a claimed phase task, and by the loop tick (§5.6) when a scanner signal calls for agent judgment. Interactive-only skills (no agent): `retro`, `next`, `contribute`, `setup`.

**Distribution.** Two install paths, one source of truth:

- **APM**: `apm install souliane/teatree`
- **CLI-first**: `git clone … && uv tool install --editable . && t3 setup` — also registers the plugin in `~/.claude/plugins/installed_plugins.json` with `installPath` pointing at the main clone (no `~/.claude/plugins/t3` symlink; always live)

On every `t3 setup` run, `dep_drift` checks `[project].dependencies` against the editable install and reinstalls + `execv`-restarts if a declared dep is missing. The same run re-syncs runtime skill links and **prunes stale ones** — a teatree-managed link whose skill was removed or renamed upstream is removed so the dropped skill stops resolving, while contribute-mode workspace links and a user's own real skill directories are left untouched. Because `t3 update` re-runs `t3 setup`, updating teatree auto-cleans skills dropped upstream.

**§11.4 Bash Permissions.** The plugin's `settings.json` ships a **broad allow, narrow deny** `permissions` list — every tool family the workflow touches is allowed, with load-bearing denies (push to default branches, `--force`, `--no-verify`, root `rm -rf`, `curl|bash`, `gh repo delete`) taking precedence; the `t3` CLI is the workflow's safety wrapper, so blocking inside the CLI is the wrong layer. Plugin config is **not self-modifiable** — edits to the allow-list are rejected by Claude Code's autonomy guardrail, so a mid-workflow classifier denial means the agent stops and asks via `AskUserQuestion` (`skills/rules/SKILL.md` § "Classifier Denial Protocol"). `t3 doctor authorizations` is read-only — it reports which generic recommended auto-mode authorizations are absent from `~/.claude/settings.json`; teatree ships **no** classifier whitelist of its own.

## 17. The Self-Improving Factory Architecture

Teatree is a durable self-healing **and** self-improving development factory. This section is the lasting architectural reference — the umbrella under [#836](https://github.com/souliane/teatree/issues/836); each component below is a separately tracked ticket implemented as deterministic teatree code, not skill prose.

Why this architecture exists, observed repeatedly: durability comes from **enforcement encoded in code/structure**, not prose that decays. A rule kept in memory/skills and relied on by vigilance recurs anyway; the same rule encoded as a gate/test/hook does not. The invariants below are the structural form of that lesson — load-bearing, binding every change to teatree itself.

### 17.1 Invariants

1. **Two layers, never conflated.** *Self-healing* (independent review, draft-locks, recovery, gates) is the substrate. *Self-improvement* runs on top: each caught failure-class becomes the smallest enforcement artifact that makes the class structurally impossible. Self-improvement is **gated by** self-healing — the system is never changed in a way the healing layer cannot catch or roll back.

2. **The flywheel.** A defect (from diff review, OR the code-health loop on un-changed code, OR an orchestrator-noticed near-miss) → the orchestrator synthesises → output is the *smallest enforcement* (gate/test/hook), never a prose rule → the failure-class is extinct. A repeat failure whose only output is memory/prose is a flywheel failure.

3. **Topology.** The orchestrator is the synthesis brain (retro synthesis, code-health triage, enforcement escalation, merge/clear decisions). Sub-agents are sensors/hands emitting structured signal into durable state, never self-judging. Skills carry judgment/methodology; teatree code carries the deterministic loops, gates, and intake. Corollary: mechanics → code, judgment → skill.

4. **Blast-radius rule.** Substrate changes are draft-locked and need a recorded human sign-off, satisfied by EITHER a per-PR `MergeClear.human_authorizer` OR the overlay standing at `autonomy = full` — the owner's standing grant to merge substrate without a per-PR sign-off. Either way the agent runs the `t3 <overlay> ticket merge` keystone (§17.4.3); the human never performs the merge. The carve-out removes ONLY the per-PR sign-off — the floor (cold-review `reviewer != maker`, SHA-bind, CI-green, not-draft, never-lockout, privacy/leak scan) holds on every substrate merge; below `full` the `human_authorizer` stays mandatory.

5. **Durability discipline is load-bearing.** Durable task/state plus pre-compaction snapshots let the orchestrator brain survive compaction/restart; keep them.

6. **Enforcement over prose, as a standing audit.** Invariant 2 makes the flywheel's *output* a gate; this invariant makes the *standing posture* explicit. Every user behavioural directive MUST be (a) codified in teatree and (b) enforced by deterministic code/gates wherever mechanizable, so skill prose shrinks to the judgment that genuinely cannot be mechanized and skills get **lighter** over time. This is a recurring retro/review responsibility: the enforcement gate (§17.6 / [#850](https://github.com/souliane/teatree/issues/850)) turns a mechanizable rule into a gate, and the recurring audit keeps reclassifying prose rules → code gates.

7. **Consolidation over drift.** Behavior encoded outside the teatree framework — personal `settings.json` hooks, dotfiles automation, overlay-local ad-hoc config, personal memory guardrails — must be considered for promotion into teatree on every retro/review pass. Genuine per-instance variance must be modelled as a documented teatree setting or config knob; undocumented divergence silently drifts and violates invariant 2.

8. **All FSM state transitions go through the `t3` CLI.** The pre-condition and pre/post transition hooks are the coherence mechanism (ledger update, attestation-binding to the HEAD/workstream the phase was earned against, privacy/AI-signature scan, `mark_merged()`). Out-of-band state mutation — raw `gh pr merge` / `glab mr merge`, or hand-editing the phase ledger / FSM state — is prohibited and **mechanically guarded** (`hook_router._BLOCKED_COMMANDS`, the same hook layer as the draft-lock and structured-question gates). The keystone IN_REVIEW → MERGED transition is §17.4; the non-bypassable gate placement is §17.6.3. The two sanctioned escapes for stale state — `lifecycle clear-ledger` and the hardened `lifecycle visit-phase … reviewing --agent-id` — are themselves `t3` commands, never manual edits.

9. **Every user-directed question is captured — sync or durable — and reaches Slack.** A user-directed question must either (a) call `AskUserQuestion` with the user reachable this turn, or (b) be recorded as a `DeferredQuestion` row when the resolved availability mode is `away`. Mode resolution is a single deterministic precedence (exposed by `t3 availability`): unexpired manual override → live presence (a recent `UserPromptSubmit`, upgrade-only) → `[teatree.availability]` cron-window → `present` (default). The away path is a *sanctioned destination* for the same `AskUserQuestion` call (converted at the `PreToolUse` layer, never bypassing the §807 structured-question gate); since the user reads Slack, every question is mirrored to their DM (away mirrors before denying) and away→present auto-drains the backlog. A live user turn (a same-session `UserPromptSubmit` within a 90 s freshness window) renders in-client even under manual-away so `/checking` works without an availability flip, and present/away capture unify on one generation-stamped, mirror-linked `DeferredQuestion` whose *answer* is delivered back — a Slack reply binds the live generation and the next prompt injects it once. The presence/window-slide mechanics live in `docs/blueprint/loop-topology.md`. Component: §17.3 C3.

10. **Orchestrator never executes work directly — every implementation, review, test, debug, and ship action is dispatched to a sub-agent.** The orchestrator's role is synthesis, classification, dispatch, and CLEAR issuance (invariant 3, §17.4.1, §17.8 clause 3); the *hands* are sub-agents (`t3:code`, `t3:review`, `t3:test`, `t3:debug`, `t3:shipper`, `/teatree-batch`'s delivery sub-agent) and the durable loop. Inlining implementation work — even a "trivial" typo Edit, a Bash run redoing a sub-agent's job, a local test cycle dodging `t3:test` — conflates judgment with execution, denies maker≠checker independence, and concentrates the compaction risk the topology spreads across durable handoffs. **Narrow exceptions:** read-only orientation in the orchestrator's own session (`Read`/`Grep`, `gh pr view`, `git status`, `AskUserQuestion`, sanctioned messaging), the `t3 …` invocations it owns (`MergeClear`, attestation, next-dispatch), and conversational replies with no repo mutation. Mechanized as gate 2 in §17.6.4 (`handle_enforce_orchestrator_boundary`): a `PreToolUse` deny on a main-agent LONG/HEAVY foreground `Bash` command, with `run_in_background: true` the escape hatch and a payload `agent_id` distinguishing sub-agent from main ([#115](https://github.com/souliane/teatree/issues/115)). It shares one `_fail_open_or_deny` chokepoint (with always-available `t3 <overlay> gate … disable` kill-switches and the `self_rescue` allowlist) with the sibling over-deny gates; the PUBLIC-egress leak gate is excluded and stays fail-CLOSED. Detail in §17.6.4.

11. **Any interactive Claude Code session that mounts this teatree install MAY drain the `PendingChatInjection` queue.** The inbound-Slack bridge (#1014) records each user DM as a `PendingChatInjection` row; the `handle_inject_pending_chat` `UserPromptSubmit` hook drains unconsumed rows into the next prompt's `additionalContext`. Drain eligibility is **decoupled from loop ownership**: the autonomous `t3 loop start` session holding `_OWNER_LOOP` never receives `UserPromptSubmit` events, so a loop-owner gate here would block *every* user reply from reaching *any* interactive session. At-most-once delivery rides primitives orthogonal to ownership — the single-use `PendingChatInjection.consume()` (`UPDATE … WHERE consumed_at IS NULL`) and the `(overlay, slack_ts)` `UniqueConstraint` — so the scanner can over-poll safely. The loop-owner gate stays on the §5.6 self-pump (singleton), not on inbound message drains.

12. **An outcome-claiming completion carries a resolvable artifact pointer, fail-closed.** The out-of-band completion surface (`tasks complete --note`) records externally-landed work. `teatree.core.completion_evidence` makes two separate judgments: a note *asserts an outcome* when an `OUTCOME_CLAIM_KINDS` verb (`merged`/`posted`/`shipped`/`deployed` + synonyms) co-occurs with a context cue (internal idioms like `merge conflict` are stripped first), and an asserting note must carry a *resolvable pointer* an auditor can follow — a URL, a cued git SHA, an MR/PR/issue ref, a note id, a Slack ts, or a real/dotted-module path — else `check_completion_evidence` refuses (`CompletionEvidenceError`). A completion with no note, or asserting no outcome, is never gated (the structural form of invariant 6's "done claims require evidence"); the exact matchers live in `completion_evidence.py`. A `failed → completed` resolution requires a mandatory `--note` ([#1949](https://github.com/souliane/teatree/issues/1949)); `claimed → completed` stays note-optional.

### 17.2 The flywheel — 17.8 Orchestrator-as-keystone contract

The flywheel diagram, components (C1 Retro / C2 Code-health loop / C3 Availability), §17.4 Orchestrator-decides / loop-executes topology, §17.5 TODO-consolidation triage, §17.6 Enforcement gate (anti-relaxation, sound tach boundaries, the shipped gate family), §17.7 Enforcement-over-prose, and §17.8 Orchestrator-as-keystone contract — all live in [docs/blueprint/factory-architecture.md](docs/blueprint/factory-architecture.md), where the section headings (`### 17.2`–`### 17.8`) are preserved for cross-references.

**Anti-pattern catalog ([#166](https://github.com/souliane/teatree/issues/166)).** `src/teatree/quality/antipatterns.yaml` is the SSOT for recurring architectural anti-patterns; `teatree.quality.catalog` loads it and a hook renders [docs/generated/antipattern-catalog.md](docs/generated/antipattern-catalog.md). The `docs-drift` CI job regenerates the catalog (with `ANTIPATTERN_CATALOG_NO_STAGE=1` so it cannot self-stage and hide drift) before the `git diff --exit-code docs/generated` gate, so a yaml→catalog skew reds CI unconditionally. Each entry's `detection` tier (`greppable` vs `judgement`) feeds the three review tiers: design-time (`architecture-design`), per-PR deterministic (`check_antipatterns.py`), periodic holistic (`ac-reviewing-codebase`). The sibling `t3 tool test-shape` flags near-identical unparametrized tests and test:source ratio regressions past a baseline — advisory `warn` by default, opt-in `block`.

**Scoped mutation testing ([#131](https://github.com/souliane/teatree/issues/131)).** `t3 mutation run` runs mutmut over only diff-touched modules in the NARROW `[tool.teatree.mutation]` safety registry; `tests/quality/test_mutation*.py` carry the ledger + kill-proof. `BaselineRatchet` ([#1926](https://github.com/souliane/teatree/issues/1926)) makes the recorded `baseline_surviving` count a ratchet — a run above it fails in BOTH `warn` and `block` mode (count may only shrink), the prerequisite to flipping `mode` to `block`.

**Test-path-mirror forward-guard.** `t3 tool test-path-mirror` enforces *tests mirror production code*: a test file's first-party imports (parsed by AST, never executed) must place it under the mirror dir `tests/teatree_<pkg>/...` of at least one imported module; a top-level src *module* maps to the `tests` root as an EXACT-only expectation. The verdict is a `BaselineRatchet` count over `[tool.teatree.test_path_mirror] baseline` — RED only when the live mis-pathed count EXCEEDS the committed floor (files predating the convention), so a NEW mis-pathed file fails CI while the floor never does; `--update-baseline` refuses to raise the floor without `--allow-regression`. A small reviewed exemption set (`tests/integration/`, `tests/e2e*`, `conftest.py`, a `# test-path: cross-cutting` pragma, …) keeps legitimate multi-package tests out of the count.

**Behavioral eval harness ([#1160](https://github.com/souliane/teatree/issues/1160)).** `src/teatree/eval/` grades agent behaviour across free deterministic lanes (real-code-path regression corpus, skill-triggers, skill-coverage, negative-control, transcript-replay, corpus-grade, skill-command-validity) and a metered AI lane (in-process `claude-agent-sdk`, weekly standalone `eval.yml`, authed by `CLAUDE_CODE_OAUTH_TOKEN`). Bare `t3 eval` runs the whole suite in one table; subcommands target one lane. The architecture:

- **#550 3-tier skill model.** Tier-1 `skill-command-validity` (free) fails on a backticked `t3 …` in a SKILL.md that no longer resolves against the live CLI registry; Tier-2 is the metered behavioral scenarios; Tier-3 `skill-prose-judge` is **advisory** (scores SKILL.md prose, never fails the suite).
- **Under-load behavioural-drift lane ([#2346](https://github.com/souliane/teatree/issues/2346)).** `EvalSpec.lane` selects the harness mode: `clean_room` (default) loads one skill into an empty context; `under_load` reproduces real-session instruction drift — the full skill bundle as system prompt plus an 8k–20k-token polluted `context_preamble` folded into the user prompt. A `--lane` filter slices the one catalog so the cheap PR-path anti-vacuity gate and the weekly metered lane run the right subset.
- **Ground-truth corpus + audit layer ([#2192](https://github.com/souliane/teatree/issues/2192), [#1861](https://github.com/souliane/teatree/issues/1861)).** `eval/corpus/` pairs captured sessions with independently-authored labels under an independent-oracle guard (a matcher label whose author is the rule author is refused). The `SessionAuditRecord` ledger (parallel to `EvalScenarioResult`, ids/slugs only) is populated by the conversation-audit pass and rendered by a confusion-matrix renderer; the corpus/audit curation CLI (`t3 eval corpus|audit|label`) is the thin surface over these engines.
- **Cost + benchmark.** A per-scenario cost ledger (`EvalScenarioResult.cost_usd`) powers two cost gates and `t3 eval benchmark`. `--gate-cost-regression` is *relative* drift vs a mutable DB baseline run; `--gate-cost-bounds` is the *absolute* ceiling — each scenario's metered cost is checked against the CHECKED-IN `evals/cost_bounds.yaml` (calibrated `bound_usd` x `(1 + margin)`), a scenario over its ceiling or a configured-but-uncosted scenario fails loud (never skip-as-pass), and the engine is the pure `teatree.eval.cost_bounds`. Both gates read the durable ledger, so they run on the persisted weekly/manual metered path, off the per-PR path. `t3 eval benchmark` runs the suite per `model@effort` variant and reports pass-rate, cost-per-pass, and honest cache-aware token metrics (billed cost is the headline).

A skill's evals live in the central catalog `evals/scenarios/<skill>.yaml` (each spec carrying an explicit `agent_path: skills/<name>/SKILL.md`), never inline in the `skills/` tree (`tests/eval_replay/test_no_inline_skill_evals.py` enforces it); a skill with no eval declares `eval_exempt: <reason>`; `t3 eval coverage` is the warn-first per-skill coverage gate. Every run ends with a plain-language closing verdict, and a setup-skipped lane reads as `NOT RUN`, never as validated. [evals/README.md](evals/README.md) is the SOT for all eval architecture detail.

## Maintenance

The BLUEPRINT stays architectural — not a prose mirror of the code — by **review-time judgment**, not a raw byte gate. There is no hard size cap and no per-PR byte-delta budget: a single hand-edited KB constant that every BLUEPRINT-touching PR had to bump made concurrent PRs re-break each other's CI endlessly, with no quality signal to show for it. Keeping the file tight is a reviewer responsibility — flag bloat, prose that restates code instead of capturing architecture, and stale or duplicated sections — captured in `skills/review/SKILL.md` § "Keep BLUEPRINT Tight" and in the periodic holistic review (`ac-reviewing-codebase`). When a section grows past appendix-class detail, split it into a linked appendix under [docs/blueprint/](docs/blueprint/); the top-level file holds the architecture, the appendices hold the depth.

Two structural pre-commit gates remain (these are sync/structure checks, not raw-size enforcement): `scripts/hooks/check_blueprint_sync.py` (BLUEPRINT updated alongside source changes) and `scripts/hooks/check_blueprint_invariant_numbering.py` ([#836](https://github.com/souliane/teatree/issues/836), the §17.1 invariants form a gapless `1..N`, with the CI `blueprint-cross-pr` job catching the concurrent-PR case).
