# tests — local conventions

See the root [`CLAUDE.md`](../CLAUDE.md) for the code-quality bar. This file adds only what is specific to `tests/`.

- **Run:** `uv run pytest` for a fast inner loop — the default is parallel (`-n auto`) with no coverage. Add `-x -q` to stop on first failure. The full coverage gate (`bash dev/test-cov.sh`, and the CI `test (3.13)` lane) is **93% branch, non-negotiable** (`fail_under = 93, branch = true`; migrations omitted). New code ships with its tests in the same commit.
- **Before pushing a src-touching PR, run `bash dev/ci-parity.sh`** — the exact full CI predicate (prek, `makemigrations --check`, `t3 tool test-path-mirror`, `dev/test-cov.sh`, `t3 ci coverage`) in one command; run `bash dev/ci-parity-fast.sh` for the fast inner loop while iterating (scoped prek + `makemigrations --check` + the incremental push gate, NO coverage floor). The push-stage `ci-critical-parity` hook runs `dev/push-gate.sh` — the fast early signal (the never-lockout contract + the incremental push gate `t3 tool push-gate --run`: scoped doctest + ast-grep, FULL on any uncertainty, behind the default-TRUE `incremental_push_gate` flag — ON scopes the diff, OFF is the pre-#122 whole-tree run). The broad `tests/quality` dir is CI-only (~420s locally even with `push_heavy` deselected; the `test (3.13)` shard covers it whole-tree). The whole-tree coverage floor is provable only by the full lane, so it stays opt-in, never a push hook (`tests/test_no_full_suite_on_pre_push.py`).
- **The CI `test (3.13)` gate is sharded, the floor is not weakened.** CI runs a 12-way `test-shard` matrix (`pytest-split`) that measures coverage with no floor, then a `test` COMBINER aggregates the shards, proves an exact partition (`scripts/ci/check_shard_completeness.py`), and enforces the 93% floor once over the combined data — identical semantics to the single-process `dev/test-cov.sh`. `tests/test_coverage_floor_guard.py::TestShardedCoverageLane` locks that the floor stays load-bearing (needs-edge to the shards, ≥2 distinct groups, partition check, shard-pass guard).
- **The DB-heavy suite restores a migrated-DB template instead of re-migrating.** `django_db_setup` (`tests/conftest.py`) overrides pytest-django's stock fixture: the FIRST xdist worker of a fresh checkout to need the DB builds it via a real `migrate` and publishes an on-disk template (`tests/_db_template.py`, content-addressed by `schema_hash()` — sqlite version + every migration file + `uv.lock` + `tests/django_settings.py`); every later worker restores that exact byte-for-byte state via `sqlite3.Connection.backup()`. CI caches `.pytest-db-template/` across runs (`actions/cache`, exact-key-only — no `restore-keys`). `tests/teatree_core/test_db_template_equivalence.py` pins that a restored DB is indistinguishable from a fresh migrate; `tests/teatree_core/test_initial_migration_seed.py::FreshMigrateSeedsDefaultLoops` is unaffected (it drives its own `migrate core zero` round-trip on the live connection regardless of which lane produced it).
- **Tests mirror `src/`.** Test path mirrors the module under test; classes/methods describe behaviour, not implementation.
- **Lean integration / functional.** Prefer the Django test client, `call_command`, and real `git` under `tmp_path`. Reserve unit tests for pure logic (parsers, formatters, branch-name builders). Mock only unstoppable externals (network, clock, third-party subprocesses). Full rule + review gate: `AGENTS.md` § "Test-Writing Doctrine".
- **A regression test must be observed RED before the fix.** A test that passes on the buggy code guards nothing — see `/t3:code` § TDD Discipline.
